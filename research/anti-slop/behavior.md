# Behavioral and Engineering Slop in AI-Generated Frontends

## 1. Executive Summary

This report catalogs 80 behavioral and engineering failures commonly found in AI-generated frontend code, organized into six sections: interaction, motion, responsive/mobile, accessibility, implementation, and validation/QA. Each issue is documented with an observable symptom, likely code-level cause, static detection method, runtime/browser test, a normative corrective rule, valid exceptions, primary sources (W3C/WCAG 2.2, WAI-ARIA APG, HTML/CSS specifications, MDN, web.dev, Core Web Vitals, and framework/library documentation), and a confidence rating that separates objectively broken, inaccessible, deceptive, or inefficient behavior from aesthetic preference. Pure taste issues were deliberately excluded at every stage.

The recurring pattern (stated here as an observed inference, not a sourced fact): AI code generators are trained to maximize *visual* fidelity, so they reproduce the appearance of product UI far more reliably than its behavior. The result is code that demos well and fails in use — controls that look wired but are not, motion that plays regardless of user settings, layouts that hold only at the viewport the model imagined, and async flows with no failure semantics.

The most consequential findings, by theme:

- **Deceptive and dead interactivity is the signature failure.** Cosmetic-only widgets whose active states switch without changing anything (2.1), forms that fabricate success (2.2), fabricated live data and perpetual loading indicators (2.3, 7.4), and pointer-only pseudo-controls (2.5) make interfaces lie about what they do. Several of these are simultaneously WCAG A/AA conformance failures and user-trust failures.
- **Focus and keyboard contracts are systematically broken.** Modal dialogs without focus containment or restoration (2.8), click-only `div` controls (5.1), div-rebuilt comboboxes with no listbox semantics (5.2), removed focus indicators (5.3), and lost focus after dynamic updates (5.4) render whole flows unusable without a mouse — all traceable to specific WCAG 2.2 success criteria and APG patterns.
- **Motion is applied without consent, purpose, or exit.** Auto-starting animation with no pause/stop control (3.2), flashing near vestibular thresholds (3.1, 5.14), universal disregard for `prefers-reduced-motion` (3.3), scrolljacking (3.5), and entrance-reveal cascades that leave content permanently hidden when observers fail (3.8). Many of these are WCAG 2.2.2 / 2.3.1 failures, not stylistic choices.
- **Responsive behavior is a single-viewport snapshot.** Horizontal overflow from fixed widths (4.1), layouts squeezed rather than recomposed (4.2), navigation that collapses structurally but not behaviorally (4.3), hover-gated functionality with no touch path (4.4), and zoom disabled via viewport meta (4.7) — the code was written for one canvas size and merely *squeezes* elsewhere.
- **Engineering foundations are brittle.** Root-level `'use client'` and hydration mismatches (6.6), effect misuse and fetch races (6.7), unstable list keys corrupting row state (6.8), layout-property animation and re-render storms degrading INP (6.14), missing virtualization (6.15), whole-library and duplicate dependencies (6.3), `!important`/magic-number cascade abuse (6.9, 6.10), and `setTimeout` race coordination with no teardown (6.11).
- **Async reality is unhandled.** Missing loading (7.1), error (7.3), and empty states (7.5); absent error boundaries that turn one widget crash into a white screen (7.2); mutations with no outcome feedback or falsified success (7.6); optimistic updates without rollback (7.7); duplicate submissions (7.8); and status changes never announced to assistive technology (7.11).

**How to use this report.** Every issue's *corrective rule* is phrased as a normative must/should statement suitable for direct adoption as an AI code-review criterion, and every issue pairs a static detection method (lint rule, grep/AST pattern, or review heuristic) with a runtime or browser test (axe-core, Playwright, Lighthouse, DevTools), so the catalog can drive both automated pipelines and manual audits. Items labeled Medium/Low confidence or marked "inference" should be applied with judgment; everything anchored to a WCAG success criterion or documented API contract is objective.

**Scope note.** The catalog covers behavioral and engineering failures. Aesthetic preferences — color choices, easing "personality," layout style, spacing taste — were researched and deliberately excluded unless they cross into an objective failure (e.g., contrast below SC 1.4.3, durations that block task completion).

## 2. Interaction Failures

Across the research briefs, the interaction cluster shows the most consistent signature of AI-generated frontend code (observed pattern/inference, not sourced fact): controls are rendered with full visual fidelity — hover states, animated hamburger icons, active-tab pills, rotating chevrons — while the behavioral wiring that makes them function is missing, half-connected, or contradicts what the visuals claim. The recurring root pattern (inference): training data rewards static appearance and behavior leaves no trace in screenshots, so state is bound to CSS classes rather than to rendered output, open paths are generated while dismissal paths are not, and platform contracts (keyboard activation, focus movement, honest feedback) are systematically dropped. The failures below span the four classes used throughout this report — objectively broken (dead controls), inaccessible (mouse-only or focus-hostile widgets), deceptive (fake success, fabricated indicators, lying state), and inefficient — and each is stated as a review standard with static and runtime detection. Purely accessibility-deep aspects of the same widgets (label association, live-region semantics, focus-indicator contrast, target size) are cross-referenced to Sections 4 and 5 rather than duplicated here.

### 2.1 Cosmetic-only widgets: activation changes styling but never content
- **Classification:** objectively broken | deceptive
- **Severity:** Critical
- **Observable symptom:** Clicking a tab moves the underline/pill while the panel below shows identical content for every tab; an accordion chevron visibly rotates but no panel ever appears; a hamburger icon animates to an X but no menu opens; checking a filter, typing in a search box, or choosing a sort order changes the control's own appearance while the list never changes; a primary CTA button with hover/active styles produces no navigation, DOM change, network request, or console output on click; an entire composite surface (settings page, chat shell, dashboard) ships with every control inert; cards carry `cursor: pointer` and hover-lift effects with no handler anywhere in their subtree.
- **Likely implementation cause:** Widget state (`const [active, setActive] = useState(0)`) is consumed only in class-string logic and never selects rendered content — panels are a single hardcoded JSX block, or the `panels` array is never rendered conditionally; filter state is local to a sidebar and never lifted to the list component, which maps over the raw array; a `<button>` ships with no `onClick`, no enclosing form/Link, or its handler lives in a React Server Component without `"use client"` so the event prop never reaches the client; template styling (chevron, affordance classes) is copied while the wrapping link/handler is dropped.
- **Why it matters:** Content behind non-default tabs/panels is unreachable for every user — mouse, keyboard, and AT alike — and the primary task of filter/search pages cannot be completed; WAI-ARIA 1.2's `tab` role requires that an active tab have a corresponding tabpanel, and WCAG SC 4.1.2 defines the name/role/value contract a permanently inert control breaks; SC 4.1.3's own status-message examples ("Searching…", "18 results returned") presume a search that actually runs. Affordance without behavior is the purest interaction slop: the UI lies at the exact moment of interaction. Keyboard-contract details for widgets that *are* wired (arrow-key tablists, mega menus) — see Section 5.
- **Static detection:** Data-flow trace: every control-state variable must be read inside the expression that produces the rendered output (panel selection, derived list); `eslint no-unused-vars` flags state getters read nowhere. Grep the list render for `.filter(`/`.sort(`/`useMemo` next to visible filter UI. AST: `<button>` (or `Button` component) with no `onClick`, no `type="submit"` inside a form, no `commandfor`, not wrapped in `<Link>`/`<a href>`; stub handlers `onClick={() => {}}`; Next.js check: file containing `onClick` but missing the `"use client"` directive. Heuristics: identical/placeholder panel copy across tabs; an accordion component with no panel element in its JSX tree; `cursor-pointer`/`hover:-translate` classes with no handler, href, role, or tabIndex in the subtree.
- **Runtime or browser test:** Playwright click-everything sweep: attach request/DOM-mutation listeners, activate each control, assert some observable effect (URL change, network call, DOM mutation, dialog). Click every tab and assert distinct panel content (`expect(page.getByRole('tabpanel')).toContainText(<tab-specific string>)`); operate each filter/sort/search control and assert the list changes; typing a no-match query empties the list. DevTools → Elements → Event Listeners: no `click` listener on a "button" is a finding.
- **Corrective rule:** Every rendered element with interactive semantics or affordances MUST produce a truthful, observable effect on activation for mouse, keyboard, and AT alike. Activating a tab or accordion header MUST swap the associated panel (APG); filter/search/sort state MUST be consumed by the code path that computes the rendered list. Controls that are not wired MUST NOT ship — remove them, render them as inert text/badges, or label the surface as a demo. Click-affordance styling (`cursor: pointer`, hover lift, press states) MUST only appear on elements that are actually activatable.
- **Valid exceptions:** Explicitly labeled prototypes, Storybook stories, Figma embeds, and design-system documentation; `<a>` *without* `href` as the spec-sanctioned placeholder (HTML Standard §4.5.1); `<button disabled>` with the real `disabled` attribute during genuinely unavailable states; UI behind a feature flag that is off in production.
- **Primary sources:** APG Tabs Pattern — https://www.w3.org/WAI/ARIA/apg/patterns/tabs/ ; APG Accordion Pattern — https://www.w3.org/WAI/ARIA/apg/patterns/accordion/ ; HTML Standard, the `button` element — https://html.spec.whatwg.org/multipage/form-elements.html#the-button-element ; HTML Standard §4.5.1, the `a` element — https://html.spec.whatwg.org/multipage/text-level-semantics.html#the-a-element ; WAI-ARIA 1.2 (`tab` role) — https://www.w3.org/TR/wai-aria-1.2/ ; WCAG 2.2 SC 4.1.2 — https://www.w3.org/TR/WCAG22/#name-role-value ; React Server Components — https://react.dev/reference/rsc/server-components ; React docs, "You Might Not Need an Effect" — https://react.dev/learn/you-might-not-need-an-effect ; MDN `cursor` — https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Properties/cursor
- **Confidence:** High

### 2.2 Fake form submission with unconditional success feedback
- **Classification:** deceptive | objectively broken
- **Severity:** Critical
- **Observable symptom:** Submitting a form shows "Thanks! We'll be in touch", a success toast, or a confetti animation — but no network request fires, no data is stored, and reloading proves nothing happened; or the form posts to a failing endpoint and the UI shows success regardless of the error; variant: a spinner that never resolves.
- **Likely implementation cause:** `onSubmit={(e) => { e.preventDefault(); setSubmitted(true); }}` with no fetch/mutation; handler body stubbed to `console.log(values)` or a `// TODO`; `setTimeout(() => setSuccess(true), 800)` simulating latency left in production; `await fetch(...)` with no `res.ok`/status check so HTTP errors resolve as "success"; success state set outside/above the promise resolution.
- **Why it matters:** This converts a fake interaction into actual data loss plus a false confirmation — the worst severity tier of fake interaction. WCAG SC 3.3.1's Understanding document notes it is "not sufficient to only re-display the form without providing any hint that the submission failed"; SC 4.1.3's status-message examples ("Your form was successfully submitted") presume the message is true and requires it to be programmatically determinable.
- **Static detection:** AST/grep: `onSubmit` bodies containing only `preventDefault` + state set + optional `console.log`; success-toast strings and `setSubmitted(true)` traced backwards to no request; `fetch(` calls with no `res.ok`/status handling and no `.catch`; `setTimeout`-only success paths.
- **Runtime or browser test:** Playwright network interception: submit the form, assert exactly one mutation request fires and success UI appears only after a 2xx response; force a 500 (route abort or MSW) and assert an *error* message appears instead. DevTools Network tab: no request on submit is an automatic fail.
- **Corrective rule:** Submission UI MUST reflect the actual outcome of an actual operation: pending state during the request, success feedback only after confirmed success, error feedback (as an assertive status message) on failure. A form with no backend MUST be clearly framed as a demo or disabled — never fake success. Simulated handlers MUST NOT ship.
- **Valid exceptions:** Optimistic UI that is visually marked pending and genuinely rolls back with an honest error on failure; offline-first queues that communicate "will sync" status; client-only demos labeled as such.
- **Primary sources:** Understanding WCAG SC 3.3.1 — https://www.w3.org/WAI/WCAG22/Understanding/error-identification.html ; WCAG 2.2 SC 4.1.3 — https://www.w3.org/TR/WCAG22/#status-messages and Understanding — https://www.w3.org/WAI/WCAG22/Understanding/status-messages.html ; MDN `HTMLFormElement: submit event` — https://developer.mozilla.org/en-US/docs/Web/API/HTMLFormElement/submit_event ; HTML Standard §4.10.1.4 Client-side form validation — https://html.spec.whatwg.org/multipage/forms.html
- **Confidence:** High

### 2.3 Fabricated indicators: fake "live" data, hardcoded badges, and perpetual loading states
- **Classification:** deceptive | objectively broken
- **Severity:** High
- **Observable symptom:** "Live visitors: 1,024", revenue charts, and uptime graphs are identical on every load, account, and timezone; animated counters replay the same constants; a notification bell permanently shows badge "3" and opens nothing (or three undismissable hardcoded sample notifications); skeleton shimmer blocks, spinners, or progress bars render on initial paint and are never replaced by content — `animate-pulse` markup ships as the final UI.
- **Likely implementation cause:** Chart components fed by array literals in the same file; "metrics" defined as module constants with no fetch/SWR/react-query/subscription anywhere in the data path; badge markup (`<span className="badge">3</span>`) copied from a dashboard UI kit with no backing state; skeleton markup generated as the *only* state with no `isLoading` flag or fetch, `aria-busy="true"` hardcoded, or a `<Suspense fallback>` whose wrapped component never resolves.
- **Why it matters:** Fabricated metrics presented as fact can drive real user decisions — the strongest deception case in this cluster; a badge is a state claim ("you have N unread items") that is false on first render and forever; WCAG SC 4.1.3 exists so that "waiting state of an application" messages are programmatically determinable and — crucially — end: MDN defines `aria-busy` as `"true"` while an element is being updated and `"false"` when complete, so a perpetual `true` tells AT the region is never readable, and WAI-ARIA 1.2's `progressbar` role is defined as indicating that "the application is making progress" — a fake progressbar asserts progress that is not happening. Where the (fake) value exists only as presentation (canvas/SVG with no text alternative), SC 1.3.1 is also engaged.
- **Static detection:** AST/grep: chart components fed by literals in the same file; numeric "metrics" as module constants; strings like `Live`, `Real-time`, `Updated just now` adjacent to static JSX; numeric literals inside badge-classed spans; `animate-pulse`/`animate-spin`/`Skeleton`/`role="progressbar"`/`aria-busy="true"` with no corresponding state variable or fetch that can turn them off; loading JSX returned unconditionally rather than behind an `isLoading` conditional.
- **Runtime or browser test:** Reload twice and diff DOM values; check the Network panel for the absence of any metrics request; click the bell and assert items correspond to a data source; load the page, wait with network idle — any spinner/skeleton still visible is a finding (Playwright: `expect(skeleton).toBeHidden()` after load); screen reader check that regions are not stuck in busy state.
- **Corrective rule:** Any UI element labeled or styled as live/real-time data MUST be backed by a real, current data source with a visible timestamp or an explicit "sample data" label; hardcoded illustrative data MUST be labeled as sample/demo, and "Live" badges on static data are forbidden. Notification badges MUST derive from live state and the bell control MUST open a real surface or not be a control. Loading indicators MUST be driven by real asynchronous state and MUST have a guaranteed terminal state (content, error, or empty state); `aria-busy="true"` MUST be cleared on completion; hardcoded skeleton markup as final UI is forbidden.
- **Valid exceptions:** Clearly labeled demos, marketing mockups identified as such, Storybook fixtures, loading snapshots replaced by real data; honest onboarding empty states ("no notifications yet"); genuine short-lived indeterminate operations with real pending state; SSR streaming shells that resolve.
- **Primary sources:** WCAG 2.2 SC 4.1.3 — https://www.w3.org/TR/WCAG22/#status-messages ; Understanding SC 4.1.3 — https://www.w3.org/WAI/WCAG22/Understanding/status-messages.html ; MDN `aria-busy` — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Attributes/aria-busy ; WAI-ARIA 1.2 (`progressbar` role) — https://www.w3.org/TR/wai-aria-1.2/ ; WCAG 2.2 SC 1.3.1 — https://www.w3.org/TR/WCAG22/#info-and-relationships ; axe-core rule descriptions (`aria-progressbar-name`) — https://github.com/dequelabs/axe-core/blob/develop/doc/rule-descriptions.md
- **Confidence:** High (that unwired charts/badges/loaders are common and that "Live"-labeled statics are deceptive; Medium that any given unlabeled static chart is a defect — context-dependent)

### 2.4 Bogus link targets and link/button role confusion
- **Classification:** objectively broken | deceptive | inaccessible
- **Severity:** High
- **Observable symptom:** Clicking a nav item, card title, or "Learn more" link does nothing, appends `#` to the URL, or scrolls the page to the top; middle-click / open-in-new-tab yields the same page; a "Features" link with `href="#features"` does nothing because no element has `id="features"`; a blue underlined `cursor: pointer` "link" that Tab skips entirely (no `href`); a "button" that navigates cannot be middle-clicked and shows no destination on hover; a "link" that deletes a record cannot be activated with Space and its href performs a state-changing GET.
- **Likely implementation cause:** Placeholder hrefs stuffed to keep the anchor "valid": `<a href="#">`, `<a href="javascript:void(0)">`, `<a href="">`, `<Link href="#">`; LLM-invented fragment hrefs (`#about`, `#services`) never matched by rendered section IDs; `<a onClick>` with no `href` (implicit role becomes `generic`, not `link`); SPA click handlers that forget `event.preventDefault()`; navigation wired via `onClick` + `router.push()` on a `<button>` (also skipping `<Link>` prefetch in Next.js); mutating actions wired as GET links (`/delete?id=3`).
- **Why it matters:** An `<a>` *with* `href` is defined by HTML as a hyperlink — users, search engines, and AT treat it as navigation; `href="#"` is a real, misleading navigation to the current page's top. eslint-plugin-jsx-a11y's `anchor-is-valid` documentation classifies `#`, `javascript:void(0)`, and missing-href anchors as invalid: an anchor without a valid `href` "can no longer accept keyboard focus or be clicked on." WCAG failure technique F42 covers scripted link emulation (SC 1.3.1/2.1.1/4.1.2), bogus hrefs break copy/drag/open-in-new-tab/bookmark (MDN) and are blocked outright by a CSP banning `unsafe-inline`, SC 2.4.4 Link Purpose is undermined because the href communicates no destination, and GET-triggered mutations are unsafe (prefetchers and crawlers follow them). APG: "It is important that both the appearance and role of a widget match the function it provides."
- **Static detection:** ESLint `jsx-a11y/anchor-is-valid` (recommended config; shipped in Next.js's default lint setup). Grep: `href="#"`, `href={'#'}`, `javascript:void`, `href=""`, `<a(?![^>]*href=)[^>]*onClick`; cross-check every internal href against real route files or router config, and every `href="#x"` against a matching `id="x"`. AST heuristics: `onClick` containing `router.push`/`navigate(` on `button`/`div`; `<a href>` whose handler calls mutation APIs.
- **Runtime or browser test:** Playwright: for every nav anchor, assert a real URL change or an explicit widget behavior (`aria-expanded` toggles); for fragment links, assert `document.querySelector(location.hash)` is non-null after click; middle-click the navigation control and assert a new page opens; press Space on action "links" and observe the contract gap. Lighthouse "Links are not crawlable" audit flags invalid hrefs.
- **Corrective rule:** Elements that navigate MUST be anchors with valid hrefs (real routes, or fragments resolving to element IDs present in the target document); elements that perform actions MUST be `<button type="button">` (mutations via POST/fetch, never GET links). Placeholder href values (`#`, `javascript:void(0)`, empty string) on elements with activation handlers are prohibited in shipped UI. Visual styling MUST NOT determine element choice; a `<button>` styled to look like a link is the permitted compromise, never the reverse wiring.
- **Valid exceptions:** `<a>` with no `href` as a true placeholder per HTML Standard §4.5.1 (e.g., the current page's entry in a nav list, carrying no click handler); same-page fragment links targeting real IDs (skip links, table-of-contents anchors); legacy pre-v13 Next.js `<Link>` wrapping an href-less `<a>`.
- **Primary sources:** jsx-a11y `anchor-is-valid` — https://github.com/jsx-eslint/eslint-plugin-jsx-a11y/blob/main/docs/rules/anchor-is-valid.md ; HTML Standard §4.5.1 — https://html.spec.whatwg.org/multipage/text-level-semantics.html#the-a-element ; MDN `<a>` ("Accessibility → onclick events") — https://developer.mozilla.org/en-US/docs/Web/HTML/Element/a ; WCAG 2.2 SC 2.1.1 — https://www.w3.org/TR/WCAG22/#keyboard ; SC 2.4.4 — https://www.w3.org/TR/WCAG22/#link-purpose-in-context ; APG Button pattern — https://www.w3.org/WAI/ARIA/apg/patterns/button/ ; APG Link pattern — https://www.w3.org/WAI/ARIA/apg/patterns/link/ ; Next.js `<Link>` API reference — https://nextjs.org/docs/app/api-reference/components/link ; Next.js Architecture: Accessibility — https://nextjs.org/docs/architecture/accessibility
- **Confidence:** High

### 2.5 Mouse-only pseudo-controls: click handlers on static elements, pointer-exclusive wiring
- **Classification:** inaccessible | objectively broken
- **Severity:** Critical — WCAG Level A failure (SC 2.1.1, SC 4.1.2); applies equally to `<div onClick>` widgets, partially retrofitted `role="button"` elements, pager `<span>`s, and real buttons wired to `onMouseDown`/`onPointerDown` only.
- **Observable symptom:** The control works with a mouse but is skipped entirely by Tab; Enter/Space do nothing; screen readers announce it as plain text with no widget role; or a screen reader says "Save, button" but the element is not in the tab order / only Enter was wired; or the action fires on physical press but not on Enter/Space, screen-reader activation commands (which dispatch `click`), switch access, or `element.click()` in tests.
- **Likely implementation cause:** `<div className="btn" onClick={save}>` / `<span onClick>` / `<li onClick>` with no `role`, no `tabIndex`, no key handlers; `<div role="button" onClick={save}>` without `tabIndex={0}`, or `tabIndex` present but a keydown handler checking only `Enter`; `<button onPointerDown={doThing}>` with no `onClick` — MDN notes that even with `role="button"`, "`onclick` only fires when clicked by the mouse cursor" on non-button elements, and that on native buttons `click` is the event keyboard and AT activation converge on.
- **Why it matters:** WCAG SC 2.1.1 (Level A) requires all functionality to be keyboard-operable — failure F54 names pointing-device-specific handlers only; SC 4.1.2 failure F59 names "using script to make div or span a user interface control in HTML without providing a role"; the APG Button pattern requires both Space and Enter activation plus programmatic role and name. A role-only retrofit can additionally mask the bug from automated scans while remaining keyboard-dead.
- **Static detection:** eslint-plugin-jsx-a11y recommended config: `no-static-element-interactions`, `click-events-have-key-events`, `interactive-supports-focus`, `no-noninteractive-element-interactions`. Grep: `<(div|span|li)[^>]*onClick` where the element has no `role=`; `role="button"` occurrences lacking `tabIndex`; elements with `onMouseDown`/`onPointerDown` but no `onClick`. axe-core runtime: `focus-order-semantics`, `nested-interactive` context.
- **Runtime or browser test:** Unplug-the-mouse walkthrough: Tab through the page and confirm every actionable element is reachable, visibly focused, and Enter- and Space-operable (Playwright: Tab loop asserting `document.activeElement`, then `press('Enter')`/`press('Space')` asserting the action fired). Focus via Tab (not `.focus()`, which bypasses tab-order bugs). Screen reader: role announced as button/link, activation dispatches the handler.
- **Corrective rule:** Every element that performs an action on activation MUST be a native `<button type="button">` (or `<a href>` for navigation). Default activation MUST be wired to `click` (which the browser synthesizes for keyboard, AT, and pointer alike); `pointerdown`/`mousedown` handlers are permitted only as supplements. `role="button"` MUST only appear together with `tabindex="0"` (or a managed roving-tabindex scheme) and keyboard handlers covering both Space and Enter — a custom role-button requires a code-review justification.
- **Valid exceptions:** Event-delegation containers catching bubbled clicks from real `<button>`/`<a>` descendants (explicitly documented in `jsx-a11y/no-static-element-interactions`); items inside composite widgets using `tabindex="-1"` with APG arrow-key roving focus; drag-and-drop initiation legitimately starting on pointerdown — but then WCAG SC 2.5.7 requires a single-pointer alternative and SC 2.1.1 still applies; redundant mouse-only enhancements where the same function is available via a real control.
- **Primary sources:** WCAG 2.2 SC 2.1.1 — https://www.w3.org/TR/WCAG22/#keyboard ; Understanding SC 4.1.2 (failures F59, F42) — https://www.w3.org/WAI/WCAG22/Understanding/name-role-value.html ; APG Button pattern — https://www.w3.org/WAI/ARIA/apg/patterns/button/ ; MDN ARIA button role — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Roles/button_role ; WCAG 2.2 SC 2.5.7 — https://www.w3.org/TR/WCAG22/#dragging-movements ; jsx-a11y `no-static-element-interactions` — https://github.com/jsx-eslint/eslint-plugin-jsx-a11y/blob/main/docs/rules/no-static-element-interactions.md ; `click-events-have-key-events` — https://github.com/jsx-eslint/eslint-plugin-jsx-a11y/blob/main/docs/rules/click-events-have-key-events.md ; `interactive-supports-focus` — https://github.com/jsx-eslint/eslint-plugin-jsx-a11y/blob/main/docs/rules/interactive-supports-focus.md
- **Confidence:** High

### 2.6 Hover-gated menus, submenus, and tooltips with no keyboard or touch path
- **Classification:** inaccessible | objectively broken
- **Severity:** High — **Critical** when the hover-gated submenu is the sole navigation path to those pages.
- **Observable symptom:** Submenus appear on mouseover only: on a touch device the first tap either navigates away (submenu never seen) or does nothing; keyboard Tab never opens the submenu and focus either skips the hidden items or — worse — lands on visually hidden but focusable links; moving the mouse diagonally from trigger to submenu makes it vanish mid-trajectory (gap/instant-close); icon buttons show explanatory tooltips only on mouse hover while keyboard focus shows nothing, and moving the pointer toward the tooltip makes it disappear.
- **Likely implementation cause:** CSS-only `.nav li:hover > ul { display: block; }` (Tailwind `group-hover:block`) with no `:focus-within` variant, no toggle `<button>`, no `aria-expanded`; JS `onMouseEnter`/`onMouseLeave` with no `onClick`/`onFocus`/`onKeyDown` counterpart; submenu positioned with a `margin-top` gap and instant `mouseleave` close (no delay, no hover bridge, no safe triangle); pure-CSS tooltips (`.trigger:hover .tooltip`) or hover-only handlers with no focus trigger, no Escape handler, and a non-focusable `<span>` trigger.
- **Why it matters:** WCAG SC 2.1.1 (Level A) requires all functionality keyboard-operable, and the Understanding document for SC 1.4.13 explicitly notes hover-triggered content must also be triggerable by keyboard focus; touchscreens have no hover state at all. SC 1.4.13 (AA) additionally requires hover/focus-triggered content to be Dismissible (Escape), Hoverable (pointer can move onto it — failure F95 is exactly "content shown on hover not being hoverable"), and Persistent; the WAI Menus Tutorial prescribes a deliberate close delay (~1 s in its example) because mouse navigation is imprecise, and APG Tooltip requires focus trigger, Escape dismissal, and `aria-describedby` wiring.
- **Static detection:** Grep stylesheets for `:hover` selectors toggling submenu/tooltip visibility (`display`/`visibility`/`opacity`/`max-height`/transforms, Tailwind `group-hover:`) and require an adjacent `:focus-within` rule; grep JSX for `onMouseEnter`/`onMouseOver` on nav items lacking `onClick`/`onFocus`/`onKeyDown`; inspect submenu CSS for a physical gap with no padding/pseudo-element bridge; flag tooltip JSX containing interactive elements (that is a mis-roled dialog/popover).
- **Runtime or browser test:** Keyboard walkthrough: Tab to the parent item, press Enter/Space/Down Arrow — submenu must open and items must be reachable; Tab to a tooltip trigger — content must appear, Escape must dismiss it without moving focus. Playwright touch emulation (`hasTouch: true` + `page.tap`): first tap opens the submenu rather than navigating. `page.mouse.move(x, y, { steps: 20 })` along a diagonal from trigger to last submenu item: panel remains visible throughout. axe-core does not catch this class — flag as a manual-test gap.
- **Corrective rule:** Any submenu or information reveal that opens on pointer hover MUST also open via keyboard (a real `<button>` with `aria-expanded`, or a `:focus-within` fallback) and MUST be operable on touch (tap-to-toggle or a separate toggle button per the WAI Menus Tutorial "button as toggle" pattern); hover-triggered additional content MUST satisfy SC 1.4.13 — Dismissible via Escape, Hoverable along the real pointer path, and Persistent. Hover may exist only as progressive enhancement on top of these. Interactive content MUST NOT live in a tooltip (use a popover/dialog).
- **Valid exceptions:** Hover reveals that are purely redundant decoration (every hover-exposed destination also reachable without hover) — still subject to SC 1.4.13; native `title`-attribute tooltips (UA-controlled, explicitly excepted by SC 1.4.13); click-to-open menus (hover is not the trigger).
- **Primary sources:** WCAG 2.2 SC 2.1.1 — https://www.w3.org/TR/WCAG22/#keyboard ; Understanding SC 1.4.13 (Dismissible/Hoverable/Persistent, F95) — https://www.w3.org/WAI/WCAG22/Understanding/content-on-hover-or-focus.html ; W3C WAI Menus Tutorial, Fly-out functionality — https://www.w3.org/WAI/tutorials/menus/flyout/ ; APG Tooltip Pattern — https://www.w3.org/WAI/ARIA/apg/patterns/tooltip/
- **Confidence:** High

### 2.7 Menus and overlays without dismissal paths (no Escape, no outside click, no route-change reset)
- **Classification:** objectively broken | inefficient
- **Severity:** Medium — rises when `aria-expanded` is left stale (SC 4.1.2 state error, see 2.10) or focus is stranded behind a persistent panel.
- **Observable symptom:** A dropdown stays open while the user Tabs to page content, clicks elsewhere, or navigates to a new route — an orphaned panel overlays the new page; Escape does nothing; a mobile menu opens once and can never close; keyboard focus, having been inside the menu, lands on content visually hidden behind the still-open panel.
- **Likely implementation cause:** No `keydown` listener for `Escape`; no document-level `pointerdown` listener with a `ref.contains(event.target)` check; no subscription to router changes (e.g., a `useEffect` keyed on `usePathname()` in Next.js) resetting open state; open state held in a persistent layout component so it survives page transitions by default; document listeners leaked by missing effect cleanup.
- **Why it matters:** The APG Menu/Menubar keyboard interaction defines "Escape: Close the menu that contains focus and return focus to the element or context… from which the menu was opened", and Tab/Shift+Tab moving focus out "close all menus and submenus"; SC 1.4.13 Dismissible requires a mechanism to dismiss hover/focus-triggered content without moving pointer or focus. An orphaned overlay after client-side navigation is an objectively broken UI state for every user.
- **Static detection:** Checklist review of each menu/overlay component for three close paths: (1) `Escape` keydown handler, (2) outside-pointer handler, (3) route-change reset — absence of any is a finding. Grep for `useEffect` cleanup removing document listeners.
- **Runtime or browser test:** Playwright: open menu → `page.keyboard.press('Escape')` → panel hidden AND focus back on the trigger; open menu → click page body → hidden; open menu → click a menu link → after navigation, no visible panel and trigger `aria-expanded="false"`.
- **Corrective rule:** Any open menu or non-modal overlay MUST close on Escape (returning focus to its trigger), on pointer activation outside it, and on route change; `aria-expanded` MUST be updated in the same commit as the visual state. (Dialog-specific dismissal defects — missing Escape on modals, backdrop-click data loss — are covered in 2.9.)
- **Valid exceptions:** Visually persistent navigation that overlays nothing (APG: "a menu that is visually persistent is a menubar") — but then it must not be positioned over page content.
- **Primary sources:** APG Menu and Menubar Pattern, Keyboard Interaction — https://www.w3.org/WAI/ARIA/apg/patterns/menubar/ ; APG Menu Button Pattern — https://www.w3.org/WAI/ARIA/apg/patterns/menu-button/ ; Understanding SC 1.4.13, Dismissible — https://www.w3.org/WAI/WCAG22/Understanding/content-on-hover-or-focus.html
- **Confidence:** High

### 2.8 Modal dialogs without focus containment, initial focus, or focus restoration
- **Classification:** inaccessible | objectively broken
- **Severity:** Critical — a modal that leaks focus is not a modal; keyboard and screen-reader users can operate hidden background controls, producing invalid state.
- **Observable symptom:** With the modal open, pressing Tab repeatedly moves the focus ring out of the dialog onto links/buttons in the dimmed background page; opening a modal with the keyboard leaves the focus ring on the (now hidden) trigger or on `body`, and the next Tab starts from the top of the background document; closing via Esc, close button, or backdrop dumps focus at the document top, so repeated modal use (open → fill → close → reopen) is effectively unusable by keyboard.
- **Likely implementation cause:** Dialog rendered as an absolutely positioned `div` with a semi-transparent overlay and no focus-trap logic, no `inert` on the app root, no native `<dialog>`; React/Vue conditional render (`{open && <Modal/>}`) with no `autoFocus`, no `ref` + `.focus()` effect, no `tabindex="-1"` fallback target; no stored reference to `document.activeElement` at open time and no `.focus()` call after unmount (React silently drops focus to `body` when the focused node unmounts).
- **Why it matters:** The APG Dialog (Modal) pattern requires that modal dialogs contain their tab sequence ("Tab and Shift+Tab do not move focus outside the dialog"), that "when a dialog opens, focus moves to an element inside the dialog" (with documented placement heuristics, including least-destructive action for irreversible flows), and that "when a dialog closes, focus returns to the element that invoked the dialog"; APG further warns that `aria-modal="true"` on a dialog that does not enforce modality has "severe negative ramifications" for AT users. Related to WCAG SC 2.4.3 (focus order must preserve meaning and operability).
- **Static detection:** Grep for `role="dialog"`/modal markup and require one of: a focus-trap library (`focus-trap-react`, `react-focus-lock`, `useFocusTrap`), an `inert` attribute on the page root, or native `<dialog>` + `showModal()` — absence of all in a custom modal is a fail; check for an initial-focus mechanism (`autoFocus`, mount `.focus()` effect, `<dialog autofocus>`); check the close handler for `triggerRef.current?.focus()` / `previousActiveElement.focus()` / `returnFocusOnDeactivate`.
- **Runtime or browser test:** Manual/Playwright: open modal, hold Tab 20 times — `document.activeElement` must always be inside the dialog; after activating the trigger, assert focus is inside the dialog (not `body`, not the trigger); record the trigger, open dialog, press Escape, assert `document.activeElement === trigger`. axe-core cannot fully detect this — scripted keyboard tests are required.
- **Corrective rule:** A modal dialog MUST constrain sequential keyboard focus within itself while open (focus trap or native `<dialog>`/`showModal()`), with Tab/Shift+Tab wrapping; on open, focus MUST move to an element inside the dialog per APG placement heuristics; on every close path (close button, Escape, backdrop click, programmatic close), focus MUST return to the invoking element or a documented APG-exception successor.
- **Valid exceptions:** Non-modal dialogs/side panels that intentionally allow page interaction — then they MUST NOT claim `aria-modal="true"`, MUST NOT block pointer interaction with the page, and focus order must remain logical (F85); invoking element removed from the DOM (focus a logical workflow element instead).
- **Primary sources:** APG Dialog (Modal) Pattern — https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/ ; Understanding SC 2.4.3 Focus Order (F85, H102) — https://www.w3.org/WAI/WCAG22/Understanding/focus-order.html ; MDN `<dialog>` (browser-provided inertness via `showModal()`, `autofocus`) — https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Elements/dialog
- **Confidence:** High

### 2.9 Modal dialogs over live backgrounds: missing Escape, scroll chaining, stacked-modal chaos, and light-dismiss data loss
- **Classification:** objectively broken | inaccessible | deceptive
- **Severity:** High
- **Observable symptom:** Pressing Escape with a modal open does nothing; the page behind the overlay scrolls under wheel/touch and scroll chaining "steals" scroll once the dialog's own boundary is reached; mouse users click background links through overlay gaps; opening a confirmation from within an edit dialog renders it underneath, stacks both backdrops, or Escape closes both at once with focus left in limbo; a stray backdrop click instantly discards a long filled-in form.
- **Likely implementation cause:** Overlay markup wires only `onClick` handlers — no `Escape` keydown listener, or `<dialog>` opened non-modally with `show()` (no Esc behavior by default); no `inert`/`aria-hidden` on the app root and no `overflow: hidden` toggle with cleanup on `body`; modal tree nested inside a `transform`ed/filtered ancestor, breaking `position: fixed`; each stacked modal hardcodes the same `z-index` and its own global keydown listener with no LIFO stack manager; backdrop `onClick={() => setOpen(false)}` wired unconditionally with form state destroyed on unmount — no dirty-tracking, no confirmation.
- **Why it matters:** APG defines Escape as closing the dialog and states that "windows under a modal dialog are inert"; MDN documents that `showModal()` renders the rest of the page inert, that non-modal dialogs do not dismiss via Esc, and that with multiple modal dialogs "pressing the Esc key should close only the last shown dialog" — a behavior `<dialog>` provides for free. WCAG SC 2.1.2's modal example treats keyboard dismissal as the mechanism preventing an effective trap; SC 3.3.4 (Error Prevention) requires that data-loss-prone flows be reversible, checked, or confirmed; MDN's `closedby` attribute (`any`/`closerequest`/`none`) shows light dismiss is a deliberate, graded choice, not a default.
- **Static detection:** Grep modal components for `Escape`/`keydown` handlers or library props (`onEscapeKeyDown`, `closeOnEsc`); verify `<dialog>` is opened with `showModal()` not `show()`; check for `inert` on the app root, body scroll-lock with cleanup, and modal JSX nested inside `transform`/`filter`/`perspective` ancestors (flag); search for global `keydown` listeners without a stack/depth check and duplicated `z-index` constants across modal call sites; grep backdrop click handlers that close unconditionally in components containing `<form>`/`<input>`/`<textarea>`.
- **Runtime or browser test:** Playwright: open dialog, press Escape — dialog closes and focus is restored; wheel over the overlay — background `scrollTop` must not change; `document.elementFromPoint()` on a background link location must return the backdrop; open modal B from modal A — Esc closes only B and focus returns into A, Tab never reaches A's controls while B is open; type into a dialog form, click the backdrop — the dialog stays open, a confirmation appears, or the data survives reopening. Verify the page is scrollable again after close (cleanup regression).
- **Corrective rule:** While a modal is open, all background content MUST be inert to pointer, keyboard, and AT (native `<dialog>`/`showModal()`, `inert`, or an APG-compliant `aria-hidden` fallback with verified focus containment), and background scrolling SHOULD be suppressed with reliable cleanup. Modals MUST close on Escape and expose a visible keyboard-focusable close control. Stacked overlays MUST be managed as a LIFO stack: only the topmost layer receives keyboard input, Escape affects only it, and closing restores focus to the layer below. If a dialog contains user-entered state, pointer-outside dismissal and Escape MUST be disabled or gated behind dirty-check + explicit confirmation — entered data MUST NOT be silently destroyed.
- **Valid exceptions:** Deliberately non-dismissible flows (mandatory wizard steps) with an explicit cancel control; non-modal drawers/sheets that coexist with page interaction (no `aria-modal`, no backdrop blocking); dialogs with no user input, where light dismiss is harmless; native `<dialog>` with sequential `showModal()` calls or a vetted library with nested-dialog support satisfies the stacking rules.
- **Primary sources:** APG Dialog (Modal) Pattern — https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/ ; MDN `<dialog>` (`showModal()` inertness, Esc behavior, multiple modal dialogs, `closedby`) — https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Elements/dialog ; Understanding SC 2.1.2 No Keyboard Trap — https://www.w3.org/WAI/WCAG22/Understanding/no-keyboard-trap.html ; WCAG 2.2 SC 3.3.4 — https://www.w3.org/TR/WCAG22/#error-prevention-legal-financial-data ; MDN `inert` — https://developer.mozilla.org/en-US/docs/Web/HTML/Global_attributes/inert ; MDN `overscroll-behavior` — https://developer.mozilla.org/en-US/docs/Web/CSS/overscroll-behavior ; MDN `popover` (`manual` popovers cannot be light dismissed) — https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Global_attributes/popover
- **Confidence:** High

### 2.10 Programmatic state contradicts visual and behavioral reality (stale ARIA, cosmetic switches, fake disabled)
- **Classification:** deceptive | inaccessible
- **Severity:** High
- **Observable symptom:** A screen reader announces "selected" on the wrong tab, never announces "collapsed/expanded" on an accordion, or announces expanded while the panel is visibly closed; a toggle knob slides but page behavior, stored preferences, and server state never change (on reload the switch reverts); the expanded panel's header looks clickable but re-click does nothing with no disabled signal; a control announced as "dimmed/unavailable" still fires its handler; a greyed-out `opacity-50 cursor-not-allowed` button still submits on click and keyboard; a "disabled" div-button still responds because the attribute is inert on non-form elements.
- **Likely implementation cause:** `aria-expanded="true"` / `aria-selected` hardcoded as JSX string literals or set once at mount and never rebound; the state variable drives the CSS class but not the ARIA attribute; switch state wired only to the knob's visual class with no `onChange` side effect, persistence, or feature gating (or pure-CSS `peer-checked:` with no JS); single-open accordion handlers that early-return on the active item without setting `aria-disabled="true"`; `aria-disabled="true"` added with no guard in the handler — MDN is explicit that it "**only** semantically exposes these elements as being disabled" and that `pointer-events: none` "does not prevent the element from being activated via the keyboard"; `disabled` placed on `<div>`/`<span>`/`<a>`, where the HTML Standard limits it to form-associated elements.
- **Why it matters:** WCAG SC 4.1.2 requires user-settable states to be programmatically set with notification of changes — a wrong state announcement is worse than none because it misleads; WAI-ARIA 1.2 defines `switch` as on/off input with `aria-checked` **required**, and the APG Switch pattern requires Space (optionally Enter) to change state with the label unchanged; APG Accordion prescribes `aria-disabled="true"` when the expanded panel may not be collapsed. A switch whose state exists only in a CSS class fails AT users silently and sighted users deceptively (they believe a privacy/notification setting changed); a lying disabled state can trigger actions the user was told were unavailable. Disabled submit buttons that gate the whole form additionally hide validation errors from every user — SC 3.3.1 requires detected errors to be identified in text.
- **Static detection:** Grep for ARIA state literals: `aria-selected="true"`, `aria-expanded="true"`, `aria-checked="false"` written as constants; the review rule is *three consumers, one source of truth* — the expression feeding `aria-selected`/`aria-expanded`/`aria-checked` must reference the same state variable that drives the visual class and panel visibility. AST: switch/checkbox whose state value is never read outside style computation; `aria-disabled="true"` on elements whose handler has no early-return guard; `cursor-not-allowed`/`opacity-50`/`pointer-events-none` "disabled" recipes without the native `disabled` attribute on form controls; `disabled` prop on non-form elements; `<button disabled={!isValid}>` with no sibling error-rendering logic.
- **Runtime or browser test:** Playwright: click trigger → `expect(button).toHaveAttribute('aria-expanded', 'true')`, click again → `'false'`; assert exactly one tab has `aria-selected="true"` after each activation; toggle a switch and assert the promised effect (class/storage/network call) plus announced state change; attempt click + Enter/Space on every disabled-looking control and assert nothing fires in any modality; fill a form with one deliberate error against a disabled submit and check whether any text explains the blocker.
- **Corrective rule:** Every state change rendered visually MUST be reflected in the same render pass in the corresponding ARIA state attribute. Every switch/toggle MUST expose role + state, be keyboard-operable, and drive a real, verifiable effect or persist its state — a switch controlling nothing MUST NOT ship. If one panel must stay expanded, its header button MUST carry `aria-disabled="true"` with matching non-interactive styling. Use the native `disabled` attribute on form controls; `aria-disabled="true"` (for focus-retained cases) MUST be paired with behavior suppression for all input modalities; `cursor: not-allowed` MUST only accompany genuinely non-operable controls; submit buttons SHOULD remain enabled so activation can surface text errors per SC 3.3.1.
- **Valid exceptions:** Native `<details>`/`<summary>` (the browser exposes expanded state natively); switches inside a form whose state is submitted and handled elsewhere; local UI-only toggles that genuinely change local rendering (a dark-mode class *is* the behavior); `aria-disabled` on a focusable control whose action is genuinely suppressed (MDN-documented pattern); disabling during an in-flight submission with a visible pending indicator.
- **Primary sources:** Understanding WCAG SC 4.1.2 — https://www.w3.org/WAI/WCAG22/Understanding/name-role-value.html ; WAI-ARIA 1.2 (`switch`, `#aria-selected`, `#aria-expanded`) — https://www.w3.org/TR/wai-aria-1.2/ ; APG Switch Pattern — https://www.w3.org/WAI/ARIA/apg/patterns/switch/ ; APG Accordion Pattern (`aria-disabled` bullet) — https://www.w3.org/WAI/ARIA/apg/patterns/accordion/ ; APG Menu Button Pattern (`aria-expanded` synchronization) — https://www.w3.org/WAI/ARIA/apg/patterns/menu-button/ ; MDN `aria-disabled` — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Attributes/aria-disabled ; MDN `<button>` (`disabled`) — https://developer.mozilla.org/en-US/docs/Web/HTML/Element/button ; Understanding WCAG SC 3.3.1 — https://www.w3.org/WAI/WCAG22/Understanding/error-identification.html
- **Confidence:** High

### 2.11 Broken form mechanics: no real `<form>`, missing `preventDefault`, accidental submits, Enter does nothing
- **Classification:** objectively broken | inaccessible
- **Severity:** High
- **Observable symptom:** A data-entry UI with fields and a "Submit" button where pressing Enter inside a field does nothing and autofill/password-manager prompts never trigger; clicking Submit reloads the whole page, the URL gains `?email=...&password=...` (sometimes leaking credentials into history), and all client state is wiped; clicking an auxiliary button ("Add tag", "Remove row") inside a form reloads the page, clears controlled state, or fires a real POST.
- **Likely implementation cause:** Controls rendered in plain `<div>`s with submission wired to `<button type="button" onClick={save}>` or `<div onClick>` instead of `<form onSubmit>`; an `onSubmit` handler that never calls `e.preventDefault()` — React documents that "by default, the browser will send the form data to the current URL and refresh the page"; `<button onClick={addRow}>` inside a `<form>` with no `type` attribute, so the HTML missing-value default (`submit`) fires — MDN warns to set `type="button"` on non-submitting buttons or they "will try to submit form data and to load the (nonexistent) response, possibly destroying the current state of the document"; `onKeyDown` handlers that `preventDefault()` Enter without re-triggering submission, or programmatic `form.submit()` (which bypasses the `submit` event and constraint validation).
- **Why it matters:** The platform ties essential behavior to the `<form>` element: the `submit` event fires on the form including when "the user presses `Enter` while editing a field" (MDN; HTML Standard §4.10.21.2 implicit submission), native constraint validation runs at submission, and browsers gate autofill/password-manager integration on forms — a div-onClick discards all of it and can strand keyboard users entirely (SC 2.1.1). An uncancelled native submission destroys SPA runtime state and leaks field values into URLs, server logs, and history when `method` defaults to GET.
- **Static detection:** AST/grep: inputs present with no `<form>` ancestor; `onClick` handlers containing `fetch(`/mutation calls on elements that are not `type="submit"`; buttons labeled Submit/Save/Log in that are not `type="submit"` inside a form; every `onSubmit` handler's first statement should be `event.preventDefault()` (or the form uses a framework action); `eslint-plugin-react` rule `react/button-has-type`; grep `<button(?![^>]*type=)`; `onKeyDown` handlers testing `key === 'Enter'` that preventDefault without calling `requestSubmit()`.
- **Runtime or browser test:** Keyboard-only: focus each text field, press Enter — submission must occur (Playwright: `page.getByLabel('Email').press('Enter')` then expect the mutation route called). Fill and click Submit: `page.url()` unchanged and no document request fires. Fill a form, click each auxiliary button inside it, assert no navigation and no request. Network tab: a document request with a query string on submit is an automatic fail.
- **Corrective rule:** Any set of controls that submits user data MUST be wrapped in a `<form>` with an `onSubmit` handler and MUST contain a `<button type="submit">`; the handler MUST call `event.preventDefault()` before async work unless a framework action mechanism cancels the default. Every `<button>` MUST declare an explicit `type`; buttons that do not submit MUST use `type="button"`. Forms MUST submit on Enter from any single-line text control; if Enter is intercepted legitimately, the handler MUST trigger `form.requestSubmit()` (which, unlike `form.submit()`, fires `submit` and runs validation).
- **Valid exceptions:** Live filters/calculators/search-as-you-type that submit nothing; non-submission actions (Cancel, Delete row); Next.js Server Actions (`<form action={serverAction}>`) and server-rendered multi-page apps where navigation semantics are intentional (then `method="post"` and a real `action` MUST be explicit); GET search forms where a shareable query URL is the goal; `<textarea>` (Enter inserts a newline) and IME composition in progress.
- **Primary sources:** MDN `HTMLFormElement: submit event` — https://developer.mozilla.org/en-US/docs/Web/API/HTMLFormElement/submit_event ; HTML Standard §4.10.3 The `form` element — https://html.spec.whatwg.org/multipage/forms.html#the-form-element ; HTML Standard §4.10.21.2 Implicit submission — https://html.spec.whatwg.org/multipage/form-control-infrastructure.html#implicit-submission ; React `<input>` reference ("Reading the input values when submitting a form") — https://react.dev/reference/react-dom/components/input ; MDN `<button>` (`type` default and Notes warning) — https://developer.mozilla.org/en-US/docs/Web/HTML/Element/button ; eslint-plugin-react `button-has-type` — https://github.com/jsx-eslint/eslint-plugin-react/blob/master/docs/rules/button-has-type.md ; MDN `autocomplete` — https://developer.mozilla.org/en-US/docs/Web/HTML/Attributes/autocomplete ; WCAG 2.2 SC 2.1.1 — https://www.w3.org/TR/WCAG22/#keyboard
- **Confidence:** High

### 2.12 No single-flight submission guard: duplicate mutations and missing pending state
- **Classification:** objectively broken | inefficient
- **Severity:** High
- **Observable symptom:** After clicking Submit nothing visible changes for 300–3000 ms; the user clicks again (or mashes the button); the server receives N identical mutations — duplicate orders, messages, charges; DevTools shows two or more identical POSTs milliseconds apart.
- **Likely implementation cause:** Submit handler fires `fetch` with no in-flight guard; the button is not disabled and no `isSubmitting`/`isPending` state exists (React 18 `useTransition`/`useFormStatus` never used); no spinner or label change; no idempotency key on the request; mutation libraries used without reading `isPending`.
- **Why it matters:** Duplicate mutation is a data-integrity defect; for checkout/legal/financial flows it undermines WCAG SC 3.3.4's error-prevention intent (submissions should be checkable/reversible/confirmed). Users reasonably interpret a static UI as a failed click and retry — the missing pending state manufactures its own trigger.
- **Static detection:** AST/grep: submit handlers without any `isSubmitting`/`isPending`/boolean guard; `<button type="submit">` with no `disabled=` binding in a form performing async work; absence of both a pending flag and an idempotency key.
- **Runtime or browser test:** DevTools Network throttled to Slow 3G: double-click Submit, count requests (must be 1). Playwright: `Promise.all([btn.click(), btn.click()])` → assert a single intercepted request; assert the button is `disabled`/`aria-disabled` while pending and a progress indicator is exposed.
- **Corrective rule:** Submission MUST be single-flight: set a pending state immediately, disable the submit control (or otherwise block re-entry), and show progress feedback; re-enable on settle. For payment/creation endpoints, also send an idempotency key. Disable only during the in-flight request — do NOT keep submit disabled as a validation strategy (see 2.10 for the disabled-submit error-hiding failure).
- **Valid exceptions:** Pure read operations (search GETs) where duplicates are harmless; server-side idempotency making client guards redundant (keep the pending feedback regardless); mutation libraries that already dedupe in-flight identical requests.
- **Primary sources:** HTML Standard §4.10.3 The `form` element — https://html.spec.whatwg.org/multipage/forms.html#the-form-element ; WCAG 2.2 SC 3.3.4 — https://www.w3.org/TR/WCAG22/#error-prevention-legal-financial-data ; Vercel Labs Web Interface Guidelines (industry guidance, labeled as such) — https://github.com/vercel-labs/web-interface-guidelines
- **Confidence:** High

### 2.13 Pagination and in-place list navigation defects: phantom pages, dropped focus, fake pagers
- **Classification:** objectively broken | inaccessible | deceptive
- **Severity:** High — **Critical** when the pager is keyboard-inoperable and the only path to content beyond page 1 (WCAG SC 2.1.1/4.1.2, both Level A).
- **Observable symptom:** User on page 8 applies a filter matching 3 items → the view stays on "page 8" of a 1-page result set: blank area or a false "No results found" while matches exist on page 1; clicking "3" at the bottom of a long list leaves the viewport at the pager and focus falls to `<body>` (the activated page button was unmounted by re-keying), so the next Tab starts from the document top; page numbers render as buttons/links but clicking any of them keeps the same list; breadcrumb segments are styled as links but inert or pointing at `#`, with `aria-current="page"` hardcoded.
- **Likely implementation cause:** `setFilter(x)` without an accompanying `setPage(1)` — page and filter state managed in separate, uncoupled handlers (server-side variant: the request still sends the old `page` param); page-change handlers with no focus target (`headingRef.current.focus()` with `tabIndex={-1}`) and no scroll adjustment; the pager re-keyed (`key={page}`) so the activated button is destroyed; pagination/breadcrumb markup shipped from table templates with literal page numbers (`[1,2,3]`) and no state or routes behind them; current page conveyed by styling only, with no `aria-current="page"` and no `<nav aria-label="Pagination">`.
- **Why it matters:** A phantom empty page makes the UI assert an empty result set that is false — users conclude the item they want does not exist. WCAG SC 2.4.3 (Level A) requires focus order preserving meaning and operability — focus dropped to `<body>` or stranded on a control whose context was replaced breaks it; SC 2.1.1 requires keyboard-operable pagination; WAI-ARIA defines `aria-current="page"` precisely for "the current page within a set of pages" and WCAG technique ARIA26 (SC 1.3.1) prescribes it — a hardcoded `aria-current` misinforms AT about where the user is. Related implementation-level defects in the same data path (derived-list-in-state syncing effects, fetch races on criteria change, impure comparators) — see Section 6; result-count status announcements — see Section 5.
- **Static detection:** Grep every filter/sort/search setter call site and check a page reset occurs in the same handler (or the page is clamped from the recomputed count); grep the page-change handler for `.focus(` and `scrollTo|scrollIntoView` (absence is the smell) and check the results heading has `tabIndex={-1}`; grep pager markup for `onClick` on non-`<button>`/`<a>` elements, missing `aria-current`, and literal page-number arrays with unwired handlers.
- **Runtime or browser test:** Playwright: navigate to the last page, apply a highly restrictive filter — assert matches render, the pager indicates page 1, and no false "no results" element appears; Tab to "Next page", press Enter — `document.activeElement` must be inside the results region or on the persisted pager (never `body`), and the list top must be scrolled into view; click page 2 — list contents or URL change and `aria-current` moves truthfully; keyboard walk: every page control reachable and Enter/Space-operable.
- **Corrective rule:** Any change to filter, search, or sort criteria MUST reset pagination to the first page (or clamp the current page to the recomputed page count in the same update). After an in-place page change, focus MUST move to the results region (e.g., its heading with `tabindex="-1"`) or remain on the activated control, the new content MUST be scrolled into view, and focus MUST NOT be dropped to `body` by unmounting the focused element. Pagination controls MUST be native `<a>` (real hrefs) or `<button>`; the current-page item MUST carry `aria-current="page"`; breadcrumbs MUST reflect the real hierarchy with working links on every non-current segment. If only one page exists, pagination MUST be omitted.
- **Valid exceptions:** Cursor-based append UIs where "page" is not user-addressable; pagination implemented as real navigation (links with hrefs causing document load — the browser then handles scroll/focus); per MDN, `aria-current` optional when the current page is rendered as non-interactive text; breadcrumb current-page item as an href-less `<a>` placeholder per HTML §4.5.1.
- **Primary sources:** Understanding SC 2.4.3 Focus Order (SCR26; failures F44/F85) — https://www.w3.org/WAI/WCAG22/Understanding/focus-order.html ; WCAG 2.2 SC 2.1.1 — https://www.w3.org/TR/WCAG22/#keyboard ; SC 4.1.2 — https://www.w3.org/TR/WCAG22/#name-role-value ; SC 1.4.1 — https://www.w3.org/TR/WCAG22/#use-of-color ; MDN `aria-current` — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Attributes/aria-current ; WCAG SC 1.3.1 (technique ARIA26) — https://www.w3.org/TR/WCAG22/#info-and-relationships ; HTML Standard §4.5.1 — https://html.spec.whatwg.org/multipage/text-level-semantics.html#the-a-element ; React docs, "You Might Not Need an Effect" — https://react.dev/learn/you-might-not-need-an-effect
- **Confidence:** High

### 2.14 Auto-advancing carousels without pause/stop/hide — and live-region bombardment while rotating
- **Classification:** inaccessible | objectively broken
- **Severity:** Critical — WCAG 2.2 SC 2.2.2 is Level A, and its Note 2 invokes Conformance Requirement 5 (Non-Interference): unstoppable moving content can make the *whole page* fail for affected users.
- **Observable symptom:** A hero/testimonial carousel advances every 3–5 seconds indefinitely with no pause button; content keeps moving while the user tries to read adjacent text; Tabbing to a link inside the current slide does not stop rotation — the focused slide rotates away; with a screen reader running, every automatic advance triggers an unsolicited announcement of new slide content; some implementations pause only while hovered and resume the instant the pointer moves.
- **Likely implementation cause:** `setInterval(next, 3000)` with no rendered control; a library configured with `autoplay: true` (Swiper Autoplay module, Embla Autoplay plugin, slick `autoplay: true`) without rendering its play/pause toggle — Splide's v4.0.0 release notes list a play-pause toggle as part of following the W3C Carousel Design Pattern, evidence the toggle is the documented companion of autoplay; no `focusin`/`mouseenter` gating of the timer; the slide wrapper carrying `aria-live="polite"` while auto-rotating, where APG specifies the wrapper's `aria-live` must be `off` during auto-rotation and `polite` only when user-driven.
- **Why it matters:** SC 2.2.2 requires a mechanism to pause, stop, or hide moving/auto-updating content that starts automatically, lasts more than five seconds, and is presented in parallel with other content (failure F16 covers scrolling content without a pause mechanism); its Understanding document explicitly rejects hover-only pausing ("stop only so long as a user has focus on it… would not be considered a 'mechanism for the user to pause'"). The APG Carousel pattern requires a rotation control — a button for stopping and restarting rotation, placed first in the carousel's Tab sequence — and requires rotation to stop on focus or hover and not resume without user request; MDN documents that live regions speak changes, so `polite` on a 3-second rotator produces a non-stop announcement stream.
- **Static detection:** Grep `setInterval(|autoplay|autoPlay|auto_rotate` in carousel code and verify a rotation-control button exists whose accessible label toggles ("Stop slide rotation"/"Start slide rotation"); grep for `aria-live` — flag any non-`off` value not conditioned on a `playing` state; check for pause handlers on focus/hover *in addition to* the explicit control.
- **Runtime or browser test:** Load the page, wait >5 s: movement continues with no available stop control → fail; activate the control — rotation stops and does not resume unless re-activated; Tab into the carousel — rotation stops; Tab out — rotation stays stopped; screen reader on, autorotation on: count announcements per minute while idle elsewhere (should be zero).
- **Corrective rule:** Any automatically rotating carousel MUST provide a keyboard-operable pause/stop control (the APG rotation control, first in the carousel's tab order), and rotation MUST NOT resume except on explicit user request; rotation MUST stop when keyboard focus enters the carousel and while the pointer hovers it; hover/focus pausing MUST supplement — never replace — the explicit stop control; while auto-rotating, the slide container's `aria-live` MUST be `off` (a user-driven carousel MAY use `aria-live="polite"` with `aria-atomic="false"`).
- **Valid exceptions:** Movement essential to the activity; preload-phase animation where interaction is impossible and progress indication is needed (SC 2.2.2 Note 4); content that stops by itself within 5 seconds; a full-page interstitial with no parallel content.
- **Primary sources:** Understanding SC 2.2.2 Pause, Stop, Hide (SC text; non-interference Note 2; failure F16) — https://www.w3.org/WAI/WCAG22/Understanding/pause-stop-hide.html ; APG Carousel Pattern (rotation control; conditional `aria-live`) — https://www.w3.org/WAI/ARIA/apg/patterns/carousel/ ; Splide v4.0.0 release notes — https://github.com/Splidejs/splide/releases ; MDN ARIA Live regions — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Guides/Live_regions
- **Confidence:** High

### 2.15 Carousel navigation and slide visibility failures: swipe-only, dead dots, focusable offscreen slides
- **Classification:** inaccessible | objectively broken
- **Severity:** Critical — the only means of reaching offscreen slides is inoperable or unnameable for keyboard/AT users (WCAG SC 2.1.1, 2.5.1, 4.1.2 — all Level A).
- **Observable symptom:** On desktop, dragging the slide sideways changes slides but there are no arrows or dots (or they are `display: none` outside mobile breakpoints); keyboard focus never lands on anything in the carousel; screen readers announce "button" (or SVG gibberish) for chevron arrows; dots are skipped by Tab and clicking them does nothing — the active dot differs only by a slightly darker gray; after "next", Tab takes focus somewhere offscreen with no visible indicator, and the user Tabs blindly through N hidden slides' links; screen readers either read all slides as one flat run-on list or skip slide content entirely.
- **Likely implementation cause:** Only touch/pointer delta handlers implemented (`onTouchStart/Move/End`, pointer-down + translate math); prev/next buttons never generated or hidden by a media query; `<button class="..."><ChevronLeft /></button>` with no `aria-label`; dots as `<span>`/`<li>` with an `active` class and no `onClick`, `role`, `tabindex`, `aria-current`/`aria-selected`; track pattern `.track { display: flex; transform: translateX(-${index * 100}%) }` keeping all slides mounted and focusable with no `inert`, `visibility: hidden`, or per-slide `tabindex` management — or the inverse: blanket `aria-hidden="true"` on slide containers that still contain focusable elements, which WAI-ARIA 1.2 prohibits and axe-core's `aria-hidden-focus` rule maps to SC 4.1.2.
- **Why it matters:** SC 2.5.1 Pointer Gestures (Level A) requires functionality operated by path-based gestures to be operable with a single pointer without the gesture (failure F105) — its Understanding document's own example is a swipe carousel that must also offer forward/backward buttons, and it states keyboard operability alone does **not** satisfy 2.5.1; SC 2.1.1 independently requires keyboard operation; SC 4.1.2 requires names and states to be programmatically determinable and SC 1.4.1 prohibits conveying the active dot by color alone; the APG Carousel pattern describes the screen-reader disorientation caused by incorrectly hidden slides, and SC 2.4.3/2.4.7 are broken in practice when focus lands on invisible content.
- **Static detection:** Grep `onTouchStart|onPointerDown|touchstart` handlers and confirm corresponding button controls exist and are not removed by media queries; axe-core rule `button-name` catches unlabeled arrows automatically; grep dot elements for absence of `onClick`/`tabindex`/ARIA; grep slide containers for visibility management conditional on the active index (`inert`, `aria-hidden` paired with descendant `tabIndex={-1}`, `visibility`/`display` classes) — absence is a red flag; axe `aria-hidden-focus` catches the aria-hidden-with-focusables variant.
- **Runtime or browser test:** Operate the carousel three ways — mouse clicks only (no drag), keyboard only, speech-control simulation — slide change must be possible in all three; keyboard Tab walk with Playwright: every control reachable, named, and operable, and for each focus stop `document.activeElement` is within the viewport and computed visible; activate each dot — the slide changes and the indicator state follows; axe (`button-name`, `aria-hidden-focus`); screen-reader browse mode: hidden slides must not interleave with the current one.
- **Corrective rule:** Swipe/drag slide navigation MUST have single-pointer, keyboard-operable equivalents — at minimum labeled previous/next buttons per APG and the SC 2.5.1 example; picker dots are an additional alternative, not a substitute. Previous/next controls MUST be buttons with accessible names; slide pickers MUST be buttons (or tabs) that both control and reflect the current slide (`aria-disabled` on the current picker per the APG grouped pattern, or `aria-selected` in the tabbed pattern), with a non-color indicator. Slides not currently shown MUST be removed consistently from both the sequential focus order and the accessibility tree (`inert`, or `visibility:hidden`/`display:none` coordinated with `aria-hidden`); `aria-hidden="true"` MUST NOT be placed on any element that is, or contains, a focusable element.
- **Valid exceptions:** Multi-slide viewports where several slides are intentionally visible (all visible slides may stay focusable); gestures essential to the function (signature capture); UA-implemented scrolling of natively scrollable containers — a plain `overflow-x` + scroll-snap strip is not a 2.5.1 violation per se, but still needs keyboard-reachable scrolling and is no excuse for omitting controls when custom JS drives the gesture; a purely decorative background carousel — but then it must be inert and hidden from AT, not interactive-looking.
- **Primary sources:** Understanding SC 2.5.1 Pointer Gestures (carousel example; F105; keyboard-not-sufficient note) — https://www.w3.org/WAI/WCAG22/Understanding/pointer-gestures.html ; APG Carousel Pattern (control labels; picker semantics; incorrectly hidden slides) — https://www.w3.org/WAI/ARIA/apg/patterns/carousel/ ; WCAG 2.2 SC 2.1.1 — https://www.w3.org/TR/WCAG22/#keyboard ; SC 4.1.2 — https://www.w3.org/TR/WCAG22/#name-role-value ; SC 1.4.1 — https://www.w3.org/TR/WCAG22/#use-of-color ; WAI-ARIA 1.2 § `aria-hidden` — https://www.w3.org/TR/wai-aria-1.2/#aria-hidden ; Understanding SC 2.4.3 — https://www.w3.org/WAI/WCAG22/Understanding/focus-order.html ; axe-core rule descriptions (`button-name`, `aria-hidden-focus`) — https://github.com/dequelabs/axe-core/blob/develop/doc/rule-descriptions.md
- **Confidence:** High

## 3. Motion Failures

Motion is the single most visible signal of "designed" in AI-generated frontends, and the failures cluster accordingly: generators faithfully reproduce the motion that screenshots and demo snippets make visible — pulses, bounces, parallax, mount cascades, spring configs — while systematically omitting the conditional machinery that never appears in an exemplar (reduced-motion branches, pause controls, effect cleanup, interruption handling, native-scroll plumbing). This characterization is an observed pattern across the research briefs and is labeled **[inference]** wherever it explains *why* generators fail; every normative claim below is grounded in primary sources (WCAG 2.2 Understanding documents, MDN, web.dev, Chrome for Developers, Apple HIG, Material M3 tokens, Motion/GSAP/Tailwind documentation). A useful property of this cluster is that almost every failure is mechanically testable — WCAG SC 2.2.2 / 2.3.1 / 2.3.3 conformance, compositor-only animation rules, frame budgets, and listener lifecycle checks — so these rules are directly usable as AI code-review criteria. Scope note: positioning collisions of fixed/sticky chrome and mobile viewport-unit (`100vh`) clipping from the scroll brief are covered in Sections 4 and 6; general INP/long-task, list-virtualization, and React-keys items from the performance brief are covered in Section 6 and appear here only where the failure is specifically animation-caused.

### 3.1 Flashing content at or above 3 flashes per second
- **Classification:** objectively broken | inaccessible
- **Severity:** Critical — direct failure of WCAG 2.2 SC 2.3.1 (Three Flashes or Below Threshold, Level A), a Non-Interference criterion that poisons conformance of the whole page; can trigger photosensitive seizures.
- **Observable symptom:** An element's opacity, background, color, or box-shadow visibly alternates more than 3 times per second — "glitch" effects, notification strobes, glow pulses, `animate-ping`-style effects or custom `@keyframes flash` with a full on/off cycle under ~333 ms; looped strobe GIF/WebP assets used as decoration.
- **Likely implementation cause:** Custom keyframes alternating luminance (opacity 0↔1, bright↔dark backgrounds) with `animation-duration: 0.2s`–`0.3s` and `animation-iteration-count: infinite`; `steps()` glitch keyframes; JS `setInterval` toggling classes faster than 3 Hz; imported looping strobe GIFs.
- **Why it matters:** This is the only motion failure that is a documented physical-safety hazard rather than discomfort. WCAG: flashing above threshold "cannot be allowed even for a second or it could cause a seizure. And turning the flash off is also not an option since the seizure could occur faster than most users could turn it off." Content passes only with ≤ 3 general flashes and ≤ 3 red flashes in any one-second period, or a flashing area below threshold (.006 steradians ≈ 341×256 CSS px).
- **Static detection:** Extract `@keyframes` bodies that alternate `opacity`/`background(-color)`/`box-shadow` between high-contrast states; flag when `animation-duration` ÷ alternations-per-cycle < 333 ms, or durations matching `0.[0-2]\d?s` combined with `infinite`. Grep `setInterval|setTimeout` chains toggling visibility/color classes with delays < 350 ms. Heuristic: flashes/s ≈ (1000 / duration_ms) × light-dark transitions per iteration — flag ≥ 3 for review. Grep asset imports for `*.gif`/`*.webp` in hero/banner contexts.
- **Runtime or browser test:** DevTools Animations panel: inspect duration/iteration of running animations and compute the theoretical flash rate. Record 2 s of the region (Playwright video or screen capture) and count luminance inversions per second; > 3/sec = fail. WCAG points to PEAT (Photosensitive Epilepsy Analysis Tool) for borderline cases. Note: axe-core has **no rule** for flash rate (only obsolete `blink`/`marquee` elements) — do not rely on it.
- **Corrective rule:** No content may flash more than three times in any one-second period unless it provably stays below the general and red-flash area thresholds. Keyframes/intervals that alternate luminance ≥ 10% of max relative luminance at ≥ 3 Hz over an area larger than ~341×256 CSS px MUST be rejected in review. Blink-to-attract-attention MUST instead use ≤ 3 Hz motion and stop within 5 seconds (SC 2.2.2 technique G11).
- **Valid exceptions:** ≤ 3 flashes in any one second; flashing area ≤ .006 steradians (~341×256 CSS px reference block); fine balanced patterns (white noise, small checkerboards). Sites targeting AAA should note SC 2.3.2 removes even the threshold exception.
- **Primary sources:** https://www.w3.org/WAI/WCAG22/Understanding/three-flashes-or-below-threshold.html (SC 2.3.1, thresholds, G19/G176/G15, PEAT, 341×256 reference area); normative anchor https://www.w3.org/TR/WCAG22/#three-flashes-or-below-threshold; https://developer.mozilla.org/en-US/docs/Web/CSS/animation-timing-function
- **Confidence:** High — normative, safety-critical, measurable.

### 3.2 Auto-starting motion with no pause, stop, or hide mechanism
- **Classification:** inaccessible | objectively broken
- **Severity:** Critical — direct failure of WCAG 2.2 SC 2.2.2 (Pause, Stop, Hide, Level A, Non-Interference) whenever content starts automatically, lasts more than 5 seconds, and runs in parallel with other content; auto-updating content gets no 5-second grace.
- **Observable symptom:** Hero badges `animate-pulse` forever, CTAs `animate-bounce` forever, logo marquees and "trusted by" tickers scroll infinitely, testimonial carousels auto-advance every few seconds, `<video autoplay loop muted>` background heroes loop with no pause button, demo GIFs loop indefinitely — and nothing on the page lets the user stop any of it.
- **Likely implementation cause:** `animation-iteration-count: infinite` (Tailwind `animate-pulse`/`bounce`/`ping`/`spin` all use `infinite`); CSS marquee keyframes (`translateX(-50%)` with `infinite`); Framer Motion `transition={{ repeat: Infinity }}`; GSAP `repeat: -1`; `setInterval(() => setSlide(i + 1), 4000)` in a `useEffect` with no pause state; the canonical hero snippet `<video autoplay muted loop playsinline>` with no pause control and no reduced-motion gate; plain `<img src="/demo.gif">`, a format offering zero playback control.
- **Why it matters:** WCAG chose 5 seconds as "long enough to get a user's attention, but not so long that a user cannot wait out the distraction" (Understanding SC 2.2.2); continuous movement is a severe barrier for users with attention deficits and can interfere with screen readers. Named failure techniques: F16 (scrolling content without pause), F50/F112 (blinking > 5 s without a stop mechanism); G152 requires animated GIFs to stop within 5 s unless a stop mechanism exists.
- **Static detection:** Grep `animate-(pulse|bounce|ping|spin)` on non-loader elements; `animation-iteration-count:\s*infinite`; `animation:[^;]*\binfinite\b`; `repeat:\s*(Infinity|-1)`; `setInterval` adjacent to slide/index setters; `<video` with `autoplay|autoPlay` AND `loop` whose subtree lacks `controls`, a pause button, and a reduced-motion guard; `<img>`/`<Image>` sources ending in `.gif` (or animated `.webp`) with no `<source media="(prefers-reduced-motion: no-preference)">` conditional. Cross-check: element is not `role="progressbar"`/`aria-busy` and has no associated pause control.
- **Runtime or browser test:** Load, wait 6–10 s, run `document.getAnimations().filter(a => a.effect.getTiming().iterations === Infinity)` — any hit on non-loader content in parallel with other content is a failure. Keyboard-only: can each carousel slide be focused and read without advancing away? Emulate `prefers-reduced-motion: reduce` (DevTools Rendering tab or Playwright `page.emulateMedia`): motion that persists is an additional failure. axe-core catches only literal `<marquee>` — manual or Playwright checks are required.
- **Corrective rule:** Moving, blinking, scrolling, or auto-updating content that starts automatically, lasts > 5 s, and is presented in parallel with other content MUST provide a keyboard-accessible pause/stop/hide mechanism (SC 2.2.2, Level A). Infinite decorative loops on CTAs, badges, and backgrounds MUST be removed or gated behind explicit user opt-in. Animated GIFs MUST stop within 5 s (loop count baked into the file per G152), be exposed via `<video controls>`, or be served only under `(prefers-reduced-motion: no-preference)` with a static `<picture>` fallback. CSS marquees SHOULD be wrapped in `@media (prefers-reduced-motion: no-preference)`; Tailwind `animate-*` decorations MUST pair with `motion-reduce:animate-none` or be authored under `motion-safe:`.
- **Valid exceptions:** Essential progress/loading indicators where interaction cannot occur during the phase (SC 2.2.2 Note 4); essential real-time tickers (stock/auction), which still need the mechanism but may jump-to-current on resume; content not presented in parallel with other content; motion started by explicit user action (out of SC 2.2.2 scope, though SC 2.3.3 may still apply — see 3.3).
- **Primary sources:** https://www.w3.org/WAI/WCAG22/Understanding/pause-stop-hide.html (SC 2.2.2, F16/F50/F112, G152, G186, G191, auto-updating note); https://www.framer.com/motion/transition/ (`repeat: Infinity`); https://motion.dev/docs/react-use-reduced-motion; https://web.dev/articles/prefers-reduced-motion (`<picture media>` pattern); https://tailwindcss.com/docs/hover-focus-and-other-states#prefers-reduced-motion
- **Confidence:** High — normative Level A criterion with explicit failure techniques.

### 3.3 Motion ignores `prefers-reduced-motion` — across CSS, JS, and library layers
- **Classification:** inaccessible
- **Severity:** High — escalates to Critical for large-amplitude vestibular-trigger motion (parallax, large-scale zoom, full-screen wipes), where the documented reactions include dizziness, nausea, migraine, "potentially needing bed rest to recover" (Understanding SC 2.3.3). Reduced motion is "a medical necessity for these users" (web.dev), not a nicety. See Section 5 (Accessibility Failures) for the broader WCAG conformance-audit context.
- **Observable symptom:** With OS "reduce motion" enabled, the page parallaxes, slides, pulses, zooms, and smooth-scrolls identically to the default state. Under Playwright `reducedMotion: 'reduce'`, transform-driven motion keeps running.
- **Likely implementation cause:** Stylesheets contain `transition`/`@keyframes` with zero `@media (prefers-reduced-motion…)` blocks; JS contains zero `matchMedia('(prefers-reduced-motion: reduce)')` checks; no Tailwind `motion-safe:`/`motion-reduce:` variants; Framer Motion apps missing `<MotionConfig reducedMotion="user">` — the library's **documented default is `reducedMotion: "never"`**, so doing nothing is objectively the wrong default; GSAP tweens/ScrollTriggers created outside `gsap.matchMedia()` conditions; unguarded `html { scroll-behavior: smooth }` / Tailwind `scroll-smooth` and `scrollIntoView({ behavior: 'smooth' })` with no media check. The partial-compliance variant: a CSS media query exists, but GSAP/WAAPI/`element.animate()`/rAF loops/Lottie/canvas renderers keep running — CSS cannot stop JS-driven motion, and this variant defeats checklist-style audits.
- **Why it matters:** WCAG SC 2.3.3 (Animation from Interactions, AAA) requires interaction-triggered non-essential motion to be disable-able; techniques C39 (CSS media query) and SCR40 (JS `matchMedia`) address different layers, and modern stacks require both. The Understanding doc explicitly names parallax scrolling; MDN names "scaling or panning large objects" as vestibular triggers. The user has declared a preference at the OS level — ignoring it is an objective behavioral defect, not a taste choice.
- **Static detection:** Fastest project-level signal: repo contains `transition|animation|@keyframes|gsap\.|framer-motion|motion/react|animate-` but zero matches for `prefers-reduced-motion|motion-safe|motion-reduce|useReducedMotion|reducedMotion` → flag the whole project. Then layer-specific checks: `framer-motion`/`motion` imports without `MotionConfig`; `gsap`/`ScrollTrigger` files without `gsap.matchMedia(`; bare `scroll-behavior:\s*smooth` and `behavior:\s*['"]smooth['"]` without a media guard; two-pass check that every JS animation API module references `matchMedia('(prefers-reduced-motion'` or a library equivalent.
- **Runtime or browser test:** DevTools → Rendering → Emulate CSS media feature `prefers-reduced-motion: reduce`, or Playwright `await page.emulateMedia({ reducedMotion: 'reduce' })`; sanity-assert `matchMedia('(prefers-reduced-motion: reduce)').matches === true`; then assert `document.getAnimations().length === 0` on decorative subtrees, sample JS-animated elements' `transform` over 600 ms (any drift = fail), pixel-diff decorative canvases 500 ms apart, assert computed `scrollBehavior` is `auto`, and scroll the page confirming no parallax/pin/scrub persists.
- **Corrective rule:** Every non-essential animation MUST have a reduced-motion alternative at **every** motion layer: CSS guards or the global duration-collapse override (see 3.4), Tailwind `motion-safe:`/`motion-reduce:` variants, `<MotionConfig reducedMotion="user">` at the app root with `useReducedMotion()` swapping `x`/`y`/`scale`/layout animations for opacity fades, GSAP setup inside `gsap.matchMedia()` keyed on `(prefers-reduced-motion: no-preference)` or a `matchMedia` kill-switch per SCR40, and `scroll-behavior: smooth` / programmatic smooth scrolling gated on the same condition. Vestibular-trigger classes — parallax, large-scale zoom, full-screen wipes — MUST degrade to static layers, instant cuts, or subtle opacity fades under `reduce`. A CSS-only media query in a JS-animation codebase MUST be treated as non-compliant.
- **Valid exceptions:** Motion essential to functionality or information (WCAG "essential": removal would fundamentally change the information — e.g., an animation-authoring tool's preview; a progress spinner per SC 2.2.2 Note 4); opacity/color-only animations (Motion keeps these enabled under reduced motion, documented); an in-page site-wide motion toggle satisfies SC 2.3.3 independently.
- **Primary sources:** https://www.w3.org/WAI/WCAG22/Understanding/animation-from-interactions.html (SC 2.3.3, C39, SCR40, parallax named); https://web.dev/articles/prefers-reduced-motion (override pattern; "it can't stop animations started via the Web Animations API"); https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion; https://motion.dev/docs/react-motion-config (default `"never"`); https://motion.dev/docs/react-use-reduced-motion; https://gsap.com/docs/v3/GSAP/gsap.matchMedia()/; https://gsap.com/resources/getting-started/accessibility/; https://tailwindcss.com/docs/hover-focus-and-other-states#motion-safe; https://developer.mozilla.org/en-US/docs/Web/CSS/scroll-behavior; https://playwright.dev/docs/api/class-page#page-emulate-media; https://developer.apple.com/design/human-interface-guidelines/motion
- **Confidence:** High — spec-backed, objectively testable per layer.

### 3.4 Broken global neutralization: `animation: none !important` deadlocks `animationend`-dependent logic
- **Classification:** objectively broken
- **Severity:** Medium — a functional regression that punishes exactly the reduced-motion cohort the override serves.
- **Observable symptom:** With reduce-motion enabled, modals never finish opening/closing, elements stay at `opacity: 0` forever, multi-step flows stall — because code awaited `animationend`/`transitionend` events that now never fire.
- **Likely implementation cause:** The naive override `@media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation: none !important; transition: none !important; } }` shipped alongside JS that does `el.addEventListener('animationend', …)` or `await once(el, 'transitionend')` to advance state; with animations fully removed, the events don't fire and the state machine deadlocks.
- **Why it matters:** web.dev's documented override deliberately collapses durations to `1ms` (with `animation-iteration-count: 1`) instead of `animation: none`, *specifically* "because some sites rely on animation running to function properly (maybe because a certain step depends on the `animationend` event firing)." The naive version converts an accessibility fix into a functional regression.
- **Static detection:** Grep the reduced-motion block for `animation:\s*none\s*!important|transition:\s*none\s*!important` applied to universal selectors; then grep for `animationend|transitionend|onAnimationEnd|onTransitionEnd` listeners. Co-occurrence = flag.
- **Runtime or browser test:** Playwright `reducedMotion: 'reduce'`: execute key flows (open/close modal, wizard steps) and assert completion; assert no element remains at `opacity: 0`/`visibility: hidden` that is visible under `no-preference`.
- **Corrective rule:** Global reduced-motion overrides MUST collapse durations (≈ `1ms`, single iteration, delays ≤ 0, `scroll-behavior: auto !important`) rather than remove animations outright, OR the codebase MUST be proven free of `animationend`/`transitionend`-dependent control flow. Prefer the duration-collapse pattern from web.dev.
- **Valid exceptions:** `animation: none` on individually chosen decorative elements that no JS depends on.
- **Primary sources:** https://web.dev/articles/prefers-reduced-motion (override pattern and rationale); https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion
- **Confidence:** High — objectively broken behavior for a user cohort; mechanism documented.

### 3.5 Scrolljacking: intercepting `wheel`/`touchmove` to replace native scrolling
- **Classification:** objectively broken | inaccessible
- **Severity:** Critical — replaces a core user-agent interaction for every input device and routinely strands keyboard, screen-reader, and Find-in-page users when implemented incompletely; keyboard entrapment is a WCAG SC 2.1.1 / 2.1.2 (Level A) exposure.
- **Observable symptom:** Scroll feels "floaty" or delayed; trackpad flick momentum is gone; the scrollbar thumb no longer maps 1:1 to content; Space/PageDown/Home/End, Find-in-page jumps, and anchor navigation land wrong or stop working; on some viewports the page cannot be scrolled at all; `window.scrollY` stays `0` while a wrapper transforms.
- **Likely implementation cause:** A `wheel` listener calling `e.preventDefault()` (requiring `passive: false`) plus a `requestAnimationFrame` loop that lerps a virtual scroll value and applies `transform: translateY(-y)` to a fixed-height content wrapper; often combined with `overflow: hidden` on `html`/`body`. Generated versions typically omit the keyboard (`keydown`), `scrollTo`, anchor, and resize handling that mature smoothing libraries implement. **[inference]** Training data is saturated with award-site clones and 20-line "smooth scroll" demos that showcase the effect while omitting the accessibility plumbing.
- **Why it matters:** Native scrolling is a platform contract: it drives the scrollbar, Find-in-page, `#fragment` navigation, `scrollIntoView()`, screen-reader virtual cursors, and the accessibility tree. A fake transform-based scroller silently breaks all of these, and blocking `wheel`/`touchmove` forces the browser to wait on the main thread before every scroll tick.
- **Static detection:** Grep/AST for `addEventListener\(['"](wheel|mousewheel|touchmove)` with `preventDefault(` in the handler or `{ passive: false }`; `overflow: hidden` on `html`/`body` combined with a `position: fixed`/`height: 100vh` wrapper plus per-frame `transform` writes driven by a scroll accumulator; hand-rolled lerp loops (`current += (target - current) * 0.`).
- **Runtime or browser test:** Manual: flick-scroll on a trackpad (momentum must feel native); scroll with Space/PageDown/arrows only; Ctrl+F a term below the fold; click an in-page `#anchor` link; drag the scrollbar thumb. Playwright: `keyboard.press('Space')` / `page.mouse.wheel(0, 1200)` and assert `window.scrollY` changes; instrument `addEventListener` via `page.addInitScript` to detect `preventDefault` on document-level `wheel`.
- **Corrective rule:** The page MUST use native document scrolling as the source of truth. Virtual/transform-based scroll replacement MUST NOT ship. If a smoothing library is contractually required, it MUST preserve the native scrollbar, keyboard scrolling, Find-in-page, and anchor navigation, MUST disable smoothing under `prefers-reduced-motion: reduce`, and MUST NOT `preventDefault()` document-level `wheel`/`touchmove` listeners it cannot fully re-implement.
- **Valid exceptions:** Genuinely app-like surfaces where scrolling is not the primary navigation (a canvas-based map, a 3D product configurator, a slide deck) — provided keyboard operability and an escape mechanism exist. A marketing landing page is never a valid exception.
- **Primary sources:** https://developer.mozilla.org/en-US/docs/Web/API/EventTarget/addEventListener (passive/preventDefault semantics); https://www.w3.org/TR/WCAG22/ (SC 2.1.1 Keyboard, SC 2.1.2 No Keyboard Trap); https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion
- **Confidence:** High.

### 3.6 Mandatory full-viewport scroll snap on the document scroller
- **Classification:** objectively broken | inaccessible
- **Severity:** High — Critical when content becomes unreachable (snap children taller than the snap area on short viewports or at 200% text zoom).
- **Observable symptom:** The page "thunks" section-to-section; small wheel nudges snap back; users cannot park between sections to read content straddling a boundary; on short landscape viewports or with text zoom, the bottom of a section can never be scrolled into view; Find-in-page highlights text that snaps away.
- **Likely implementation cause:** `html { scroll-snap-type: y mandatory; }` + `section { height: 100vh; scroll-snap-align: start; }` generated for a "full-screen storytelling" design, applied to arbitrary content on the root scroller instead of a contained carousel; missing `proximity` fallback and no media-query disable. **[inference]** Full-page snap demos are extremely common landing-page training examples; the model maps "Apple-style product page" prompts to `100vh` + mandatory snap without checking content height or input modality.
- **Why it matters:** CSS Scroll Snap gives the UA power to override the user's scroll destination; `mandatory` means the UA *must* re-snap whenever content sits between snap points. Applied to the whole document it removes continuous reading position and conflicts with text zoom (WCAG 1.4.4/1.4.10-adjacent breakage). MDN's scroll-snap documentation scopes snapping to scroll *containers* (carousels, galleries), not whole documents.
- **Static detection:** Grep `scroll-snap-type:\s*(y|block|both)\s+mandatory` applied to `html`, `body`, or a top-level wrapper; flag co-occurrence with `height: 100vh`/`100dvh` on snap children whose content is not height-constrained (no internal `overflow: auto`).
- **Runtime or browser test:** Manual: viewport 1366×640, text zoom 200% — verify every line of every section can be scrolled into view and a user can rest between sections. Playwright: for each snap child, assert `el.scrollHeight <= viewport.height` or independent scrollability; assert the root scroller lacks mandatory snap. DevTools Elements → scroll-snap badge overlay enumerates snap areas.
- **Corrective rule:** Mandatory scroll snapping MUST NOT be applied to the document-level scroller or to sections whose content can exceed the viewport. Snap MUST be confined to bounded containers (image carousels, card strips), preferring `proximity`; any document-level snap MUST degrade to no snap under short viewports, text zoom, or reduced-motion user need.
- **Valid exceptions:** True paged experiences (onboarding slides, full-screen image viewers) where each snap child is guaranteed ≤ viewport and all content is reachable via other means (buttons/dots as well as scroll).
- **Primary sources:** https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_scroll_snap; https://developer.mozilla.org/en-US/docs/Web/CSS/length (viewport units); https://www.w3.org/TR/WCAG22/ (SC 2.1.1, when keyboard scrolling is impeded)
- **Confidence:** High.

### 3.7 Horizontal-scroll sections that hijack the vertical wheel
- **Classification:** objectively broken | inaccessible
- **Severity:** High — seizes the user's primary scroll input mid-page and routinely strands users; keyboard- and screen-reader-hostile when horizontal content exists only in the scrubbed state (WCAG SC 2.1.1 exposure).
- **Observable symptom:** While scrolling down, the page suddenly stops moving vertically and cards slide sideways; continued wheeling does nothing visible (pin region exhausted or trackpad diagonal input dropped); keyboard scrolling skips or sticks inside the section; on touch devices the vertical swipe stalls.
- **Likely implementation cause:** A `wheel` handler with `preventDefault()` mapping `deltaY → scrollLeft` without bounds checking or direction filtering (trackpads emit both axes); or a GSAP-ScrollTrigger-style `pin` + scrub whose spacer height is computed from a hard-coded pixel value instead of the track's real `scrollWidth`, breaking on resize/localization; no keyboard or reduced-motion alternative path; missing `overscroll-behavior`/`touch-action` management.
- **Why it matters:** This is scrolljacking localized to a section: it overrides the input-to-viewport mapping for all users. The correct non-hijacking primitive exists: a real horizontal scroller (`overflow-x: auto` + `scroll-snap-type: x proximity` + `scroll-snap-stop`), which keeps input semantics native.
- **Static detection:** Grep `deltaY` adjacent to `scrollLeft`/`translateX`/`preventDefault` in `wheel` handlers; `pin: true` or sticky-pin wrappers with hard-coded px heights; absence of `touch-action`/`overscroll-behavior` declarations on the section; absence of a keyboard-focusable real-scroller fallback.
- **Runtime or browser test:** Manual: wheel with a mouse (pure vertical deltas), a trackpad (mixed deltas), and keyboard-only (Tab into the section, then arrows/PageDown) — every path must traverse the section and continue the page. Playwright: dispatch wheel events and assert the page always reaches content below the section; resize the viewport and re-assert (spacer math). Emulate `prefers-reduced-motion: reduce` and verify the section degrades to a plain horizontal scroller or stacked layout.
- **Corrective rule:** Horizontal content MUST be implemented as a genuine horizontal scroll container (`overflow-x: auto`, optional `scroll-snap-type: x proximity` and `scroll-snap-stop`) navigable by native horizontal input, keyboard, and assistive tech. Vertical-to-horizontal wheel conversion or pinned scrub MAY only wrap that real scroller as enhancement, MUST be disabled under `prefers-reduced-motion: reduce`, and MUST never trap scroll: the page below must remain reachable by every input modality.
- **Valid exceptions:** None for trapping implementations; a non-trapping pinned scrub with a complete fallback (native scroller + reduced-motion opt-out + keyboard path) is acceptable.
- **Primary sources:** https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_scroll_snap (native horizontal snap primitives incl. `scroll-snap-stop`); https://developer.mozilla.org/en-US/docs/Web/API/EventTarget/addEventListener (wheel/preventDefault semantics); https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion; https://www.w3.org/TR/WCAG22/ (SC 2.1.1)
- **Confidence:** High.

### 3.8 Entrance-reveal state failures: content stuck hidden, multi-second mount cascades, and replays on re-entry
- **Classification:** objectively broken | inaccessible
- **Severity:** Critical when the reveal never fires (primary content permanently invisible — a functional outage, not polish); High for mount cascades that withhold content and degrade LCP; Medium for replay-on-re-entry.
- **Observable symptom:** Three variants of the same hidden-by-default pattern: (a) whole sections are blank — present in the DOM, invisible on screen — because the IntersectionObserver/`useEffect` reveal never runs (earlier JS error, hydration failure, misconfigured threshold, `display: none` subtree); (b) the page loads as a shell and header/hero/cards ripple in over 1–3 s, with text unreadable until the cascade finishes (worse on slow hydration); (c) scrolling up to re-read a section retriggers the full entrance animation, content flickering as classes re-toggle.
- **Likely implementation cause:** CSS-first hidden state (`.reveal { opacity: 0; transform: translateY(20px) }` or Tailwind `opacity-0`) with the visible state gated solely on JS and no no-JS fallback; observer `threshold: 1.0` on an element taller than the viewport (MDN: `threshold: 1.0` requires "every pixel is visible"); observer `root` that is not an ancestor of the target; React StrictMode double-mounts observing detached nodes; Framer Motion `initial={{ opacity: 0, y: 20 }}` on every section with `delayChildren`/stagger and no `initial={false}` for first paint; observer callbacks that add *and remove* the visibility class on every crossing without `observer.unobserve()` (or a `useInView` hook without `{ once: true }`).
- **Why it matters:** Permanently invisible primary content defeats the page's purpose, SEO rendering, and every accessibility API at once. Mount cascades measurably delay access past the ~1.0 s flow-of-thought limit (NN/g) and can directly degrade LCP: Chromium excludes `opacity: 0` elements from LCP candidates, so a hero mounted hidden is not "contentful" until hydration + animation. Replay-on-re-entry re-imposes "hidden → visible" on content the user has already read; MDN documents `unobserve` as the canonical one-shot pattern.
- **Static detection:** Grep `opacity-0` / `opacity:\s*0` / `visibility:\s*hidden` on content classes and require a JS visible-state counterpart **and** a no-JS fallback (`@media (scripting: none)` reset or `<noscript>`); flag `threshold: 1` on unbounded-size elements; count above-the-fold elements with `initial={{ opacity: 0 }}` and compute worst-case time-to-visible (max `delay` + `duration` + hydration) — flag when the LCP element mounts hidden or cumulative delay exceeds ~500 ms on primary content; flag observer callbacks containing both `classList.add` and `classList.remove` on the same class with no `unobserve`/`disconnect`.
- **Runtime or browser test:** Playwright: scroll through the page and assert every reveal element reaches computed `opacity: 1` (with zero page errors); reload with JavaScript disabled and assert primary content remains visible; compare Lighthouse/WebPageTest LCP with mount animations disabled (opacity-0 exclusion); assert the primary heading is visible within ~1 s of `domcontentloaded`; scroll an element in/out 3× and count `animationstart` events — must be ≤ 1 per element.
- **Corrective rule:** Content MUST be visible by default in HTML/CSS; reveal animations MUST be progressive enhancement — JS adds the hidden start state only when it can guarantee the observer is attached, or uses CSS-only `animation-timeline: view()`. Above-the-fold primary content MUST NOT mount hidden; entrance animation is allowed only on secondary elements, MUST total ≤ ~500 ms to full visibility, and SHOULD be skipped on repeat visits / back-forward-cache restores (`initial={false}`). One-shot reveals MUST fire at most once per element per page lifecycle (`unobserve()` on first reveal, `{ once: true }`, or a persisted ref); lists that re-render on filter/sort MUST NOT re-run entrance staggers.
- **Valid exceptions:** Decorative flourishes whose absence leaves content fully usable; explicitly requested splash/intro experiences; route transitions between already-loaded views where content replaces content instantly; deliberately scrubbed storytelling (then the compositor rules of 3.9 apply).
- **Primary sources:** https://developer.mozilla.org/en-US/docs/Web/API/Intersection_Observer_API (threshold semantics, initial callback, root/descendant requirement, `unobserve`); https://web.dev/articles/lcp (opacity-0 exclusion from LCP candidates); https://www.nngroup.com/articles/response-times-3-important-limits/; https://motion.dev/docs/react-animation (`initial={false}`); https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_scroll-driven_animations; https://developer.apple.com/design/human-interface-guidelines/motion. The hydration/SSR robustness angle of variant (a) is also covered in Section 6 (Frontend Implementation Failures).
- **Confidence:** High for the stuck-hidden and content-blocking-cascade variants; High that unbounded re-triggering is a defect rather than a style choice.

### 3.9 Animating layout properties instead of `transform`/`opacity` — including per-tick scroll-linked thrash
- **Classification:** inefficient | objectively broken
- **Severity:** High — guaranteed style-recalc + layout + paint work every frame for the animation's duration; jank is directly measurable, non-composited animations inflate CLS (Chrome for Developers), and the cost lands hardest on low-end devices during the most input-sensitive moment (scrolling).
- **Observable symptom:** Accordions, drawers, tooltips, hover-grow cards, and parallax layers stutter; sibling content reflows during the animation; scroll progress bars move in jittery steps; DevTools Performance shows purple "Layout"/"Recalculate Style" bars every frame, "Forced reflow" warnings, and Paint flashing over large regions; Lighthouse lists the nodes under "Avoid non-composited animations".
- **Likely implementation cause:** `transition: width 0.3s` / `transition: height …`; `@keyframes` animating `top`/`left`/`marginTop`/`max-height`; Framer Motion `animate={{ width: … }}` / `height: "auto"` expansions; `gsap.to(el, { width: … })`; scroll handlers writing `el.style.top = window.scrollY * 0.5 + 'px'` per tick, often interleaved with same-tick `getBoundingClientRect()`/`offsetTop` reads (the canonical layout-thrash read→write cycle) and using `width` instead of `transform: scaleX()` for progress bars; `transition: all` / Tailwind `transition-all` catch-alls that silently animate whatever property changes next — including layout properties added later — and make duration/easing review impossible.
- **Why it matters:** At 60 fps the frame budget is ~16.7 ms (≈ 10 ms of usable work after browser overhead). Layout is normally scoped to the entire document, so layout-animating components burn the budget every frame across affected subtrees; `transform`/`opacity` on their own layer trigger only composition. web.dev's canonical guidance is to keep animations on the compositor; geometry-property changes "can appear janky (stuttery) on low-end phones or when performance-heavy tasks are running on the main thread."
- **Static detection:** Grep/stylelint custom rule: `transition[^;}]*\b(width|height|top|left|right|bottom|margin|padding|max-height|max-width)\b`; `@keyframes` blocks setting those properties; JSX `animate=\{\{[^}]*\b(width|height|top|left)\b`; `gsap\.(to|from|fromTo)\([^)]*\b(width|height|top|left)\b`; within scroll/wheel listeners, assignments to `style.(top|left|right|bottom|margin*|width|height)` or layout reads (`getBoundingClientRect|offsetHeight|scrollHeight`) adjacent to style writes; `transition:\s*all\b` / `\btransition-all\b`.
- **Runtime or browser test:** Chrome DevTools → Performance → record during the animation: non-zero "Rendering"/"Layout" per frame = failure; Rendering tab → Paint flashing and FPS meter (dropping below ~55 = fail); expand scroll frames for "Forced reflow" insights naming the call stack; Lighthouse → "Avoid non-composited animations" lists offending nodes with reasons; Long Animation Frame API attribution in RUM.
- **Corrective rule:** Movement, scaling, and reveal animations MUST use `transform` (translate/scale/rotate) and `opacity`. Size reveals MUST use `transform: scaleX/Y` (with counter-scale on children if distortion matters), FLIP techniques, or Motion's `layout` engine. Scroll-linked JS MUST batch DOM writes inside `requestAnimationFrame` keyed off the latest scroll position, MUST read layout values once per frame (reads first, writes second), and SHOULD prefer CSS scroll-driven animations (`animation-timeline: scroll()`/`view()`) or IntersectionObserver-toggled transitions over per-tick math; scroll progress bars MUST use `transform: scaleX()` on a pre-sized element. `transition-property` MUST name explicit properties — `transition: all` MUST NOT ship. Visual responses to scroll/pointermove MUST NOT route through React state at native event frequency (use rAF-batched ref/DOM writes; the React-rendering angle is covered in Section 6).
- **Valid exceptions:** One-off, rare transitions where transform cannot express the effect (animating to `height: auto` on a small accordion where jank is unmeasurable); paint-only `color`/`background-color` transitions for simple state feedback; a single forced synchronous layout in a once-per-frame FLIP measure-then-set pair.
- **Primary sources:** https://developer.mozilla.org/en-US/docs/Web/Performance/Guides/Animation_performance_and_frame_rate; https://web.dev/articles/animations-guide; https://web.dev/articles/rendering-performance; https://developer.chrome.com/docs/lighthouse/performance/non-composited-animations; https://web.dev/articles/avoid-large-complex-layouts-and-layout-thrashing; https://developer.mozilla.org/en-US/docs/Web/API/Element/scroll_event; https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_scroll-driven_animations; https://motion.dev/docs/react-animation; https://tailwindcss.com/docs/hover-focus-and-other-states
- **Confidence:** High — engine-level, measurable, with an automated Lighthouse audit.

### 3.10 Event- and loop-hygiene failures: leaked listeners, uncancelled rAF/interval loops, non-passive scroll blocking, ungated background work
- **Classification:** objectively broken | inefficient
- **Severity:** High — leaked loops are a correctness bug (every mount adds another live loop, so re-entering a page doubles the per-frame work until the app visibly degrades); the remaining variants are measurable waste on the primary input path.
- **Observable symptom:** Animations speed up or duplicate after navigating away and back; scrolling grows jankier with each SPA route change; the first scroll after load stutters or ignores input; console shows "Unable to preventDefault inside passive event listener invocation"; fans spin on an idle background tab or while the animated element is scrolled far out of view.
- **Likely implementation cause:** `useEffect(() => { setInterval(tick, 16) }, [])` with no returned `clearInterval`; self-rescheduling rAF loops with no `cancelAnimationFrame` and no liveness flag; `addEventListener('scroll', …)` in lifecycle code with no `removeEventListener` (or inline handlers re-registered every render); GSAP timelines not killed on cleanup; `wheel`/`touchstart`/`touchmove` listeners on non-document targets without an explicit `passive` decision (per MDN, browsers default `passive: true` only on `Window`/`Document`/`body` — listeners elsewhere default to blocking) or calling `preventDefault()` inside a passive listener (a silent no-op with a console warning); marquee/particle/count-up loops with no `visibilitychange` listener and no IntersectionObserver gating.
- **Why it matters:** React's own docs teach that Effects which subscribe/schedule must return a cleanup, and that React deliberately remounts components in development to surface exactly this bug. MDN documents the `setInterval`/`clearInterval` and `requestAnimationFrame`/`cancelAnimationFrame` pairings. The `scroll` event "can fire at a high rate" (MDN), so duplicate or blocking handlers multiply an already hot path. rAF auto-pauses in hidden tabs, but `setInterval` does not — and neither API knows whether the animated element is in the viewport; MDN's Intersection Observer overview lists gating animation work by visibility as a core use case.
- **Static detection:** AST/regex audit of `useEffect` bodies containing `setInterval(`/`setTimeout(`/`requestAnimationFrame(`: flag when the effect does not return a function containing the matching `clearInterval`/`clearTimeout`/`cancelAnimationFrame`; flag `addEventListener` without a same-scope `removeEventListener`/`AbortController`; flag GSAP `gsap.to/timeline` in effects without `.kill()`/context revert. For every `addEventListener` with type in `{wheel, mousewheel, touchstart, touchmove}`, require an explicit third-argument `passive` decision; flag handlers that are passive (or attached to `window`/`document`/`body`) yet call `preventDefault()`. Flag components containing `setInterval(` but no `visibilitychange` and no `IntersectionObserver`.
- **Runtime or browser test:** React dev build: mount → unmount → remount with an instrumented tick log — frequency must stay constant. Playwright: 5 route round-trips, then assert constant listener counts (`getEventListeners(window)['scroll']` in console) and stable main-thread busy time. Watch the console for the passive-listener warning while exercising modals/drawers. Emulate `visibilitychange` with `visibilityState: 'hidden'` and assert interval callbacks cease; scroll the animated element out of view and assert callbacks stop in a Performance trace.
- **Corrective rule:** Every `setInterval`/`setTimeout`/`requestAnimationFrame`/event listener created inside component lifecycle code MUST be cancelled/removed in the corresponding cleanup (or registered with an `AbortSignal` aborted on unmount); self-rescheduling rAF loops MUST check a liveness flag before rescheduling. Every `wheel`/`touchstart`/`touchmove`/`mousewheel` listener MUST declare `passive` explicitly — `true` unless it genuinely cancels the event (then `false`, provably cheap, scoped to the smallest element, never the document) — and MUST NOT call `preventDefault()` from a passive listener (prefer CSS `overscroll-behavior`/`touch-action` for suppressing overscroll). Continuous JS-driven animation MUST pause when `document.visibilityState === 'hidden'` and SHOULD run only while its target intersects the viewport.
- **Valid exceptions:** App-lifetime singletons created outside components (React docs' "initializing the application" pattern); legitimate cancelation cases (range sliders, pinch-zoom canvases, drawer drag gestures) with `passive: false` scoped to the smallest element; animations whose state must advance in real time while hidden (rare — and even those should compute from timestamps, not tick counts); rAF-driven animation of always-visible chrome (rAF already auto-pauses in hidden tabs).
- **Primary sources:** https://react.dev/learn/synchronizing-with-effects; https://developer.mozilla.org/en-US/docs/Web/API/Window/setInterval; https://developer.mozilla.org/en-US/docs/Web/API/Window/requestAnimationFrame; https://developer.mozilla.org/en-US/docs/Web/API/EventTarget/addEventListener (passive semantics, document-level default flip, removeEventListener/AbortSignal); https://developer.mozilla.org/en-US/docs/Web/API/Element/scroll_event; https://developer.mozilla.org/en-US/docs/Web/API/Intersection_Observer_API
- **Confidence:** High for missing cleanup and hidden-tab waste (normative patterns); Medium-High for offscreen gating (best practice rather than hard correctness).

### 3.11 Repaint-heavy effects and shotgun layer promotion: animated shadows/filters, large-area backdrop blur, permanent `will-change`
- **Classification:** inefficient
- **Severity:** Medium — High for large-area `backdrop-filter` on mobile GPUs; the `will-change` variant degrades whole-page memory/compositing rather than a single animation.
- **Observable symptom:** GPU/fan spin-up and frame drops during "shadow grows on hover" or "blur fades in" effects; scroll jank and battery drain under a fixed frosted navbar or full-screen blurred modal; Paint flashing covering the entire blurred region on every scroll frame; DevTools Layers panel showing dozens of promoted layers on static content and paradoxically *worse* scroll performance on low-memory devices; blurry text from forced layer creation.
- **Likely implementation cause:** `transition: box-shadow .3s` with large blur/spread radii; keyframes interpolating `filter: blur(Npx)` or `backdrop-filter`; `backdrop-filter: blur(12px+)` (or Tailwind `backdrop-blur-*`) on full-width sticky headers, hero overlays, or modal backdrops over content that scrolls or animates beneath; `will-change: transform` hardcoded in base component CSS (cards, buttons, list items, `[will-change:transform]` on every row) "just in case" and never removed.
- **Why it matters:** web.dev: "anything involving a blur (such as a shadow, for example) is going to take longer to paint than drawing a red box" — repainting a blurred shadow every frame across a large area can saturate the raster budget even though Layout is skipped. `backdrop-filter` applies the filter to the live pixels painted behind the element (MDN), so the result must be recomputed whenever the backdrop changes; cost scales with blurred area × radius, and an ancestor with `filter`, `opacity < 1`, `mask`, `clip-path`, or `mix-blend-mode` silently becomes a backdrop root that clips the effect's scope — a correctness trap as well. MDN is normative on `will-change`: "intended to be used as a last resort", "don't apply will-change to too many elements", and stylesheet declarations keep the optimization "for much longer than it would have otherwise" — the documented anti-pattern.
- **Static detection:** Grep `box-shadow`/`filter`/`backdrop-filter` inside `@keyframes` or as a `transition-property` (including `transition: all` on elements carrying shadows/filters; Framer `boxShadow`/`filter` keys in `animate`/`whileHover`); flag `backdrop-blur-*` or `backdrop-filter` on `fixed`/`sticky`/full-viewport elements (`inset-0`, `w-screen`, `min-h-screen`) with radii above ~`blur(12px)`; grep `will-change` in stylesheets and flag any occurrence on multi-element selectors or any occurrence not paired with script that later resets it to `auto`.
- **Runtime or browser test:** DevTools → Rendering → Paint flashing while triggering the effect; Performance recording → Painting time per frame while scrolling with the blurred element visible; test on a physical mid-range Android device or with CPU/GPU throttling; DevTools Layers panel — count composited layers on a "static" page and compare memory with the declarations removed.
- **Corrective rule:** Shadow/filter animations SHOULD cross-fade `opacity` between two pre-rendered layers (pseudo-element technique) rather than interpolating the shadow/filter itself; direct interpolation is acceptable only for small areas that profile cleanly. `backdrop-filter` SHOULD be restricted to small, mostly static regions with a small radius (≤ ~8–12 px), be disabled or reduced on mobile, and MUST NOT sit over continuously animating or scrolling content without a profile demonstrating 60 fps on a mid-range device. `will-change` MUST NOT live in static stylesheets by default; it MAY be applied via JS shortly before a known animation and MUST be removed afterward (e.g., on `animationend`).
- **Valid exceptions:** Small elements (buttons, chips) with modest radii; one-shot non-repeating transitions that profile cleanly; small dialogs/menus on capable devices; non-scrolling fixed backdrops behind short-lived modals; a single continuously animating element (live visualizer) or MDN's documented slide-deck stylesheet case; a draggable element during the drag gesture.
- **Primary sources:** https://web.dev/articles/animations-guide (blur paint cost, "use `will-change` sparingly"); https://web.dev/articles/rendering-performance (paint step); https://developer.mozilla.org/en-US/docs/Web/CSS/backdrop-filter (description, "Backdrop root"); https://developer.mozilla.org/en-US/docs/Web/CSS/will-change (Warning + "Proper usage")
- **Confidence:** High on the cost mechanisms (documented and trivially reproducible); Medium on any fixed blur-radius threshold — the normative rule must be profile-based rather than numeric.

### 3.12 Interaction-hostile motion: non-interruptible conflicting tweens and animations that gate input
- **Classification:** objectively broken | deceptive
- **Severity:** High — the visible state diverges from the logical state and can persist stuck; input is rejected or misdirected while the UI appears ready, including misrouted destructive clicks.
- **Observable symptom:** Flick a pointer across a button/menu repeatedly: the element lurches, teleports to an end state, or stays semi-transparent/mis-positioned after the gesture ends. Clicking during a drawer/modal exit animation hits the invisible fading element instead of the button beneath; a submit button stays `disabled` until `onAnimationComplete` fires; rapid successive opens are ignored because the first exit hasn't finished.
- **Likely implementation cause:** GSAP 3's default `overwrite: false` — a new `gsap.to()` on the same property does not kill the in-flight tween; both run and the older one's stale values win (GSAP's own FAQ: "I am seeing some odd/unexpected behavior… try `gsap.defaults({overwrite: 'auto'})`"); WAAPI `element.animate()` called per event without `.cancel()`/`fill` discipline; Framer Motion `AnimatePresence` keeping exiting elements mounted with no `pointer-events: none`; state transitions gated on animation callbacks (`onAnimationComplete={() => setReady(true)}`) or `setTimeout` durations mirroring `transition-duration`; `gsap.from()` in `useEffect` under React StrictMode double-invoke.
- **Why it matters:** Apple HIG: "Let people cancel motion… don't make people wait for an animation to complete" and "As much as possible, don't make people wait for an animation to complete before they can do anything, especially if they have to experience the animation more than once." An animation system that cannot reverse mid-flight produces objectively incorrect end states; input gating breaks task flow and can misroute destructive clicks.
- **Static detection:** Grep `gsap\.(to|from|fromTo)\(` inside event handlers (`onMouseEnter/Leave`, `onClick`) where the vars object lacks `overwrite` and the project has no `gsap.defaults({ overwrite: ... })`; `element.animate(` calls without a preceding `.cancel()`; `onAnimationComplete|transitionend|animationend` used to gate `setState` enabling UI; `setTimeout` durations matching CSS durations; `AnimatePresence` children missing `pointerEvents` handling; buttons whose `disabled` prop depends on an `isAnimating` flag.
- **Runtime or browser test:** Playwright: hover/unhover a target 20× at 50 ms intervals, then assert the computed style equals the documented resting state and `element.getAnimations().length === 0`; trigger an exit and click the underlying element 50 ms in — assert via `document.elementFromPoint` that the click lands on the intended element, not the exiting overlay; measure time from user action to interactive-ready state — must be less than the animation duration (i.e., not gated on it).
- **Corrective rule:** Any animation that can be re-triggered while running MUST be interruptible with velocity/state continuity: in GSAP 3 set `overwrite: 'auto'` (per-tween or via `gsap.defaults`) or reuse/kill tween instances; in WAAPI cancel or replace prior Animations on retrigger; in Motion drive state through the `animate` prop (interruptible by design) rather than imperative re-creation. Animation MUST be cosmetic, never a gate: interactive elements MUST be operable immediately (or the operation accepted and the animation interrupted), exiting elements MUST set `pointer-events: none` for the entire exit, and exit durations for frequently used UI SHOULD be ≤ 200 ms.
- **Valid exceptions:** One-shot, non-retriggerable sequences (a splash intro that plays exactly once with input disabled); true modal commit flows where concurrent input would corrupt state (e.g., payment in-flight) — but then the UI must visibly show the busy state, not look idle.
- **Primary sources:** https://gsap.com/resources/conflict/; https://gsap.com/resources/3-migration/ (GSAP 3 `overwrite` default `false`); https://developer.apple.com/design/human-interface-guidelines/motion; https://motion.dev/docs/react-animation (AnimatePresence keeps exiting elements in the DOM)
- **Confidence:** High — documented library behavior with reproducible broken states.

### 3.13 Timing, easing, and feedback-continuity failures: overlong durations, uncapped staggers, non-settling springs, misapplied `linear` easing, instant unannounced jumps
- **Classification:** inefficient | objectively broken
- **Severity:** Medium — bounded cost per event that compounds across every interaction; the degenerate spring case (`damping: 0`) oscillates literally forever and escalates to the SC 2.2.2 failure in 3.2.
- **Observable symptom:** Hover states crawl in (`transition: all 1s ease`, `duration-1000` on small components); a 10-item list trickles in over 2–5 s from `delay: index * 0.2`; a badge or toast lands and visibly bounces 2–4+ times or never stops vibrating; menus and modals glide in at constant speed and stop dead (`ease: "linear"`); clicking a disclosure makes content pop into existence so abruptly that users double-click because the first click appeared to do nothing.
- **Likely implementation cause:** Literal long durations (`transition: all 1s`, `duration-1000`, `transition={{ duration: 1 }}`, global `MotionConfig transition={{ duration: 1 }}`) applied to small/interactive elements; per-item `delay: index * X`, `stagger`/`delayChildren` values uncapped by list length (demo corpora use 3–5 items where the math looks fine); spring configs with `damping` < ~15 against `stiffness` ≥ 300 or `bounce` ≥ 0.4 on small elements (Motion default `bounce` is 0.25; docs note `damping: 0` oscillates indefinitely); `animation-timing-function: linear` / `ease: "linear"` / Tailwind `ease-linear` copied from loop contexts (marquees, spinners, where it is correct) onto discrete transitions; conditional renders (`{open && <div>…}`) or `display: none ↔ block` flips with no transition at all.
- **Why it matters:** Material M3 duration tokens put simple small-component motion in the 50–200 ms band (short1–short4) and 250–400 ms (medium1–medium4) for standard transitions; NN/g: 0.1 s is the threshold where feedback feels like direct manipulation and ~1.0 s is the limit for an uninterrupted flow of thought — exceeded at just 4 items with a 300 ms stagger (total reveal = (N−1) × stagger + duration, re-paid on every mount/filter change). M3 defines `linear` as the token for *continuous* motion while discrete transitions use `standard`/`standardDecelerate`; Apple HIG: "Aim for brevity and precision in feedback animations," and feedback motion "can help people understand how something works" when enter/exit pairs are directionally consistent.
- **Static detection:** Regex durations `\b(duration|transition-duration)\s*[:=]\s*["']?(0\.[6-9]|[1-9]\d*\.?\d*)s` on small/interactive components; Tailwind `duration-(700|1000)`; extract `stagger(`/`delayChildren`/`stagger:` values and `index *` delay expressions and compute worst-case last-item time-to-visible against realistic list sizes — flag > ~800 ms, and any stagger on lists that re-mount on filter/sort; grep spring configs (`damping:\s*\d+` ≤ 10 with `stiffness` ≥ 300, `bounce:\s*0\.[4-9]`, and `damping:\s*0` anywhere → Critical); grep `timing-function:\s*linear|ease:\s*["']linear["']|ease-linear` excluding rules containing `infinite`/`repeat`; review heuristic: every user-triggered conditional render or visibility toggle should carry a matching transition or an explicit "instant is intended" note.
- **Runtime or browser test:** `getComputedStyle(el).transitionDuration` on hover/press targets (anything > ~400 ms for small components is suspect); Playwright: measure container-render → last-child reaching `opacity ≥ 0.99` against the budget; frame-capture spring transitions and count direction reversals (> 1 visible overshoot on a small element = fail; final rest within ~500 ms); DevTools Animations panel: inspect easing curves (constant slope on a discrete transition = flag); capture frames around state toggles — a one-frame 0→100% swap on a large region flags the unannounced jump.
- **Corrective rule:** Hover/press/focus feedback SHOULD be ≤ 150 ms; small-component state transitions SHOULD be 100–300 ms; durations > 500 ms MUST be reserved for large-surface or cross-view transitions and justified in review. Stagger intervals SHOULD be ≤ 50 ms per item and MUST be capped so total reveal ≤ ~800 ms regardless of list length (`delay: Math.min(index, 5) * 0.05`, or stagger only the first viewport). Springs on small UI elements MUST settle within ~1 perceptible overshoot cycle and ~500 ms; `damping: 0` or any non-converging configuration MUST NOT ship. Discrete enter/exit motion MUST use decelerate-biased easing for entrances (M3 `standardDecelerate`, `ease-out`) and accelerate-biased for exits; `linear` MUST be reserved for continuous loops, progress indication, opacity crossfades, and scroll-scrubbed motion where the input supplies the velocity. User-triggered state changes MUST be perceivable and causally linked — either instantaneous-but-obvious or a short (100–300 ms) transition — with directionally consistent enter/exit pairs.
- **Valid exceptions:** Progress indication and illustrative/onboarding motion (up to ~1 s); large full-screen transitions and deliberately cinematic hero moments gated behind reduced-motion support; explicitly choreographed hero/feature sections with few items and first-visit onboarding sequences; games, celebration effects (confetti), and drag-to-dismiss physics where bounce communicates elasticity; continuous rotation/marquee loops and indeterminate progress bars (correct `linear` use); high-frequency updates (live data, slider values during drag) where any interpolation would lag behind input.
- **Primary sources:** https://codebrowser.dev/flutter/flutter/packages/flutter/lib/src/material/motion.dart.html (M3 duration/easing tokens generated from the Material Design token database); https://www.nngroup.com/articles/response-times-3-important-limits/; https://developer.mozilla.org/en-US/docs/Web/CSS/animation-timing-function; https://www.framer.com/motion/transition/ (`bounce` default 0.25; `damping: 0` oscillates indefinitely; `delayChildren`/`stagger` semantics); https://developer.apple.com/design/human-interface-guidelines/motion; https://web.dev/articles/prefers-reduced-motion (animation's feedback role)
- **Confidence:** High for > 500 ms micro-interaction feedback, uncapped linear-index staggers, and non-converging springs (objectively wasteful/broken); Medium on exact budgets; the easing and feedback-continuity parts are spec-backed distinctions whose harm is perceived quality rather than breakage.

## 4. Responsive and Mobile Failures

Responsive failure is the signature defect of visually driven code generation. As an observed pattern (inference, not sourced fact), AI generators optimize for fidelity to a single desktop-width target — a screenshot, mockup, or prompt description — and reproduce its geometry as fixed declarations, while the states that define responsive behavior (320 px reflow, touch input, pinch zoom, landscape, split-view) are invisible in that target and are therefore systematically under-generated. The result is frontends that can be pixel-faithful at 1440 px and objectively broken at nearly every other width: pages that scroll horizontally, grids that squeeze instead of recompose, hover-gated controls that touch users cannot reach, and viewport tags that silently disable all responsive CSS. Because each failure below is testable against a normative standard — chiefly WCAG 2.2 SC 1.4.10 (Reflow), SC 1.4.4 (Resize Text), SC 2.5.8 (Target Size), and SC 1.3.4 (Orientation) — the corrective rules are written as review-ready must/should statements. Two related clusters are deliberately housed elsewhere and only cross-referenced here: text clipping under 200% text resize (SC 1.4.4, failure F69) is covered from the accessibility angle in Section 5, and navigation interaction wiring (dead hamburger toggles, `aria-expanded` desync, overlay inertness) is covered in Section 2.

### 4.1 Fixed pixel widths and min-widths cause horizontal page overflow at narrow viewports
- **Classification:** objectively broken | inaccessible
- **Severity:** Critical — direct WCAG 2.2 SC 1.4.10 (Reflow, Level AA) failure; content and controls become unreachable without two-dimensional scrolling.
- **Observable symptom:** At a 320–480 px wide viewport (or a 1280 px desktop viewport zoomed to 400%), the page gains a horizontal scrollbar; text columns, images, cards, or buttons extend off-screen to the right.
- **Likely implementation cause:** Hard-coded absolute widths on layout containers — `width: 1200px`, `min-width: 900px`, Tailwind `w-[1200px]` / `min-w-[48rem]`; `flex` rows with `flex-nowrap` and children that have `flex-shrink: 0` or intrinsic `min-width: auto` content (long unbreakable strings, `<pre>`, images without `max-width: 100%`); `grid-template-columns` with fixed px tracks; `width: 100vw` on full-bleed sections (overflows by the vertical scrollbar width, ~15 px).
- **Why it matters:** SC 1.4.10 requires content to "be presented without loss of information or functionality, and without requiring scrolling in two dimensions" at 320 CSS px width (equivalent to 1280 CSS px at 400% zoom). Fixed-width fragments break this for mobile users and for low-vision desktop users zooming to 400%.
- **Static detection:** Grep stylesheets/JSX for `(min-)?width:\s*[4-9]\d{2,}px` or `\d{4,}px` on layout elements; Tailwind: `w-\[\d+px\]`, `min-w-\[`, `flex-nowrap` without a wrap fallback, `w-screen` on elements that also have padding/margins, `whitespace-nowrap` on long text. AST heuristic: any `width`/`min-width` in px on a top-level container without an accompanying `max-width: 100%`.
- **Runtime or browser test:** Playwright: at viewport 320×568 assert `document.scrollingElement.scrollWidth <= 320` (allowing scrollbar tolerance); sweep widths 320→1440 in ~50 px steps and fail on any `scrollWidth > innerWidth`. Manual: DevTools responsive mode dragged down to 320 px, or a 1280 px window at 400% zoom (the canonical Reflow test). Lighthouse "Content is not sized correctly for the viewport" audit.
- **Corrective rule:** Layout MUST reflow without horizontal scrolling at 320 CSS px width. No fixed px `width`/`min-width` on page-level containers; use `max-width` plus fluid units; media MUST have `max-width: 100%`; `100vw` MUST NOT be used for content width unless compensated for scrollbar width.
- **Valid exceptions:** Content requiring two-dimensional layout for usage or meaning per SC 1.4.10's exception (data tables, maps, video, games, presentations, persistent-toolbar editing interfaces) — and only that excepted region may scroll in two dimensions, ideally inside its own scroll container (see 4.5).
- **Primary sources:** WCAG 2.2, SC 1.4.10 Reflow — https://www.w3.org/TR/WCAG22/#reflow; Understanding SC 1.4.10 — https://www.w3.org/WAI/WCAG22/Understanding/reflow.html (failure F102; techniques C31/C32/C33; advisory C37 `max-width` for media); web.dev, Responsive web design basics ("Size content to the viewport") — https://web.dev/articles/responsive-web-design-basics
- **Confidence:** High — normative WCAG AA failure with a mechanical test.

### 4.2 Multi-column grids squeeze into unreadably narrow columns instead of recomposing
- **Classification:** objectively broken | inaccessible
- **Severity:** High — fails SC 1.4.10's "without loss of information" intent in practice; where text genuinely overflows, clips, or overlaps, it is an outright Reflow/Resize-Text failure.
- **Observable symptom:** A 3–6 column card grid, pricing table, or stats row stays multi-column at 375 px: each column computes to 60–110 px wide, words wrap one letter per line, badges overlap, and comparison layouts become illegible.
- **Likely implementation cause:** `grid-template-columns: repeat(4, 1fr)` (Tailwind `grid-cols-4`) with no responsive variants — the correct mobile-first pattern is a `grid-cols-1` base with `md:grid-cols-4`; fixed `flex-basis` percentages without `flex-wrap`; no `repeat(auto-fit, minmax(220px, 1fr))`-style intrinsic recomposition; media queries keyed to the wrong breakpoint so a 2-column state persists at 360 px.
- **Why it matters:** WCAG sufficient techniques C31 (Flexbox reflow) and C32 (media queries plus grid to reflow columns) exist precisely because multi-column content must recompose into fewer — ultimately one — columns at narrow widths; squeezing columns defeats reflow without technically triggering horizontal scrolling, so it evades naive overflow tests.
- **Static detection:** Grep for `grid-cols-[3-9]` / `repeat\([3-9]` without any responsive prefix or `auto-fit`/`auto-fill` in the same rule; Tailwind heuristic: `grid-cols-N` or `w-1/3` classes lacking any `sm:`/`md:`/`lg:` variant anywhere in the component; `flex` rows whose children have percentage widths while the container has no `flex-wrap`/`flex-col` variant.
- **Runtime or browser test:** DevTools responsive sweep 1440→320 px; fail if any text column computes below ~120 px effective width while siblings still render side-by-side, or if text overlaps/clips (Playwright per-breakpoint element screenshot comparison).
- **Corrective rule:** Multi-column layouts MUST reduce column count as width shrinks and MUST reach a single-column (or content-appropriate minimal) arrangement by 320–480 px, via mobile-first breakpoints or intrinsic `auto-fit`/`minmax()` recomposition. Columns MUST NOT be retained at widths that make their content illegible.
- **Valid exceptions:** True two-dimensional data (grids/tables) under the SC 1.4.10 exception; icon/avatar strips where each cell is a self-contained non-text unit that remains legible.
- **Primary sources:** Understanding SC 1.4.10 — https://www.w3.org/WAI/WCAG22/Understanding/reflow.html (sufficient techniques C31, C32; "Adjusting content into a single column"); web.dev, Responsive web design basics (flexbox/grid/multicol for flexible grids) — https://web.dev/articles/responsive-web-design-basics
- **Confidence:** High.

### 4.3 Navigation is not restructured into an operable mobile pattern
- **Classification:** objectively broken | inaccessible
- **Severity:** Critical when menu items become unreachable (loss of functionality); High when links merely wrap into dense, undersized rows.
- **Observable symptom:** At phone width the desktop navbar either (a) wraps into 2–3 rows of small, tightly packed text links, (b) is clipped by `overflow: hidden` so right-side items ("Pricing", "Login") disappear, or (c) collapses behind a hamburger icon — a restructuring that only counts as conforming if the toggle actually works.
- **Likely implementation cause:** A single `<nav>` flex row with `overflow-hidden` and no mobile disclosure widget; responsive classes that only shrink font/padding (`md:text-base text-xs`) instead of restructuring; nav links at 13–18 px height with zero vertical padding on mobile (see 4.6); the hamburger pattern emitted as decoration without a wired toggle (wiring failures are Section 2 territory).
- **Why it matters:** Clipped or dead navigation is loss of functionality — Understanding SC 1.4.10's failure F102 ("content disappearing and not being available when content has reflowed") applies when narrow states lose access to nav features. The same Understanding document explicitly blesses collapsing navigation behind a hamburger/"More" menu as a conforming reflow pattern, so restructuring is the required fix, not an optional enhancement. Tiny wrapped links also fail SC 2.5.8 target-size spacing (Understanding SC 2.5.8, Figure 8: menu items at 18 px height fail, 24 px pass).
- **Static detection:** Heuristic: `<nav>` containing more than 4 sibling links with no conditional mobile menu component in the same file; grep for `overflow-hidden` on header/nav combined with `whitespace-nowrap`; a hamburger `<button>` with no `onClick`/`aria-expanded`/state hook in the component (dead-toggle wiring is reported in Section 2).
- **Runtime or browser test:** At 375 px verify every desktop nav destination is still reachable (crawl all links); tap the hamburger with Playwright touch emulation and assert a menu opens and is keyboard-operable; check computed heights of nav links ≥ 24 px or spacing-exception compliant.
- **Corrective rule:** At every viewport width, all navigation functionality MUST remain available and operable. Narrow viewports MUST restructure navigation into an operable pattern (disclosure menu with a working toggle, priority-plus-"more" pattern, or vertical list) — never clip it via overflow or shrink it into sub-24 px wrapped link rows.
- **Valid exceptions:** Single-page sites with 2–3 short links that fit at full size without wrapping.
- **Primary sources:** Understanding SC 1.4.10 — https://www.w3.org/WAI/WCAG22/Understanding/reflow.html (failure F102; conforming example "navigation changes … behind a hamburger menu button … no horizontal scrolling"); Understanding SC 2.5.8 — https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html (Figure 8)
- **Confidence:** High. (For the mobile menu's interaction wiring — dead toggles, missing accessible names, non-inert overlay backgrounds — see Section 2, Interaction Failures.)

### 4.4 Hover-dependent interactions with no touch (or keyboard) equivalent
- **Classification:** objectively broken | inaccessible
- **Severity:** High — content/functionality is literally unreachable on touch devices and for keyboard users; violates SC 2.1.1 (Keyboard, Level A) and typically the SC 1.4.13 conditions.
- **Observable symptom:** Dropdown menus, tooltips, image-card overlays, or "reveal-on-hover" action buttons (edit/delete icons on table rows) cannot be triggered on a phone: a tap either does nothing, follows the link immediately, or flashes the overlay and then navigates away.
- **Likely implementation cause:** CSS-only `:hover` disclosure (`.parent:hover .menu { display: block }`) with no `:focus-within`/click path; React `onMouseEnter`/`onMouseLeave` handlers with no `onClick`/`onFocus`; action buttons styled `opacity-0 group-hover:opacity-100`; no use of `@media (hover: hover)` to scope hover-only affordances to devices that can hover.
- **Why it matters:** SC 1.4.13 requires hover-triggered additional content to be dismissible, hoverable, and persistent; SC 2.1.1 requires keyboard operability of all functionality; Media Queries Level 4 defines the `hover`/`pointer` interaction features precisely so authors can adapt to input modality instead of assuming it. Touch is the primary input for most mobile users, and hover-gated functionality simply does not exist for them.
- **Static detection:** Grep JSX for `onMouseEnter|onMouseOver` handlers with no sibling `onClick|onFocus`; CSS AST: rules where `:hover` changes `display`/`visibility`/`opacity` on interactive content without a matching `:focus`/`:focus-within` rule or `@media (hover: hover)` guard; Tailwind `group-hover:` revealing buttons.
- **Runtime or browser test:** Playwright with a `hasTouch: true` context: tap the trigger and assert the disclosed content appears and stays; keyboard Tab to the trigger and assert the same content appears; press Escape and assert dismissal (SC 1.4.13 Dismissible).
- **Corrective rule:** Any content or control revealed on hover MUST also be revealable by click/tap and by keyboard focus, MUST remain visible until dismissed (e.g., Escape) or no longer valid, and MUST itself be hoverable. Hover-only enhancements MUST be gated behind `@media (hover: hover) and (pointer: fine)` and MUST never be the sole path to functionality.
- **Valid exceptions:** Purely decorative hover effects that expose no additional content or functionality; UA-controlled tooltips (native `title` attribute) are excepted from SC 1.4.13.
- **Primary sources:** WCAG 2.2, SC 1.4.13 — https://www.w3.org/TR/WCAG22/#content-on-hover-or-focus; Understanding SC 1.4.13 — https://www.w3.org/WAI/WCAG22/Understanding/content-on-hover-or-focus.html (failure F95; dismissible/hoverable/persistent conditions); MDN, `@media` `hover` feature — https://developer.mozilla.org/en-US/docs/Web/CSS/@media/hover; MDN, `@media` `pointer` feature — https://developer.mozilla.org/en-US/docs/Web/CSS/@media/pointer
- **Confidence:** High. (Navigation-specific hover menus and pointer-diagonal submenu collapse are covered in Section 2, Interaction Failures.)

### 4.5 Data tables drag the entire page into two-dimensional scrolling
- **Classification:** objectively broken | inaccessible
- **Severity:** High — data tables themselves have an SC 1.4.10 exception, but dragging the *entire page* into horizontal scrolling is a documented failure pattern (Understanding SC 1.4.10, Figure 13 Fail/Pass pair).
- **Observable symptom:** At phone width a 6+ column table pushes the whole page sideways; the header nav, footer, and body paragraphs all acquire horizontal scroll; sticky table headers misalign; or the table is squeezed until cell text is unreadable.
- **Likely implementation cause:** `<table>` rendered directly in the page flow with no wrapping `<div style="overflow-x:auto">` (Tailwind: an `overflow-x-auto` wrapper around a `min-w-*` table); `table-layout: fixed` with px column widths; no restructure strategy (stacked cards, priority columns, or a horizontal scroll region) for small viewports.
- **Why it matters:** Understanding SC 1.4.10 explicitly recommends rendering the table "within its own scrollable container" so non-excepted content — headings, search fields, pagination, paragraphs — still reflows; the exception covers the table region, not its surroundings.
- **Static detection:** JSX/AST heuristic: a `<table>` whose ancestor chain (up to the layout container) contains no element with `overflow-x: auto|scroll`; grep for `table-layout: fixed` combined with px widths; Tailwind: `<table` with no `overflow-x-auto` wrapper in the component.
- **Runtime or browser test:** At 320–375 px assert page-level `scrollWidth <= innerWidth` while the table region scrolls internally (wrapper's `scrollWidth > clientWidth`, page's does not exceed the viewport); manually verify headings and pagination outside the table do not scroll horizontally.
- **Corrective rule:** Wide data tables MUST be contained in a region that scrolls independently (`overflow-x: auto` wrapper, keyboard-focusable where scrollable per keyboard-operability norms), or MUST be restructured for narrow viewports; non-tabular content around the table MUST reflow normally.
- **Valid exceptions:** The table region itself may scroll in two dimensions (SC 1.4.10 exception); genuinely two-dimensional interfaces (spreadsheets) may keep full-table layouts.
- **Primary sources:** Understanding SC 1.4.10 — https://www.w3.org/WAI/WCAG22/Understanding/reflow.html (Figures 9–13; "Tabular data and grid-based UI"; Figure 13 Fail/Pass pair)
- **Confidence:** High.

### 4.6 Touch targets below 24×24 CSS px on mobile
- **Classification:** inaccessible | objectively broken
- **Severity:** High — direct WCAG 2.2 SC 2.5.8 (Target Size Minimum, Level AA) failure when size and spacing conditions are unmet; adjacent undersized targets cause mis-taps that activate the wrong action.
- **Observable symptom:** Icon-only buttons (hamburger, close ×, social icons, table-row actions, carousel dots, pagination numerals) render at 12–20 px; on a phone, taps miss or hit the adjacent control — worst when destructive actions (delete) sit 4 px from safe ones (view).
- **Likely implementation cause:** Sizing the *icon* instead of the *hit area* — `<button class="h-4 w-4 p-0">` wrapping a 16 px SVG; links sized by `text-xs` line-height only (~18 px) with no padding; `padding: 0` resets on icon buttons; checkbox/radio replacements at 16 px with no label hit area; `onClick` attached to the `<svg>` itself so the target equals the icon box.
- **Why it matters:** SC 2.5.8 requires targets ≥ 24×24 CSS px or a spacing equivalent (24 px-diameter circles centered on each undersized target must not intersect any other target or circle — adjacent icon buttons in a `gap-1` row fail even the exception). Platform guidance is stricter: Apple HIG specifies 44×44 pt default control size (28×28 pt minimum); web.dev/Material guidance is ~48×48 device-independent pixels with ~8 px spacing, and Lighthouse fails tap targets below 48×48 px with ≥ 25% overlap — a useful engineering bar beyond the WCAG floor. Zoom does not change CSS-px size, so authors cannot claim zoom fixes it.
- **Static detection:** axe-core `target-size` rule (maps to SC 2.5.8) in CI; Tailwind grep: buttons/links with `w-4|h-4|w-5|h-5` and `p-0|p-0.5|p-1`; JSX heuristic: `<button>` containing only an `<svg>` with no padding classes (rule of thumb: `p-0` + `w-4` = 16 px target; technique C42 uses `min-height`/`min-width` on the target container).
- **Runtime or browser test:** axe DevTools / `@axe-core/playwright` with the `target-size` rule enabled at a mobile viewport; Playwright `boundingBox()` sweep over every `button, a[href], [role=button]` asserting ≥ 24×24 or a passing circle-intersection check; Lighthouse tap-target audit on mobile; manual thumb test on a real phone (can you hit page "3" without hitting "2"/"4"?).
- **Corrective rule:** All pointer targets MUST be ≥ 24×24 CSS px or satisfy the SC 2.5.8 spacing exception; primary actions and mobile-first controls SHOULD meet 44×44 px (Apple HIG) / 48×48 dp (web.dev). Enlarge the hit area with padding or transparent borders — the icon is the visual, the button is the target; never rely on the visible glyph size. Adjacent destructive actions SHOULD be separated beyond the minimum spacing and/or guarded.
- **Valid exceptions:** SC 2.5.8's five exceptions — Spacing, Equivalent (another compliant control performs the function, e.g., Previous/Next buttons for carousel dots), Inline (links within sentences/blocks of text), User-agent control (unstyled native controls), Essential (map pins, dense data visualizations).
- **Primary sources:** WCAG 2.2, SC 2.5.8 — https://www.w3.org/TR/WCAG22/#target-size-minimum; Understanding SC 2.5.8 — https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html (technique C42; spacing-circle math); Apple Human Interface Guidelines, Accessibility — https://developer.apple.com/design/human-interface-guidelines/accessibility (44×44 pt default, 28×28 pt minimum); web.dev, Accessible tap targets — https://web.dev/articles/accessible-tap-targets (48 px, 8 px spacing, `@media (any-pointer: coarse)`); Deque University, axe `target-size` rule — https://dequeuniversity.com/rules/axe/4.10/target-size; Lighthouse "Tap targets are not sized appropriately" — https://developer.chrome.com/docs/lighthouse/seo/tap-targets
- **Confidence:** High. (Navigation-specific target failures are also flagged in Section 2, Interaction Failures; the broader WCAG perceivability/operability context is in Section 5.)

### 4.7 Viewport meta tag missing, or configured to block zoom
- **Classification:** objectively broken | inaccessible
- **Severity:** High — a missing tag effectively disables all responsive CSS on mobile; zoom-blocking attributes fail SC 1.4.4 (and map to 1.4.10) on browsers that honor them.
- **Observable symptom:** Two variants. (a) Missing tag: on a phone the page loads zoomed-out with microscopic text in a ~980 px virtual viewport; media queries keyed to `max-width: 480px` never fire; users must pinch-zoom and pan for every interaction. (b) Zoom blocked: pinch-to-zoom does nothing on Android Chrome and older iOS Safari, so low-vision users cannot magnify the page at all.
- **Likely implementation cause:** (a) No `<meta name="viewport" content="width=device-width, initial-scale=1">` in `index.html` / the document head; in Next.js App Router, no `export const viewport` in the root layout; the tag lives in boilerplate that component-level generation never touches. (b) `maximum-scale=1, user-scalable=no` copied from "app-like" boilerplate, or added deliberately to suppress iOS Safari's input-focus auto-zoom (see 4.8) — converting a Medium layout annoyance into a High accessibility failure.
- **Why it matters:** Mobile browsers render pages without the viewport meta in a ~980 px virtual viewport and shrink the result to fit (MDN; web.dev), defeating every responsive technique and forcing two-dimensional zoom-and-pan — the exact condition SC 1.4.10/1.4.4 exist to prevent. W3C ACT rule b4f0c3 ("Meta viewport allows for zoom") fails any `user-scalable=no` and any `maximum-scale` below 2, and maps to SC 1.4.4 (required for AA conformance); blocking zoom removes the primary accommodation low-vision mobile users rely on.
- **Static detection:** Grep the served HTML entry point for `<meta name="viewport"` (in Next.js, check the root layout for `export const viewport`/`generateViewport`) and fail the review if absent. Regex on the document head / `viewport` export: `user-scalable\s*=\s*no` or `maximum-scale\s*=\s*(1(\.\d+)?|[01]\.\d+)` (the ACT rule fails `maximum-scale < 2`); also flag `minimum-scale` above 1.
- **Runtime or browser test:** Lighthouse "Has a `<meta name=viewport>` tag with `width` or `initial-scale`" audit; DevTools device emulation — a page without the tag renders visibly zoomed out, and `window.innerWidth` ≈ 980 instead of the device width. axe-core `meta-viewport` rule (implementing ACT b4f0c3) via axe DevTools or CI; manual pinch-zoom on a physical Android device / older iOS.
- **Corrective rule:** Every page MUST include `<meta name="viewport" content="width=device-width, initial-scale=1">` (or the framework equivalent, e.g., the Next.js `viewport` export). The tag MUST NOT contain `user-scalable=no` and MUST NOT set `maximum-scale` below 2 (best practice: omit both). Zoom suppression MUST NOT be used to mask layout or input-zoom bugs.
- **Valid exceptions:** None for web pages intended for general use. Per MDN, iOS 10+ ignores the zoom-blocking attributes by default and desktop browsers ignore the meta entirely — the failure materializes on browsers that still honor the restriction, which the ACT rule's Accessibility Support section acknowledges.
- **Primary sources:** MDN, Viewport meta tag — https://developer.mozilla.org/en-US/docs/Web/HTML/Viewport_meta_tag; web.dev, Responsive web design basics ("Set the viewport"; Lighthouse audit reference) — https://web.dev/articles/responsive-web-design-basics; W3C ACT Rules, "Meta viewport allows for zoom" (b4f0c3) — https://www.w3.org/WAI/standards-guidelines/act/rules/b4f0c3/; Understanding SC 1.4.4 — https://www.w3.org/WAI/WCAG22/Understanding/resize-text.html (Test Rules: "Meta viewport allows for zoom")
- **Confidence:** High.

### 4.8 Input font-size under 16 px triggers iOS Safari auto-zoom on focus
- **Classification:** objectively broken
- **Severity:** Medium — not a spec violation, but an objectively reproducible platform-behavior conflict that disorients users and breaks layouts.
- **Observable symptom:** On iPhone Safari, tapping a search box, login field, or chat textarea zooms the page in; after submitting, the layout stays zoomed and horizontally scrolls until the user manually pinches out.
- **Likely implementation cause:** Inputs styled with `font-size: 14px` or smaller (Tailwind `text-sm`/`text-xs` on `<input>`/`<textarea>`/`<select>`) to match 13–14 px desktop form typography. The behavior is an iOS Safari heuristic: it auto-zooms editable text fields whose computed font-size is below 16 px. (The 16 px threshold is empirically reproducible platform behavior; Apple publishes no specification for it — treat it as a tested heuristic, not a quoted standard.)
- **Why it matters:** Unexpected zoom is a forced viewport change the user did not request; combined with fixed-width or overflowing layouts (4.1, 4.5) it strands users in a magnified, horizontally scrolling page. It also tempts developers into disabling zoom entirely (4.7), converting a Medium issue into a High accessibility failure.
- **Static detection:** Grep for `font-size` below 16 px on `input, textarea, select` selectors; Tailwind: `text-(xs|sm)` classes applied to form field elements; design-token check: form-field text token below 16 px.
- **Runtime or browser test:** Manual test on a real iPhone or Xcode Simulator: focus each text field and observe viewport zoom; a BrowserStack/real-device pass for forms. Desktop emulation does NOT reproduce this behavior.
- **Corrective rule:** Editable text fields SHOULD have a computed font-size of at least 16 px on mobile (or `font-size: max(16px, 1em)`); zoom suppression via the viewport meta MUST NOT be used as the workaround.
- **Valid exceptions:** Non-editable/read-only fields; desktop-constrained internal tools never used on iOS Safari.
- **Primary sources:** MDN, Viewport meta tag (platform viewport/zoom behavior context) — https://developer.mozilla.org/en-US/docs/Web/HTML/Viewport_meta_tag. No Apple-authored documentation of the 16 px threshold exists; the behavior is verifiable on-device (practitioner references and framework bug trackers corroborate it for discovery only, not as evidence).
- **Confidence:** Medium — High that the behavior exists and is disruptive; Medium that it is "objectively broken" in the normative sense (platform heuristic, no spec).

### 4.9 Sticky/fixed chrome consumes most of the mobile viewport
- **Classification:** objectively broken | inaccessible
- **Severity:** High — when sticky chrome fully hides a keyboard-focused element it is a documented SC 2.4.11 failure (F110); even without keyboard use it violates the Reflow intent by strangling the readable area.
- **Observable symptom:** On a phone in portrait (worse in landscape), a 90–140 px sticky header plus a cookie banner plus a sticky bottom CTA bar leave a ~150–250 px slit of readable content; tabbing moves focus behind the sticky bars.
- **Likely implementation cause:** `position: sticky/fixed` header + `fixed bottom-0` CTA + cookie banner stacked, all sized for desktop; no `@media` adjustment to un-fix or shrink them at small heights (advisory technique C34); landscape media queries absent, so sticky chrome scales with width rather than height. (Missing `scroll-padding-top`, which hides in-page anchor targets under the header, is covered with navigation in Section 2.)
- **Why it matters:** Understanding SC 1.4.10 has a dedicated sticky/fixed-content discussion ("such sticky or fixed content can make reading content difficult if not impossible") and advises static positioning or user-toggleable display at small viewports; SC 2.4.11 (Focus Not Obscured Minimum, Level AA) fails sticky chrome that completely hides a focused component.
- **Static detection:** Grep for `sticky|fixed` positioning classes on header/footer/banner components; sum their heights against a 568×320 viewport heuristic; check for the absence of `scroll-padding` on `html`/`body` when sticky headers exist; check for height-based media queries (`max-height`), which are almost never generated.
- **Runtime or browser test:** DevTools emulating iPhone SE landscape (667×375): measure remaining content height; keyboard Tab through the page and assert the focused element is never fully hidden (SC 2.4.11); click an in-page anchor and check the target heading is visible.
- **Corrective rule:** At narrow widths AND short heights (landscape phones, split view), sticky chrome MUST be minimized, made static (technique C34), or user-dismissible; sticky/fixed content MUST NEVER entirely hide the keyboard-focused component (use `scroll-padding` per technique C43); combined sticky chrome SHOULD NOT exceed roughly one third of the smaller viewport dimension.
- **Valid exceptions:** Small single-bar sticky headers (≤ ~64 px) that don't obscure focus; persistent toolbars in editing interfaces (SC 1.4.10 exception); user-opened/dismissible overlays per SC 2.4.11 Note 2.
- **Primary sources:** Understanding SC 1.4.10 — https://www.w3.org/WAI/WCAG22/Understanding/reflow.html (sticky/fixed-content discussion; advisory C34 "Using media queries to un-fixing sticky headers/footers"; Figures 18–19); Understanding SC 2.4.11 — https://www.w3.org/WAI/WCAG22/Understanding/focus-not-obscured-minimum.html (failure F110; technique C43 `scroll-padding`)
- **Confidence:** High. (The anchor-navigation occlusion angle — fragment targets landing under the header for lack of `scroll-padding-top` — is covered in Section 2, Interaction Failures.)

### 4.10 Orientation lock and "please rotate your device" interstitials
- **Classification:** objectively broken | inaccessible
- **Severity:** High when content is actually blocked — F97 and F100 are documented SC 1.3.4 failures; Medium for landscape layouts that merely degrade.
- **Observable symptom:** Rotating a phone to landscape shows a full-screen "Please rotate your device" overlay, a sideways-rotated page (CSS transform hack), or a broken layout with the sticky header eating half of the short viewport.
- **Likely implementation cause:** Screen Orientation API `screen.orientation.lock('portrait')`; `@media (orientation: landscape)` rules that display a blocking overlay or rotate the page 90° via `transform: rotate()`; a layout only ever designed and tested in portrait widths.
- **Why it matters:** SC 1.3.4 requires that content not "restrict its view and operation to a single display orientation" unless a specific orientation is essential — users with wheelchair-mounted devices physically cannot rotate the screen. Failure F97 covers orientation locking, F100 covers reorient-request messages, and the ACT test rule "Orientation of the page is not restricted using CSS transforms" exists for the transform hack.
- **Static detection:** Grep for `screen.orientation.lock`, and for `orientation: landscape|portrait` media queries that toggle a full-screen overlay or apply `transform: rotate(90deg)` on root containers.
- **Runtime or browser test:** DevTools device-toolbar rotate toggle on an emulated phone; verify all content and functionality remain available in both orientations.
- **Corrective rule:** Content MUST be viewable and operable in both portrait and landscape. Orientation MUST NOT be locked or gated behind a reorient message unless a specific orientation is essential to the content.
- **Valid exceptions:** SC 1.3.4's "essential" cases — bank-check capture, piano apps, projector slides, VR; technique G214 (a control offering an alternate orientation presentation) is sufficient.
- **Primary sources:** WCAG 2.2, SC 1.3.4 — https://www.w3.org/TR/WCAG22/#orientation; Understanding SC 1.3.4 — https://www.w3.org/WAI/WCAG22/Understanding/orientation.html (failures F97, F100; technique G214)
- **Confidence:** High.

### 4.11 Breakpoints chosen by device guesswork leave in-between widths broken
- **Classification:** objectively broken | inefficient
- **Severity:** Medium — the breakpoint choice itself is heuristic, but the resulting untested intermediate states frequently produce objective overflow/clipping failures (issues 4.1–4.5 resurface at 500–900 px).
- **Observable symptom:** The layout is fine at 1440 px and 375 px, but at iPad portrait (768 px), phone landscape (812×375), half-screen desktop split-view (~640–960 px), or a 1024 px small laptop, sidebars overlap content, hero text wraps absurdly, or grids jump directly from 4 columns to 1 with a broken intermediate state.
- **Likely implementation cause:** Media queries keyed to assumed device widths (`@media (max-width: 768px)` as the only breakpoint, or exactly 375/768/1024/1440); desktop-first `max-width` queries piling overrides onto overrides; width used as a proxy for input type (hover-only UI enabled at ≥ 1024 px, breaking touch laptops); no height-based queries at all.
- **Why it matters:** web.dev's responsive guidance is explicit: "Don't define breakpoints based on device classes… let content determine how the layout changes," and content "should not rely on a particular viewport width to render well." A two-state design has exactly one tested point per state; everything between is unverified, and Understanding SC 1.4.10 acknowledges non-canonical viewport sizes (960 px docked windows) as in-scope conditions.
- **Static detection:** Heuristic: exactly one breakpoint in the whole stylesheet, or breakpoints only at 375/768/1024/1440; `max-width` (desktop-first) queries in a mobile-delivered codebase; `@media (min-width: 1024px)` enabling hover-only behaviors (cross-check against `@media (hover: hover)`); absence of any `@media (max-height: …)` for landscape handling.
- **Runtime or browser test:** Automated viewport sweep (Playwright: step 320→1600 px in ~40 px increments, asserting no horizontal overflow and no element overlap via bounding-box intersection checks on key components); manual pass at 640, 768, 834, and 960 px and at 812×375 landscape.
- **Corrective rule:** Breakpoints MUST be chosen where the content actually starts to fail (content-out), tested across the continuous width range 320→1600 px (including landscape and split-view sizes), and MUST NOT be used as a proxy for input modality — use `@media (hover: hover)` / `(pointer: coarse)` for that.
- **Valid exceptions:** Framework-default breakpoints (Tailwind sm/md/lg) are acceptable as an implementation convention provided the intermediate states are actually tested and layouts recompose gracefully between them.
- **Primary sources:** web.dev, Responsive web design basics ("How to choose breakpoints"; interaction media features `hover`/`pointer`/`any-hover`/`any-pointer`) — https://web.dev/articles/responsive-web-design-basics; web.dev, Accessible tap targets (`@media (any-pointer: coarse)` instead of size guessing) — https://web.dev/articles/accessible-tap-targets; Understanding SC 1.4.10 — https://www.w3.org/WAI/WCAG22/Understanding/reflow.html ("Acknowledging viewport vs resolution discrepancies")
- **Confidence:** Medium — the continuous-testing rule is objective; breakpoint numerology is a heuristic proxy for it.

## 5. Accessibility Failures

Accessibility failures are the largest and most mechanically verifiable class of slop in AI-generated frontends, and they cluster around a single structural cause (observed pattern/inference, not sourced fact): generators are trained and prompted on static visual artifacts — mockups, screenshots, component-library snippets — in which keyboard focus, tab order, live contrast under real conditions, and OS-level motion preferences have no visual signature, so the generated code reproduces the look while dropping the behavioral envelope that makes it operable. The pattern is empirically visible in the corpus generators imitate: the WebAIM Million 2026 analysis found that home pages using ARIA averaged 59.1 detected accessibility errors versus 42.0 on pages without it, and WebAIM attributes part of the 2026 error regression to "increased reliance on 3rd party frameworks and libraries, and automated or AI-assisted coding practices ('vibe coding')" (https://webaim.org/projects/million/). A second observed tendency (inference): ARIA is sprinkled decoratively — roles duplicated, labels invented, `aria-hidden` overapplied — because "ARIA present" correlates with "accessible" in training data, which is exactly backwards per the APG's "No ARIA is better than bad ARIA" guidance (https://www.w3.org/WAI/ARIA/apg/practices/read-me-first/). Every issue in this section is anchored to a WCAG 2.2 success criterion with a testable failure condition, so each corrective rule is directly usable as an AI code-review standard. Dialog-internal interaction failures (backdrop, scroll lock, inert background, light dismiss) are catalogued in Section 2; library-level motion implementation failures are catalogued in Section 3 — this section covers only the accessibility-contract angle of those clusters and cross-references both.

### 5.1 Click-only `<div>`/`<span>` "buttons" and links
- **Classification:** objectively broken | inaccessible
- **Severity:** Critical — the control is unreachable and inoperable for every keyboard and assistive-technology user; violates WCAG SC 2.1.1 Keyboard (Level A) and SC 4.1.2 Name, Role, Value (Level A).
- **Observable symptom:** Cards, icon tiles, "×" close icons, and list rows respond to mouse clicks but are skipped entirely by Tab; Space/Enter do nothing on them; screen readers announce them as plain text or not at all.
- **Likely implementation cause:** `<div className="card" onClick={open}>` or `<span onClick={close}>×</span>` with no `role="button"`, no `tabIndex={0}`, no `onKeyDown` for Enter/Space. MDN: interactive components authored using non-interactive elements are not listed in the accessibility tree. (Inference from the brief: a clickable-looking card in a screenshot is reproduced by the shortest path, `<div onClick>`, and native `<button>` UA styles get "cleaned up," so divs dominate.)
- **Why it matters:** The W3C "Using ARIA" first rule mandates native elements (`<button>`, `<a href>`), which supply focusability, key behavior, role, and states for free; the third rule requires all interactive ARIA controls to be keyboard-operable. A clickable div satisfies neither.
- **Static detection:** ESLint `jsx-a11y/click-events-have-key-events`, `jsx-a11y/no-static-element-interactions`, `jsx-a11y/no-noninteractive-element-interactions`, `jsx-a11y/interactive-supports-focus`; AST grep for `onClick=` on `div|span|li|article|section` elements.
- **Runtime or browser test:** Manual keyboard walk (authoritative): every mouse-clickable element must be focusable, and Enter and Space must trigger the same action. axe-core catches some instances via name/experimental focus rules; do not rely on it alone.
- **Corrective rule:** Elements that perform an action MUST be native `<button>` (or `<a href>` for navigations). A non-native element may act as a control ONLY with an appropriate role, `tabindex="0"`, Enter AND Space key handling, and a visible focus indicator — full parity with the native element, per Using ARIA rules 1 and 3.
- **Valid exceptions:** Genuinely non-interactive containers that only augment an inner real control (e.g., a card wrapper whose title contains the actual link); pointer-only conveniences that duplicate an accessible control (e.g., map panning).
- **Primary sources:** Using ARIA (first/third rules) — https://www.w3.org/TR/using-aria/ ; WCAG SC 2.1.1 — https://www.w3.org/TR/WCAG22/#keyboard ; Understanding SC 2.1.1 — https://www.w3.org/WAI/WCAG22/Understanding/keyboard.html ; WCAG SC 4.1.2 — https://www.w3.org/TR/WCAG22/#name-role-value ; MDN `tabindex` "Accessibility concerns" — https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Global_attributes/tabindex
- **Confidence:** High — Level A conformance issue, mechanically detectable.

### 5.2 Custom select/combobox/autocomplete rebuilt as `<div>`s without the APG contract
- **Classification:** objectively broken | inaccessible
- **Severity:** Critical — fails SC 2.1.1 (keyboard inoperable) and SC 4.1.2 (no programmatic role/state/value), both Level A; the control works only for sighted pointer users.
- **Observable symptom:** A styled "Select country" opens on click, but Tab skips it or lands on a dead div; Arrow keys do nothing; Enter doesn't open/select; Escape doesn't close; screen readers announce "group" or nothing and never report the current value, expanded state, or option count.
- **Likely implementation cause:** Generated markup is a click-only shell — `<div class="select"><div class="selected">…</div><ul><li onClick=…>` — with no `role="combobox"`, `aria-expanded`, `aria-controls`, `aria-activedescendant`, no `role="listbox"`/`role="option"`, no key map; or partial ARIA (a `role="listbox"` with no options, an `aria-expanded` that never flips). (Inference: native `<select>` is nearly unstyleable, mockups show custom dropdowns, and the APG keyboard map has no visual signature.)
- **Why it matters:** The APG Combobox pattern defines the full contract: input/button with `role="combobox"` + `aria-expanded` + `aria-controls` → popup `role="listbox"` with `role="option"` children; DOM focus stays on the input while `aria-activedescendant` tracks the active option; Arrow Down opens/moves, Enter accepts, Escape closes, Home/End jump; a live region announces result counts (SC 4.1.3 Status Messages). WebAIM Million 2026: of pages using ARIA menus, 22% introduced accessibility barriers due to missing required markup/interactions — the same phenomenon.
- **Static detection:** Grep for `role="listbox"` without `role="option"` children; `aria-expanded` on non-combobox divs; classNames matching `/select|dropdown|autocomplete|combobox/` on `div` + `onClick` with no `onKeyDown`; review checklist for the five combobox ARIA attributes plus the key map.
- **Runtime or browser test:** Keyboard script: Tab to control → Arrow Down opens list → arrows move highlight → Enter commits → Escape closes → Tab leaves. Screen reader (NVDA/VoiceOver): value, expanded state, active option, and result count announced. axe-core catches the structural half (`aria-required-children`, `aria-required-attr`).
- **Corrective rule:** A custom select/autocomplete MUST implement the APG Combobox pattern end-to-end (roles, states, `aria-activedescendant`, full key map) or MUST NOT be custom: prefer native `<select>` or `<input>` + `<datalist>`, or a vetted headless component (Radix, React Aria, Headless UI) over a hand-rolled div.
- **Valid exceptions:** None for a from-scratch div widget that skips the contract; native elements and APG-conformant libraries are the compliant paths.
- **Primary sources:** APG Combobox pattern — https://www.w3.org/WAI/ARIA/apg/patterns/combobox/ ; APG Developing a Keyboard Interface — https://www.w3.org/WAI/ARIA/apg/practices/keyboard-interface/ ; Using ARIA third rule — https://www.w3.org/TR/using-aria/ ; WCAG SC 2.1.1 — https://www.w3.org/TR/WCAG22/#keyboard ; WCAG SC 4.1.2 — https://www.w3.org/TR/WCAG22/#name-role-value ; WebAIM Million 2026 — https://webaim.org/projects/million/
- **Confidence:** High — the APG defines the exact required behavior; absence is objectively verifiable.

### 5.3 Focus indicator removed or below visible-contrast minimum
- **Classification:** inaccessible
- **Severity:** Critical — direct failure of WCAG SC 2.4.7 Focus Visible (Level AA); sighted keyboard-only users can no longer operate the page at all. Combines SC 1.4.11 Non-text Contrast and named failure technique F78.
- **Observable symptom:** Tabbing through the page produces no visible change, or a barely-visible pale ring (measured: `sky-200` #BAE6FD ring on white = 1.33:1 against a 3:1 threshold); focus also disappears inside dark sections when the ring color was chosen for light backgrounds.
- **Likely implementation cause:** Reset-style CSS (`:focus { outline: none; }`, `* { outline: 0; }`) or Tailwind `outline-none` / `focus:outline-none` without a compensating `focus-visible:ring-*`, box-shadow, border, or background change; ring tokens (`ring-sky-200`, `ring-indigo-100`) chosen for looks. (Inference: "remove the ugly outline" snippets are overrepresented in training CSS, and static screenshots never show keyboard focus.)
- **Why it matters:** SC 2.4.7 requires a visible keyboard focus indicator; F78 names `:focus {outline: none}` as the canonical failing example; SC 1.4.11 requires the indicator's contrasting portion to be ≥3:1 against adjacent colors; WCAG 2.2 adds SC 2.4.11 Focus Not Obscured (Minimum, AA) for indicators hidden under sticky elements.
- **Static detection:** Grep `outline\s*:\s*(none|0)\b`, `\boutline-none\b`, `\bfocus:outline-none\b` and `box-shadow: none` near `:focus` without a co-located `focus-visible:`/`focus:` ring, outline, or border replacement; a mid-tone ring such as `indigo-500` on white passes at 4.47:1 — the failure is the *light* ring or the *missing* ring. axe cannot fully detect this; code review must.
- **Runtime or browser test:** Keyboard-only walk through every interactive element across every background section (F78 test procedure); Playwright: Tab to each element and assert computed `outline`/`box-shadow` differs from the unfocused state under `:focus-visible`; Chrome DevTools → force `:focus-visible` state. (Non-text contrast of other components: see Section 5.11.)
- **Corrective rule:** Every focusable element MUST present a clearly visible focus indicator during keyboard focus (SC 2.4.7) whose contrasting area achieves ≥3:1 against adjacent colors (SC 1.4.11); removing the UA outline is permitted ONLY with an equally or more visible author-supplied replacement (technique G195); prefer `focus-visible:` styles with a 2px+ solid indicator and offset. Never ship `outline-none` without an equivalent replacement.
- **Valid exceptions:** UA-default focus styles the author has not modified are exempt from the 1.4.11 contrast requirement; `outline: none` on a `tabindex="-1"` programmatic-focus container (e.g., a dialog wrapper) is acceptable because the user did not Tab to it — but focusable controls inside must still show indicators.
- **Primary sources:** WCAG SC 2.4.7 — https://www.w3.org/TR/WCAG22/#focus-visible ; Understanding SC 2.4.7 — https://www.w3.org/WAI/WCAG22/Understanding/focus-visible.html ; Failure F78 — https://www.w3.org/WAI/WCAG22/Techniques/failures/F78.html ; Understanding SC 1.4.11 (focus-indicator contrast, Figures 8–14) — https://www.w3.org/WAI/WCAG22/Understanding/non-text-contrast.html ; SC 2.4.11 — https://www.w3.org/TR/WCAG22/#focus-not-obscured-minimum
- **Confidence:** High — named WCAG failure technique with a test procedure.

### 5.4 Focus management failures: modals that leak or lose focus, focus dropped after updates and route changes
- **Classification:** objectively broken | inaccessible
- **Severity:** Critical for the modal case — a modal that leaks focus breaks SC 2.4.3 Focus Order (Level A) and is unusable for keyboard/AT users; High for focus dropped after dynamic updates; Medium for route changes (best practice grounded in SC 2.4.3/2.4.1).
- **Observable symptom:** (a) With a modal open, Tab moves the focus ring onto links/buttons in the dimmed background page; (b) opening a modal leaves focus on the now-hidden trigger, and closing it dumps focus on `<body>`; (c) deleting a list row or chip with the keyboard makes the next Tab press restart from the browser chrome; (d) in an SPA, activating a link swaps the view while focus dies on the removed element and the new page starts reading from nowhere.
- **Likely implementation cause:** Overlays rendered as portaled, conditionally-rendered `div`s with no `.focus()` on open, no Tab/Shift+Tab containment, no stored reference to the invoking element, no Escape handler; state handlers model data (`items.filter(...)`) but never move focus when the focused node unmounts; routers re-render the outlet without moving focus to the new view's heading/main or updating `document.title`. (Inference: focus choreography is a two-event correlation invisible in every screenshot, demo GIF, and single-shot generation.)
- **Why it matters:** The APG Dialog (Modal) pattern is explicit: on open, focus moves inside; Tab/Shift+Tab are contained while open; Escape closes; on close, "focus returns to the element that invoked the dialog," or, if it no longer exists, to "another element that provides logical work flow" — the same logical-workflow rule applies to list deletions and filtering. `aria-modal="true"` on a dialog that does not enforce modality has, per the APG, "severe negative ramifications" for AT users. WCAG Understanding SC 2.1.2 cites the modal with keyboard dismissal as the mechanism preventing an effective trap. For overlays generally, failure F85 covers dialogs not adjacent to their trigger in sequential navigation order.
- **Static detection:** Overlay components lacking `ref`/`.focus()` on open, a Tab-cycling keydown handler (or focus-trap/inert/native `<dialog>`), an Escape handler, and a stored trigger ref; every destructive/removal handler (delete/remove/filter/sort) without a sibling focus-restoration statement; router config lacking an after-navigation focus/title effect. axe-core `aria-dialog-name` catches unnamed dialogs; run scans with the dialog OPEN (closed overlays are invisible to static scans).
- **Runtime or browser test:** Keyboard script: activate trigger → `document.activeElement` inside the dialog → Tab/Shift+Tab cycle within it only → Escape closes → focus back on the trigger. Delete item *i* via keyboard → assert focus lands on the next/previous item or a named container, not `body`. SPA: activate an in-app link → focus on the new view's `h1`/`main` (with `tabindex="-1"`) and `document.title` updated. All automatable in Playwright.
- **Corrective rule:** Modal overlays MUST move focus inside on open, contain Tab/Shift+Tab with wrap while open, close on Escape, and return focus to the invoking element on every close path (close button, Escape, backdrop, programmatic) or to a logical-workflow element if the invoker no longer exists; background content MUST be inert. Whenever an action removes, disables, or reorders the focused element, focus MUST move in the same commit to the logical next element. Client-side route changes SHOULD move focus to the new view's top-level heading or main landmark and update the document title. Native `<dialog>` with `showModal()` provides much of this for free and is the preferred default.
- **Valid exceptions:** Non-modal panels/popovers intentionally do not trap focus — then they must not claim `aria-modal="true"` and must still close on Escape with focus return; toasts and alerts must NOT take focus; hash-anchor navigation may rely on native anchor focus; deletions confirmed inside a modal that itself returns focus are already managed.
- **Primary sources:** APG Dialog (Modal) pattern — https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/ ; WAI-ARIA 1.2 `dialog` role — https://www.w3.org/TR/wai-aria-1.2/#dialog ; MDN `<dialog>` — https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Elements/dialog ; WCAG SC 2.4.3 — https://www.w3.org/TR/WCAG22/#focus-order ; Understanding SC 2.4.3 (F85) — https://www.w3.org/WAI/WCAG22/Understanding/focus-order.html ; Understanding SC 2.1.2 (modal example) — https://www.w3.org/WAI/WCAG22/Understanding/no-keyboard-trap.html ; APG Developing a Keyboard Interface — https://www.w3.org/WAI/ARIA/apg/practices/keyboard-interface/
- **Confidence:** High for modals (APG + WCAG-aligned; named vendor audit criterion in the brief); Medium for route changes (established best practice without a single dedicated SC). Cross-reference: backdrop, scroll-lock, inert-background, light-dismiss, and stacking failures of the same dialog cluster are covered in Section 2.

### 5.5 Focus-order defects: positive `tabindex` and missing bypass mechanism
- **Classification:** objectively broken | inaccessible
- **Severity:** High for positive `tabindex` (breaks SC 2.4.3 Focus Order, Level A, whenever the sequence diverges from reading order; axe-core rates it Serious); Medium for a missing skip link (SC 2.4.1 Bypass Blocks, Level A — costs keyboard users dozens of Tab presses per page rather than blocking them outright).
- **Observable symptom:** (a) The Tab key jumps erratically around the page — from the search box to a footer link and back to the header — because `tabindex="1"`, `"2"`, … pull those elements to the front of the sequence in numeric order; (b) on every page load, keyboard users must Tab through the entire header/nav (logo, 5–7 links, search, CTA, hamburger) before reaching the content.
- **Likely implementation cause:** Positive `tabindex` values emitted to force a visual/DOM-mismatched order instead of reordering the DOM (legacy-tutorial pattern, per the brief's inference); layouts emit `<header>` + nav with no first-focusable skip link pointing at `<main id="main">`, or a skip link whose target lacks `id`/`tabindex="-1"` so activation never moves focus. (Inference: skip links are invisible in screenshots — they appear only on focus — so they are absent from the visual priors.)
- **Why it matters:** SC 2.4.3 requires the tab sequence to preserve meaning and operability; MDN: "only use `0` and `-1` as `tabindex` values… write the document with the elements in a logical sequence." SC 2.4.1 requires a mechanism to bypass blocks repeated across pages; technique G1 (skip link at page top that actually moves focus to the main area) is the canonical sufficient technique.
- **Static detection:** Grep `tabindex\s*=\s*["']?[1-9]`; ESLint `jsx-a11y/tabindex-no-positive`; axe-core rule `tabindex`. For bypass: check the base layout for an early `<a href="#...">` with /skip/i text targeting a `<main>` with matching `id`; axe-core rules `bypass` and `skip-link`.
- **Runtime or browser test:** Tab from the address bar and compare the sequence against visual reading order. Load page, press Tab once: a "Skip to main content" control should appear; press Enter: focus must land in/at the main landmark and the next Tab must enter main-content links.
- **Corrective rule:** Positive `tabindex` values MUST NOT appear in markup; tab order MUST come from DOM order plus `tabindex="0"` (add to sequence) and `tabindex="-1"` (programmatic focus only). Any page with repeated blocks MUST provide, as one of the first focusable elements, a working bypass whose activation moves focus to the main landmark; it may be visually hidden until focused (G1 explicitly allows this).
- **Valid exceptions:** `tabindex="-1"` on programmatically focused targets (dialog containers, skip-link targets, error summaries) is correct and is not this failure; pages with no repeated blocks need no bypass; well-formed landmarks alone can serve AT users, but a G1-style skip link remains the robust cross-audience mechanism.
- **Primary sources:** MDN `tabindex` — https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Global_attributes/tabindex ; WCAG SC 2.4.3 — https://www.w3.org/TR/WCAG22/#focus-order ; Understanding SC 2.4.3 — https://www.w3.org/WAI/WCAG22/Understanding/focus-order.html ; WCAG SC 2.4.1 — https://www.w3.org/TR/WCAG22/#bypass-blocks ; Technique G1 — https://www.w3.org/WAI/WCAG22/Techniques/general/G1 ; axe-core rule descriptions — https://github.com/dequelabs/axe-core/blob/develop/doc/rule-descriptions.md
- **Confidence:** High — Level A SCs with named techniques and lint/axe coverage.

### 5.6 Keyboard traps in embedded widgets and key-capturing components
- **Classification:** objectively broken | inaccessible
- **Severity:** Critical when inescapable — WCAG SC 2.1.2 No Keyboard Trap (Level A) is among the most severe failures: the user is stuck and cannot continue using the page; High when an escape exists but is undocumented.
- **Observable symptom:** Tabbing into an embedded code editor, map, video player, or third-party chat/payment iframe swallows all further Tab/arrow input; Shift+Tab doesn't escape either; keyboard-only users, who have no mouse, are stranded.
- **Likely implementation cause:** Third-party iframes (editors, maps, media players) that capture keys, copied from vendor snippets optimized for engagement; custom keydown handlers that `preventDefault()` Tab; focus loops between two widgets. (Inference from the brief: key-capture ships inside the iframe where generated code never inspects it, and generated `onKeyDown` handlers frequently preventDefault broadly without preserving Tab.)
- **Why it matters:** SC 2.1.2: "If keyboard focus can be moved to a component of the page using a keyboard interface, then focus can be moved away from that component using only a keyboard interface." Technique G21 is the corresponding general technique. Where containment is intentional — a true modal — the Section 5.4 contract (containment + Escape release) is the compliant form.
- **Static detection:** Inventory all `<iframe>` embeds and custom `keydown` listeners; flag `preventDefault()` calls that include the Tab key; grep `keydown` handlers for missing `key === 'Escape'`/`key === 'Tab'` pass-through; check vendor docs of each embed for keyboard-egress behavior.
- **Runtime or browser test:** Manual keyboard-only gate (no reliable automated check): Tab into every embed/widget and confirm Tab, Shift+Tab, or a documented alternative (e.g., Ctrl+M in editors, Escape) moves focus out; verify NVDA/VoiceOver browse-mode egress.
- **Corrective rule:** Every focusable component MUST allow keyboard focus to leave it via unmodified standard keys (Tab/Shift+Tab) or a documented, advertised alternative; key handlers MUST NOT swallow Tab. Embeds that trap without an escape mechanism MUST be wrapped, replaced, or given an adjacent documented bypass.
- **Valid exceptions:** Genuine modal dialogs contain focus by design but MUST release on Escape/close per the APG; code editors may intercept Tab for indentation only when they provide and announce an alternative focus-escape keystroke.
- **Primary sources:** WCAG SC 2.1.2 — https://www.w3.org/TR/WCAG22/#no-keyboard-trap ; Understanding SC 2.1.2 — https://www.w3.org/WAI/WCAG22/Understanding/no-keyboard-trap.html ; Technique G21 — https://www.w3.org/WAI/WCAG22/Techniques/general/G21 ; WCAG SC 2.1.1 — https://www.w3.org/TR/WCAG22/#keyboard
- **Confidence:** High — Level A SC; severity is case-dependent as noted.

### 5.7 `aria-hidden="true"` on focusable elements or their containers
- **Classification:** objectively broken | inaccessible
- **Severity:** Critical — focus lands on "nothing": keyboard users can Tab onto controls that are invisible to assistive technology, and screen-reader users get silence where operable UI exists; violates Using ARIA's fourth rule and SC 4.1.2 / SC 1.3.1.
- **Observable symptom:** Tab focus visibly moves to a button or link, but the screen reader announces nothing (or the virtual cursor skips the whole region) — e.g., a mobile nav marked `aria-hidden="true"` while still displayed and tabbable, or icon buttons inside an `aria-hidden` toolbar.
- **Likely implementation cause:** Pattern-matched "hide from screen readers" handling: a desktop nav and mobile nav both rendered, one "hidden" with `aria-hidden="true"` instead of `display:none` or `inert`; decorative-icon hygiene overapplied to parents (`<div aria-hidden="true"><button>…`); modals hide the background with `aria-hidden` but fail to make it unfocusable. WebAIM Million 2026 measured an average of 23.3 `aria-hidden="true"` attributes per home page — heavy cargo-cult usage.
- **Why it matters:** MDN: "Do not use `aria-hidden="true"` on focusable elements," nor on their ancestors; the ARIA-in-HTML spec excludes focusable elements and `body` from `aria-hidden`'s allowance; Using ARIA's fourth rule: doing so "will result in some users focusing on 'nothing.'"
- **Static detection:** ESLint `jsx-a11y/no-aria-hidden-on-focusable`; axe-core rule `aria-hidden-focus` ("Ensure aria-hidden elements are not focusable nor contain focusable elements", Serious); review heuristic: any `aria-hidden="true"` on an element that has `tabindex` or wraps interactive descendants.
- **Runtime or browser test:** `axe.run()` catches it; manual: Tab the page with the DevTools Accessibility tree open — no focusable node may sit under an `aria-hidden` branch; check responsive breakpoints separately (the mobile-nav path is the usual offender).
- **Corrective rule:** `aria-hidden="true"` MUST NOT be applied to focusable elements or to ancestors of focusable elements. Content that should be unreachable by keyboard and AT MUST instead be removed via `display:none`, the `hidden` attribute, or the `inert` attribute. Reserve `aria-hidden` for decorative, non-interactive content.
- **Valid exceptions:** Truly decorative, non-focusable content (icon glyph spans inside a named button); `aria-hidden` on modal backgrounds is legitimate only when the background is simultaneously made unfocusable — which is exactly what `inert`/`showModal()` provide.
- **Primary sources:** MDN `aria-hidden` — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Attributes/aria-hidden ; WAI-ARIA 1.2 `aria-hidden` — https://www.w3.org/TR/wai-aria-1.2/#aria-hidden ; ARIA in HTML (focusable exception) — https://www.w3.org/TR/html-aria/ ; Using ARIA §2.4 fourth rule — https://www.w3.org/TR/using-aria/ ; axe-core rule descriptions — https://github.com/dequelabs/axe-core/blob/develop/doc/rule-descriptions.md ; WebAIM Million 2026 — https://webaim.org/projects/million/
- **Confidence:** High — explicitly prohibited by multiple sources plus a dedicated axe rule.

### 5.8 Broken name/role/value: invalid ARIA attributes, dangling ID references, mismatched accessible names
- **Classification:** objectively broken | inaccessible
- **Severity:** High — SC 4.1.2 Name, Role, Value (Level A) and SC 2.5.3 Label in Name (Level A): names, states, and relationships the author intended are silently not exposed, and nothing errors at build time because browsers ignore unknown attributes.
- **Observable symptom:** A screen reader announces a control with no name, a missing description, or a state that never changes; intended associations (error text ↔ input) don't exist for AT users; a button visibly reading "Save draft" does not respond when a Voice Control user says "click Save draft" because `aria-label="Persist record"` overwrote the accessible name.
- **Likely implementation cause:** Typos (`aria-lable`, `aria-haspopup="ture"`), copy-pasted IDs that no longer exist after refactoring (`aria-controls="menu-1"` after the menu was renamed), duplicate IDs from list rendering, values outside the attribute's token set; `aria-label` applied on top of visible text, where it takes complete precedence in the accessible-name computation and erases the visible string for AT. (Inference: ARIA attributes are unchecked plain strings; cross-file ID wiring is exactly the consistency single-shot generation gets wrong.)
- **Why it matters:** SC 4.1.2 requires name/role/value to be programmatically determinable — a mistyped attribute or dangling IDREF means the intended accessibility information silently does not exist; axe-core rates `aria-valid-attr`/`aria-valid-attr-value` Critical. SC 2.5.3 requires the accessible name of a visibly-labeled control to contain the visible label text (technique G208), so speech-input users can activate controls by speaking what they see.
- **Static detection:** ESLint `jsx-a11y/aria-props`, `aria-proptypes`, `role-has-required-aria-props`, `role-supports-aria-props`; axe-core rules `aria-valid-attr`, `aria-valid-attr-value`, `aria-required-attr`, `aria-allowed-attr`, `duplicate-id-aria`, `label-content-name-mismatch`; script check that every `aria-labelledby/describedby/controls/owns/activedescendant` ID exists exactly once; AST comparison of `aria-label` strings against visible text content.
- **Runtime or browser test:** `axe.run()` (these rules run by default) in CI via `@axe-core/playwright`; DevTools Accessibility pane: compare computed "Name" against rendered text for every control; Voice Control / Dragon: speak the visible label and verify activation.
- **Corrective rule:** Every `aria-*` attribute MUST be a defined WAI-ARIA attribute with a value valid for its type, and every ID-reference attribute MUST resolve to exactly one existing element — CI MUST fail on `aria-valid-attr`/`aria-valid-attr-value`/`duplicate-id-aria` violations. Where visible text labels a control, the accessible name MUST contain that text verbatim, preferably leading (SC 2.5.3); if the visible text is sufficient, add no `aria-label` at all (first rule of ARIA).
- **Valid exceptions:** None for invalid attributes or dangling IDREFs. Controls whose visible label is purely iconic (an "×" glyph) legitimately need an author-provided name; descriptive text may follow the visible label inside the name ("Save draft — unsaved changes are kept 30 days" contains "Save draft" → passes).
- **Primary sources:** WCAG SC 4.1.2 — https://www.w3.org/TR/WCAG22/#name-role-value ; Understanding SC 4.1.2 — https://www.w3.org/WAI/WCAG22/Understanding/name-role-value.html ; WAI-ARIA 1.2 — https://www.w3.org/TR/wai-aria-1.2/ ; WCAG SC 2.5.3 — https://www.w3.org/TR/WCAG22/#label-in-name ; Understanding SC 2.5.3 — https://www.w3.org/WAI/WCAG22/Understanding/label-in-name.html ; Technique G208 — https://www.w3.org/WAI/WCAG22/Techniques/general/G208 ; axe-core rule descriptions — https://github.com/dequelabs/axe-core/blob/develop/doc/rule-descriptions.md
- **Confidence:** High — fully machine-checkable against the ARIA 1.2 attribute registry and an ACT rule (2ee8b8) behind the label-name check.

### 5.9 ARIA over-application and broken document structure: redundant or overriding roles, landmark soup, skipped headings
- **Classification:** inefficient | inaccessible
- **Severity:** Medium — degrades screen-reader navigation rather than blocking it; roles that override native semantics (e.g., `<h2 role="tab">`, `<button role="link">`) are High for those instances. Maps to SC 1.3.1 Info and Relationships (Level A) when structure is misrepresented; otherwise axe best-practice (`heading-order`, landmark rules).
- **Observable symptom:** Screen readers announce double or incorrect semantics ("button button"); heading-jump navigation hears `h1` → `h4` → `h2` and cannot infer the document structure; the landmark rotor lists "navigation, navigation, navigation, region, region…" with no names, burying the useful landmarks.
- **Likely implementation cause:** Pattern-matched sprinkling: `<button role="button">`, `<nav role="navigation">`, `<main role="main">`; roles copied onto elements that already carry those implicit semantics or used to re-purpose elements (`<a role="button">` without href); multiple unlabeled `<nav>`/`<aside>` per page; heading level chosen for font size (`<h4 className="text-xs uppercase">`) or hard-coded per component regardless of mounting depth. (Inference: redundant landmark roles are a 2009–2015 compatibility pattern frozen into training HTML, "add ARIA for accessibility" priors cause over-application, and component-local generation cannot coordinate page-level structure. WebAIM Million 2025 found skipped heading levels on ~39% of pages.)
- **Why it matters:** The ARIA-in-HTML spec marks roles matching an element's implicit ARIA semantics NOT RECOMMENDED ("unnecessary and can potentially lead to unintended consequences"); Using ARIA's second rule: "Do not change native semantics, unless you really have to." Landmarks help only when few and distinguishable; when several of the same type exist, each needs a distinguishing accessible name. MDN: "Do not skip heading levels… Do not use heading elements to resize text" — headings are the primary in-page screen-reader navigation mechanism.
- **Static detection:** ESLint `jsx-a11y/no-redundant-roles`; W3C Nu Html Checker (reports unnecessary roles); grep `role="(button|navigation|main|banner|contentinfo|list)"` against the host element's implicit semantics; count landmarks in rendered markup — >1 `nav`/`complementary`/`region` without unique names; build a heading outline and flag level jumps > 1 and `<p>`/`<div>` styled as headings; axe rules `aria-allowed-role`, `heading-order`, `p-as-heading`, `landmark-unique`, `landmark-no-duplicate-*`, `landmark-one-main`, `region`.
- **Runtime or browser test:** axe DevTools / WAVE headings-outline and landmarks views; screen-reader heading list and landmark rotor: every landmark distinct and meaningful, exactly one `main`, one top-level `banner`, one top-level `contentinfo`; Playwright: collect all `h1–h6` and assert no skipped increments.
- **Corrective rule:** Markup MUST NOT assign roles matching the element's implicit ARIA semantics and MUST NOT override native semantics where a correct native element exists (Using ARIA rules 1–2); when two or more landmarks of the same type exist, each MUST have a unique accessible name (`aria-label`/`aria-labelledby`); `<section>` becomes a region only when named. Heading levels MUST reflect document structure — one logical `<h1>`, no skipped levels, visual sizing done in CSS; reusable components SHOULD accept a heading-level prop instead of hard-coding.
- **Valid exceptions:** A single unlabeled `<nav>` per page is fine; `role="search"` is a useful addition; closing a subsection may legitimately "skip" upward (h4 → h2 when the h3 subsection ends — MDN explicitly allows this); depth beyond h6 legitimately uses `role="heading" aria-level="7"`.
- **Primary sources:** ARIA in HTML (implicit semantics, NOT RECOMMENDED) — https://www.w3.org/TR/html-aria/ ; Using ARIA §2.2 second rule — https://www.w3.org/TR/using-aria/ ; MDN Heading elements — https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Elements/Heading_Elements ; WCAG SC 1.3.1 — https://www.w3.org/TR/WCAG22/#info-and-relationships ; Technique G141 — https://www.w3.org/WAI/WCAG22/Techniques/general/G141 ; axe-core rule descriptions — https://github.com/dequelabs/axe-core/blob/develop/doc/rule-descriptions.md ; WebAIM Million — https://webaim.org/projects/million/
- **Confidence:** High that these degrade AT navigation and violate authoring conformance; Medium severity because visual users are unaffected and tasks are rarely blocked outright.

### 5.10 Text contrast below 4.5:1: brand-fill buttons, gray "secondary" text, placeholder-as-label, text over imagery
- **Classification:** inaccessible
- **Severity:** High — hard WCAG SC 1.4.3 Contrast (Minimum) (Level AA) failure on the most-read and most-clicked content in the UI; some variants are severe (2.5:1). The placeholder-as-label variant is a double failure adding SC 3.3.2 Labels or Instructions (Level A).
- **Observable symptom:** (a) Primary CTA labels washed out on mid-tone brand fills — measured: white on `indigo-500` #6366F1 = 4.47:1 (fails; no rounding permitted), `violet-500` = 4.23:1, `blue-500` = 3.68:1, `sky-500` = 2.77:1, `emerald-500` = 2.54:1; (b) timestamps, captions, helper text, and card subtitles in faint gray — `gray-400` on white = 2.54:1, `slate-400` = 2.56:1; (c) forms whose only label is gray placeholder text (~2.5:1) that disappears once the user types; (d) white hero text over photos or multi-stop gradients that vanishes over bright regions, with contrast unverifiable and varying per image.
- **Likely implementation cause:** `<button class="bg-indigo-500 text-white">` — the `-500` step picked as "the brand color" without a ratio check; `text-gray-400` / `text-muted-foreground` tokens never contrast-checked against their paired backgrounds; `<input placeholder="Email">` emitted without a `<label>`; absolutely-positioned text over `<img>`/`background-image`/`linear-gradient` with no scrim and no per-region verification. (Inference: muted grays, mid-tone fills, and hero-over-photo compositions dominate the static training corpus, which never demonstrates sunlight legibility, filled-form state, or per-glyph background luminance.)
- **Why it matters:** SC 1.4.3 requires 4.5:1 for normal text and 3:1 for large text (≥18pt/24px, or ≥14pt/~18.5px bold); computed ratios must not be rounded — 4.49:1 fails. The criterion makes no "secondary text" exception and explicitly applies to placeholder text; Understanding lists failure F83 for background images that do not provide sufficient contrast, and with imagery the least-contrasting region behind any glyph must pass. When the placeholder is the only label, SC 3.3.2 (labels or instructions for input) fails the moment the field is filled; MDN documents that placeholders are not a `<label>` replacement and interfere with form completion.
- **Static detection:** Grep `bg-(indigo|violet|blue|sky|emerald|cyan|teal|green|rose|pink|fuchsia|purple|orange|amber|red)-(400|500)` co-occurring with `text-white`; grep `text-(gray|slate|zinc|neutral|stone)-(300|400)\b`; check `--muted-foreground`-style tokens against paired background tokens; AST check for `<input>`/`<textarea>` with `placeholder=` but no `<label htmlFor>`/`aria-label`/`aria-labelledby`; flag text elements overlapping media or `bg-gradient-to-*` without an intermediate scrim. Maintain a project lookup table of allowed bg/fg pairs.
- **Runtime or browser test:** axe-core `color-contrast` rule (accounts for opacity) via axe DevTools or `@axe-core/playwright`; Chrome DevTools → CSS Overview → Contrast issues; DevTools color picker shows the computed ratio with AA/AAA lines. Known blind spot: axe `color-contrast` does NOT report text over `background-image` — verify those manually with the eyedropper at the lightest point under text while resizing the viewport. Placeholder test: fill the field and check whether any label persists.
- **Corrective rule:** All text conveying information MUST meet ≥4.5:1 against its actual background (≥3:1 only for large text per SC 1.4.3 definitions) — for Tailwind brand buttons typically `-600`/`-700` fills with white text (white on `indigo-600` = 6.29:1) or dark text on `-300`/`-400` fills; de-emphasis must stay within the passing range (`gray-500` on white = 4.83:1, size/weight changes). Every input MUST have a persistent visible label (or `aria-label` for true icon-adjacent cases); placeholder text is supplementary, must itself meet ≥4.5:1, and surviving hints belong in an `aria-describedby` element. Text over variable backgrounds MUST use a mechanism guaranteeing ≥4.5:1 at every glyph — uniform scrim, gradient scrim over the text region, solid backdrop, or a restricted image area; "it looked fine on the designer's image" is not conformance.
- **Valid exceptions:** Genuinely inactive controls (SC 1.4.3 "Incidental"), logotypes, and purely decorative text are exempt; large text meeting 3:1 passes; text over an image region verified to pass at its lightest point is conformant; search boxes with an adjacent labeled button may rely on `aria-label`.
- **Primary sources:** Understanding SC 1.4.3 (thresholds, no-rounding, placeholder inclusion, F83, inactive exception) — https://www.w3.org/WAI/WCAG22/Understanding/contrast-minimum.html ; WCAG SC 3.3.2 — https://www.w3.org/TR/WCAG22/#labels-or-instructions ; MDN `::placeholder` — Accessibility — https://developer.mozilla.org/en-US/docs/Web/CSS/::placeholder ; Tailwind CSS Colors (palette values) — https://tailwindcss.com/docs/colors ; Deque axe `color-contrast` (opacity handling, background-image blind spot) — https://dequeuniversity.com/rules/axe/4.10/color-contrast
- **Confidence:** High — normative AA threshold with deterministic computation.

### 5.11 Non-text visuals below 3:1 and meaning carried by color alone
- **Classification:** inaccessible | deceptive
- **Severity:** High — SC 1.4.1 Use of Color (Level A; failure F81 explicitly covers required/error fields identified by color differences only) and SC 1.4.11 Non-text Contrast (Level AA) fail simultaneously on forms, status displays, and charts — the data-dense core of the page. The boundary-only variant (input borders) is Medium when a visible label or placeholder identifies the field, which the Understanding doc counts as identifying information.
- **Observable symptom:** (a) A failed form field just gets a red border — no icon, no message — and success is a green flash; deuteranopes cannot tell which fields failed (red-500 vs green-500 = 1.65:1 lightness difference, below the 3:1 that would let luminance substitute for hue; ~8% of men have color-vision deficiency); (b) the error message itself, `text-red-500` #EF4444 on white at 3.76:1, also fails SC 1.4.3 for small text; (c) inputs and card hit areas bounded by `border-gray-200` (1.24:1) / `border-gray-300` (1.47:1) on white are nearly invisible; (d) chart series differ only in hue at near-equal luminance, gridlines render in `gray-300` (1.47:1), and unlabeled adjacent pie slices cannot be distinguished (Understanding 1.4.11 Figure 20 failure); (e) a "Submit" button dimmed with `opacity-50` + `cursor-not-allowed` looks disabled but still fires — and its label composites to ~2.2:1, failing SC 1.4.3 because a still-active control is not exempt as "inactive."
- **Likely implementation cause:** Conditional classes only — `className={error ? "border-red-500" : "border-gray-200"}` with no `<p role="alert">`, icon, or `aria-invalid`/`aria-describedby`; default Tailwind light-gray border utilities copied from mockups; chart-library quick-start palettes with legend-only labeling and no dash/pattern/direct-label channel; `opacity` applied to a whole active button as a one-class "loading" state. (Inference: "red border = error" and hue-only chart palettes are the canonical training-data visuals; borders and series luminance are never evaluated against adjacent colors, and generators do not model that opacity re-computes the contrast of every child.)
- **Why it matters:** SC 1.4.1 prohibits color as the only visual means of conveying information or prompting a response; a lightness difference counts as the additional cue only at ≥3:1. SC 1.4.11 requires visual information needed to identify UI components and graphical objects required to understand content at ≥3:1 against adjacent colors (the Understanding doc's passing boundary example is `#767676` on white; its pie-chart case study fails unlabeled adjacent slices that don't contrast). SC 1.4.3's inactive-component exception covers genuinely inactive controls only — an activatable control's opacity-reduced label must still pass 4.5:1, and axe's `color-contrast` rule accounts for opacity.
- **Static detection:** AST/grep for conditional class strings toggling only `border-(red|green|amber)-*` / `text-(red|green)-*` on state with no sibling icon/message/aria attribute; grep `border-(gray|slate|zinc|neutral)-(100|200|300)\b` on interactive elements (heuristic: any 1px border lighter than ~#949494 on white fails 3:1); review chart configs for hue-only series without `borderDash`/markers/patterns/direct labels and gridline colors lighter than ~#949494; grep `opacity-(50|60|70)|cursor-not-allowed` on interactive elements lacking `disabled`/`aria-disabled` + event suppression.
- **Runtime or browser test:** DevTools Rendering → emulate vision deficiencies (deuteranopia, achromatopsia, blurred vision): can you still identify the errored field, the input boundaries, and each chart series? Eyedropper luminance checks between series colors and between lines/grid and background (≥3:1 both). axe `color-contrast` flags the opacity-composited label. Attempt to activate the dimmed control — if it responds, it is not exempt. Screen reader: is the error announced (`role="alert"`, `aria-invalid="true"`)?
- **Corrective rule:** Error, success, and status indication MUST include a non-color cue (icon, visible message, shape, or weight/underline change; a ≥3:1 lightness difference is the weaker alternative), and form errors MUST additionally be programmatically associated (`aria-invalid` + `aria-describedby` → message) with error text itself ≥4.5:1 (`red-600`/`red-700`-class on white). Where a border or background edge identifies a component or its hit area, that edge MUST contrast ≥3:1 with the adjacent color. Chart series MUST be distinguishable without hue perception (direct labels, dash styles, markers, patterns, or ≥3:1 luminance separation), and every graphical object required for understanding MUST achieve ≥3:1. Loading/inactive states MUST be real — `disabled` or `aria-disabled="true"` with activation suppressed — and never reduce an active control's label below conforming contrast. (Focus-indicator contrast is covered in Section 5.3.)
- **Valid exceptions:** Color-only enhancement when an equivalent text/icon cue exists; components identified by other conforming means (adjacent label text); inactive components and UA-determined appearances; heat-map-type graphics where the gradient is the essential measurement representation (1.4.11 essential exception); chart information fully available in a conforming adjacent form (data table, visible labels + values); genuinely disabled controls dimmed by opacity (exempt as inactive).
- **Primary sources:** Understanding SC 1.4.1 (3:1 luminance route, F81, G111) — https://www.w3.org/WAI/WCAG22/Understanding/use-of-color.html ; Understanding SC 1.4.11 (boundaries, adjacent colors, pie-chart case study, essential exception) — https://www.w3.org/WAI/WCAG22/Understanding/non-text-contrast.html ; Understanding SC 1.4.3 (inactive-component exception scope) — https://www.w3.org/WAI/WCAG22/Understanding/contrast-minimum.html ; Deque axe `color-contrast` (opacity accounted) — https://dequeuniversity.com/rules/axe/4.10/color-contrast
- **Confidence:** High on the failure conditions (normative thresholds, deterministic measurement); severity of the boundary variant depends on whether the border is the sole identifier, as noted.

### 5.12 Content clipped or lost under zoom and text-spacing overrides; truncation with no accessible full text
- **Classification:** objectively broken | inaccessible
- **Severity:** High — normative AA failure of SC 1.4.4 Resize Text (failure F69 names exactly the clip-on-resize pattern) and SC 1.4.12 Text Spacing (failure F104); content is literally lost for low-vision users. The text-spacing variant alone is Medium since it triggers only under user-applied overrides; truncation without a reveal mechanism is an objective availability failure for all users.
- **Observable symptom:** At 200% browser zoom (or text-only zoom), card descriptions, button labels, and nav items are cut off mid-glyph with no scrollbar to reveal them; with the standard text-spacing override applied (line-height 1.5×, letter-spacing 0.12×, word-spacing 0.16×, paragraph spacing 2×), heading bottoms lose descenders or enlarged text overlaps the paragraph below; table cells and card titles end in "…" with no hover, focus, keyboard, or detail-view path to the full string.
- **Likely implementation cause:** Pixel-perfect card CSS: `h-[72px] overflow-hidden` on text blocks, `h-10` + `leading-none` buttons, fixed-height headings (`h-8 leading-8 overflow-hidden`), `line-clamp-2` on essential text, `tracking-tighter` on display headings whose containers cannot absorb 0.12em extra tracking, author `!important` on spacing properties, and defensive `truncate` / `text-ellipsis` with no `title`, tooltip, expandable row, or linked detail page. (Inference: generators optimize for fidelity to a single static frame; fixed heights are the cheapest way to align generated grids, and zoom/user-stylesheet conditions appear in no training screenshot.)
- **Why it matters:** SC 1.4.4 requires text resizable to 200% without loss of content or functionality; its Understanding doc lists F69 ("clipped, truncated or obscured" on resize) and the ACT rule "Zoomed text node is not clipped with CSS overflow," and permits truncation ONLY if the full content is available on focus/activation AND an indication of that availability is provided. SC 1.4.12 requires no loss of content or functionality under the four-property override; its Understanding doc states ellipsis caused by the override fails if no reveal mechanism exists. Absent a mechanism, this is objective content loss, not a style choice.
- **Static detection:** Grep `h-\[\d+px\]|h-\d+` combined with `overflow-hidden` on text-bearing elements; `line-clamp-`, `whitespace-nowrap` on informative content; `!important` on `line-height|letter-spacing|word-spacing|margin`; `tracking-tighter` on long strings; `truncate|text-ellipsis` without a co-located reveal mechanism (`title` alone is a weak, pointer-only fix). Heuristic: any `height` (not `min-height`) on a text container is a review flag.
- **Runtime or browser test:** Manual: browser zoom to 200% and text-only zoom; check every card/nav/modal for clipping. Apply the WCAG text-spacing override (the Understanding doc links the standard bookmarklet approach) and inspect every text block. Automatable in Playwright: compare `scrollHeight`/`scrollWidth` vs. client box at 200% zoom and with the four spacing properties injected; find elements where `scrollWidth > clientWidth` and assert a keyboard-reachable reveal affordance.
- **Corrective rule:** Text containers MUST grow with their content — `min-height` instead of `height`, no `overflow: hidden` on text-bearing boxes, no author `!important` blocking user spacing — and content MUST survive 200% zoom and the full SC 1.4.12 override set without clipping or overlap. Truncated content MUST be retrievable in full through a mechanism available to pointer AND keyboard users (expandable control, tooltip on focus+hover, detail page, or full text in the accessible name where appropriate), and the truncation MUST be visibly indicated.
- **Valid exceptions:** Captions and images of text are excepted from SC 1.4.4; scripts that do not use a given spacing property are excepted for that property; non-essential preview text fully available on the linked destination is explicitly allowed by the Understanding docs; user-resizable columns.
- **Primary sources:** Understanding SC 1.4.4 (F69, truncation conditions, ACT rules) — https://www.w3.org/WAI/WCAG22/Understanding/resize-text.html ; Understanding SC 1.4.12 (override metrics, F104, "Use of ellipses") — https://www.w3.org/WAI/WCAG22/Understanding/text-spacing.html
- **Confidence:** High as an availability defect; the WCAG failure for truncation is conditional on resize/spacing context as noted.

### 5.13 Undersized and crowded pointer targets
- **Classification:** inaccessible
- **Severity:** High — SC 2.5.8 Target Size (Minimum) (Level AA, new in WCAG 2.2); small adjacent targets cause mis-taps for touch users and users with tremors, and the worst offenders are high-frequency controls (close, delete, pagination) where a mis-tap activates a *different* action.
- **Observable symptom:** Modal close "×" buttons, toolbar icon buttons, and chip-remove buttons with 16–20px hit areas packed in `gap-1` rows; carousel indicator dots of 8–12px spaced 4px apart; pagination numerals with 16–20px targets shoulder-to-shoulder; table rows ending in 3–4 icon buttons (view/edit/delete) where the destructive action sits 4px from safe ones.
- **Likely implementation cause:** `<button class="p-0"><svg class="h-4 w-4"/></button>` — a 16px icon with no padding makes a 16×16 target; `onClick` attached to the `<svg>` itself makes the target exactly the icon box; carousel dots as `<button class="h-2 w-2 rounded-full">`; pagination links with `px-1 py-0.5`. Several 20px targets in a 4px-gap row fail even the spacing exception because their 24px-diameter circles intersect. (Inference: dense toolbars and minimal dot pagination are canonical admin-UI visuals, and target geometry is a runtime property invisible in a static mockup.)
- **Why it matters:** SC 2.5.8 requires targets ≥24×24 CSS px, or the spacing exception: 24px-diameter circles centered on each undersized target must not intersect any other target or circle. Zoom does not change CSS-px size — authors cannot claim zoom fixes it. The SC's explicit intent is to "reduce the likelihood of accidentally activating the wrong control," which adjacent destructive actions violate directly. Lighthouse's tap-target audit (48×48 px with a 25% overlap rule) is a useful stricter engineering bar beyond the WCAG floor.
- **Static detection:** AST/JSX heuristic: `<button>` containing only an `<svg>` with `h-4 w-4`/`h-5 w-5` and `p-0|p-1` → computed target < 24px; grep carousel-dot patterns (`rounded-full` + `h-[123]`) inside map-rendered buttons; pagination items with horizontal padding < 8px; action cells with >2 icon buttons at `p-0`/`p-1`. Rule of thumb: `p-0` + `w-4` = 16px target; fix with padding or `min-h-6 min-w-6` (technique C42).
- **Runtime or browser test:** axe-core `target-size` rule (axe ≥4.10; computes unobscured size and the circle-intersection check); Playwright `locator.boundingBox()` sweep over `button, a[href], [role=button]` asserting ≥24×24 or passing circle math; manual thumb test on a real phone: hit page "3" without hitting "2"/"4", and edit without hitting delete.
- **Corrective rule:** Every pointer target MUST be ≥24×24 CSS px (aim higher for primary controls) or satisfy the spacing exception with non-intersecting 24px circles; use padding/min-size to grow the hit area without growing the icon — the icon is the *visual*, the button is the *target*. Destructive actions SHOULD additionally be separated from safe actions by more than the minimum spacing and/or guarded.
- **Valid exceptions:** Inline targets within sentences; UA-determined sizes; an equivalent conforming control on the same page (a conforming Previous/Next pair can serve as the Equivalent exception for carousel dots); essential presentation (map pins); targets temporarily obscured by user-opened overlays.
- **Primary sources:** Understanding SC 2.5.8 (size/spacing math, adjacent-20px-button failure figure, C42, zoom independence, Equivalent exception) — https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html ; Deque axe `target-size` — https://dequeuniversity.com/rules/axe/4.10/target-size ; Lighthouse tap-targets audit (stricter bar) — https://developer.chrome.com/docs/lighthouse/seo/tap-targets
- **Confidence:** High.

### 5.14 Flashing content and vestibular-harmful motion with no reduced-motion path
- **Classification:** objectively broken | inaccessible
- **Severity:** Critical for flashing at ≥3 flashes/second — direct SC 2.3.1 Three Flashes or Below Threshold (Level A) failure, a documented physical-safety (seizure) hazard, and a Non-Interference violation that makes the whole page non-conformant; Critical for auto-playing motion with no pause/stop/hide mechanism when it moves >5s in parallel with other content (SC 2.2.2, Level A); High for ungated vestibular-trigger motion (SC 2.3.3 Animation from Interactions, AAA) ignoring an explicit OS-level user preference.
- **Observable symptom:** (a) A banner, badge, hero background, or "glitch" effect strobes several times per second via CSS keyframes, `steps()`, or JS interval; (b) full-bleed background video loops indefinitely with no pause control; demo GIFs, auto-advancing carousels, and infinite logo-ticker marquees run forever with no stop mechanism; (c) with OS "reduce motion" enabled, parallax layers, large-scale zoom/page-wipe transitions, and scroll-linked motion play identically to the default state.
- **Likely implementation cause:** `@keyframes` alternating high-contrast colors at ≤0.34s durations (`animation: flash 0.25s infinite`) or `setInterval` toggling classes at <350ms; the canonical `<video autoplay muted loop>` hero snippet with no `controls`, no custom pause button, and no reduced-motion gate; plain `<img src="*.gif">` (no playback control in the format); `@keyframes marquee … infinite` or `setInterval` slide advancement with no pause state; zero `@media (prefers-reduced-motion…)` blocks or `matchMedia` checks anywhere in the codebase. (Inference: "glitch" effects, background-video heroes, and logo marquees are template staples copied verbatim, and the reduced-motion variant is an invisible conditional state absent from virtually all training exemplars.)
- **Why it matters:** SC 2.3.1 permits nothing to flash more than 3 times in any 1-second period unless below the general/red flash thresholds (≈341×256 CSS px reference area) — flashing "cannot be allowed even for a second or it could cause a seizure. And turning the flash off is also not an option since the seizure could occur faster than most users could turn it off." SC 2.2.2 requires a pause/stop/hide mechanism for moving content that starts automatically, lasts >5s, and runs in parallel; auto-updating content gets no 5-second grace (F16 for scrolling content, G152 for GIFs). SC 2.3.3's Understanding doc names parallax scrolling explicitly and documents vestibular reactions up to "nausea, migraine headaches, and potentially needing bed rest to recover"; sufficient techniques are C39 (CSS media query) and SCR40 (JS `matchMedia`). MDN names "scaling or panning large objects" as vestibular triggers. Automation caveat: axe-core has NO rule for flash rate or reduced-motion handling — this cluster requires custom linting plus emulated-media tests.
- **Static detection:** Grep `@keyframes` toggling `background`/`color`/`opacity` between high-contrast extremes with `animation-duration` <0.34s and `infinite`; `setInterval` chains toggling visibility/color at <350ms; `<video` with `autoplay` + `loop` and no pause control or reduced-motion guard in the subtree; `<img>` srcs ending in `.gif`/animated `.webp`; `marquee|translateX(-50%)` keyframes with `infinite`; repo-wide check: if `prefers-reduced-motion`, `motion-safe:`, `motion-reduce:`, `useReducedMotion` all return zero matches while animation utilities return matches — flag the project (a cheap, near-certain signal per the brief).
- **Runtime or browser test:** DevTools Animations panel to read duration/iteration of running animations; PEAT for borderline video/GIF flash analysis. Load page and wait >5s: video/marquee/carousel still moving with no pause affordance = fail. Playwright: `page.emulateMedia({ reducedMotion: 'reduce' })`, reload, and assert auto-playing media is paused or replaced by poster/static fallback and known animated elements show `animation-name: none` or ~1ms durations with JS-driven transforms stopped. Manual OS-level reduce-motion pass for vestibular sign-off.
- **Corrective rule:** Content MUST NOT flash more than 3 times in any 1-second period unless provably below the general and red flash thresholds. Any auto-playing moving content persisting beyond 5 seconds in parallel with other content MUST provide a keyboard-accessible pause/stop/hide control; animated images MUST stop within 5 seconds or be served only under `(prefers-reduced-motion: no-preference)` with a static fallback. Every project shipping non-essential animation MUST implement reduced-motion handling at both the CSS layer (C39: media-query guards or `motion-safe:`/`motion-reduce:` variants) and the JS layer (SCR40: `matchMedia` with change listener or library equivalents) — a CSS-only override in a JS-animation codebase MUST be treated as non-compliant. Parallax, large-zoom, and full-screen wipe transitions MUST be replaced by instant cuts or opacity fades under `reduce`.
- **Valid exceptions:** ≤3 flashes in any one second passes automatically; flashing area below the ~341×256 CSS px threshold passes; movement essential to the information (a genuinely instructional loop, real-time tickers — which still need the pause mechanism); content not presented in parallel with other content; animation whose removal would fundamentally change the functionality (WCAG "essential"). Sites targeting AAA: SC 2.3.2 removes even the threshold exception for flashing.
- **Primary sources:** Understanding SC 2.3.1 (thresholds, reference area, PEAT) — https://www.w3.org/WAI/WCAG22/Understanding/three-flashes-or-below-threshold.html ; Understanding SC 2.2.2 (5s rule, F16, G152, auto-updating note) — https://www.w3.org/WAI/WCAG22/Understanding/pause-stop-hide.html ; Understanding SC 2.3.3 (vestibular documentation, C39, SCR40) — https://www.w3.org/WAI/WCAG22/Understanding/animation-from-interactions.html ; MDN `prefers-reduced-motion` — https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion ; web.dev reduced-motion override pattern — https://web.dev/articles/prefers-reduced-motion ; Playwright media emulation — https://playwright.dev/docs/api/class-page#page-emulate-media
- **Confidence:** High — objectively dangerous or normatively defined and measurable. Cross-reference: library-level implementation failures behind these symptoms (Motion/Framer `reducedMotion` defaulting to `"never"`, ungated GSAP/ScrollTrigger, unguarded smooth scrolling, broken `animation: none` overrides) are detailed in Section 3 (Motion Failures).

## 6. Frontend Implementation Failures

The failures in this section are engineering defects: most are invisible in a screenshot and many pass a happy-path manual test, yet they measurably degrade load performance, break under routine content or environment change, or ship objectively wrong behavior. An observed pattern across the research briefs (inference, not sourced fact): AI generators optimize for the statically rendered result and the shortest path to "it runs," so attributes, boundaries, and cleanup code that carry no visual weight — image dimensions, `font-display`, server/client boundaries, effect teardown, key stability — are systematically under-produced, while tutorial-era idioms (`'use client'` at the root, `useEffect` for everything, `import _ from 'lodash'`, `z-index: 9999`) are reproduced at training-corpus frequency. Because these defects are non-visual, they also survive the screenshot-based feedback loops through which generated UI is typically iterated (inference). Each issue below is written as a review standard: checkable statically (lint/grep/AST) and confirmable at runtime (Lighthouse, DevTools, Playwright, axe-core). Adjacent failure modes whose primary impact is interaction, motion, responsive behavior, or accessibility are catalogued in Sections 2–5 — including alt-text and icon-font failures, decorative-element overlap, `overflow` clipping of focus indicators, Radix `asChild` contract breakage, and reduced-motion handling; this section covers the engineering root causes where they are most at home.

### 6.1 Layout shift during load: unsized media, late-injected UI, and uncontrolled font swap
- **Classification:** objectively broken | inefficient
- **Severity:** High
- **Observable symptom:** Text and buttons jump as images or video finish loading; a cookie consent or announcement bar pushes the whole page down 40–80 px one to several seconds after paint; users mid-click hit the wrong element. Lighthouse reports CLS above 0.1 (poor above 0.25) and fails the `unsized-images` and `font-display` audits; DevTools Performance shows red Layout Shift bars. On slow networks text stays invisible up to ~3 s (FOIT) or visibly re-wraps when the web font swaps in.
- **Likely implementation cause:** `<img>`/`<video>` without `width`/`height` or CSS `aspect-ratio` (or Next.js `<Image fill>` inside a parent with no defined size, so the parent collapses); `useEffect(() => setShowBanner(true), [])` inserting a new in-flow element above page content without reserving space, or third-party scripts that `document.body.prepend(...)`; `@font-face` with no `font-display`, no `size-adjust`/`ascent-override`/`descent-override`/`line-gap-override` on the fallback face, and no preload for critical fonts.
- **Why it matters:** CLS is a Core Web Vital; unexpected movement makes users lose reading position and click the wrong targets (web.dev's canonical example: confirming instead of canceling an action). Only shifts within 500 ms of a user interaction count as expected, so a banner appearing on a timer is always an unexpected shift, and a banner that entirely covers a focused component additionally fails WCAG SC 2.4.11 (its Understanding document uses exactly this cookie-banner example, with technique C43 and failure F110). Web fonts are a documented common CLS cause, and `size-adjust` exists specifically to metric-match fallbacks.
- **Static detection:** Grep `<img`/`<video` lacking both `width`+`height` and an `aspect-ratio` rule; ESLint `@next/next/no-img-element` in Next.js projects; grep `useEffect`/`onMounted`/`setTimeout` blocks that toggle banner/announcement visibility into the document flow and check the slot reserves `min-height` from first paint or is an overlay; grep `@font-face` blocks missing `font-display`. Lighthouse CI audits `unsized-images` and `font-display` automate most of this.
- **Runtime or browser test:** Lighthouse (mobile, throttled) for CLS — plus timespan/user-flow mode, because the default load-only audit misses post-load shifts (lab CLS ≠ field CLS per web.dev); DevTools Performance → Layout Shifts track while idling 5–10 s; Slow 3G filmstrip for font visibility and swap reflow; Playwright: measure a sentinel element's `getBoundingClientRect().top` before/after asset load.
- **Corrective rule:** Every rendered content image or video MUST reserve its space before the resource loads (`width`/`height` attributes, CSS `aspect-ratio`, or a sized container for `fill`-style images). UI injected after first paint MUST NOT displace existing visible content — reserve the slot from initial render or render as a non-displacing overlay that never fully obscures the focused component (SC 2.4.11; technique C43). Every `@font-face` MUST declare an explicit `font-display` (`optional` for maximum stability, `swap` when the web font is required); `swap` fallbacks SHOULD be metric-matched via `size-adjust` and friends, and critical fonts SHOULD be preloaded.
- **Valid exceptions:** Shifts within 500 ms of a qualifying user interaction; user-initiated disclosure (accordions); CSS background images; SVGs with intrinsic dimensions; icon fonts, where `swap` is inappropriate (use `block`, or better, SVG).
- **Primary sources:** https://web.dev/articles/cls ; https://web.dev/articles/optimize-cls ; https://web.dev/articles/font-best-practices ; https://developer.mozilla.org/en-US/docs/Web/CSS/@font-face/size-adjust ; https://nextjs.org/docs/app/api-reference/components/image ; https://www.w3.org/WAI/WCAG22/Understanding/focus-not-obscured-minimum.html (SC 2.4.11; C43; F110)
- **Confidence:** High

### 6.2 Broken and inefficient asset delivery: unoptimized images, mis-prioritized lazy loading, case-mismatched paths, oversized base64 inlining
- **Classification:** inefficient | objectively broken
- **Severity:** High
- **Observable symptom:** Multi-MB hero/content images (2–8 MB transfer on initial load; Lighthouse LCP > 2.5 s; `uses-responsive-images`/`uses-optimized-images` audits fail); dozens of below-fold image requests before first scroll; the hero pops in late with the `lcp-lazy-loaded` audit firing; images render on the developer's Windows/macOS machine but 404 after deployment to a Linux host; the main JS bundle balloons with inlined base64 blobs that re-download on every deploy.
- **Likely implementation cause:** Design-export originals (3000–5000 px PNG/JPEG) referenced directly with no `srcset`/`sizes`, no WebP/AVIF, or raw `<img>`/`unoptimized` bypassing `next/image` — and missing `sizes`, so the browser assumes `100vw` and downloads the largest candidate (Next.js documents this explicitly); blanket `loading="lazy"` applied to every image including the LCP hero (web.dev's A/B test showed up to ~18% LCP regression from lazy-loading in-viewport images), or global `loading="eager"` defaults; filename casing drift (`Logo.PNG` on disk referenced as `logo.png`) that case-insensitive dev filesystems forgive and case-sensitive web servers do not; hand-pasted `data:image/...;base64,` strings, bundler inline limits (`dataUrlCondition`, `assetsInlineLimit`) raised to cover large files, or a full-size base64 image used as `blurDataURL`.
- **Why it matters:** Image weight is the dominant LCP and bandwidth driver (MDN's lazy-loading guide cites medians near ~900 KB desktop / ~850 KB mobile per page as the problem lazy loading exists to fight); every extra megabyte is real cost on metered connections. Case mismatches are deterministic 404s on the server and undetectable on the dev machine (MDN: most web servers are case-sensitive). Base64 inflates bytes by ~33%, cannot be cached independently of the bundle, and blocks JS parsing — webpack's own automatic inlining applies only below an 8 kB default threshold, and Next.js warns a large `blurDataURL` "may hurt performance."
- **Static detection:** Scan the assets directory for rasters > ~300 KB; grep raw `<img>` without `srcset` (`@next/next/no-img-element` automates this in Next.js); assert the LCP candidate has no `loading="lazy"` and carries `priority`/`fetchpriority="high"`/`preload`; CI check resolving every asset reference and import specifier against the git tree with case-sensitive matching (CaseSensitivePathsPlugin-style) plus a lowercase-hyphen naming lint; grep `data:image`/`data:font` and flag matches longer than a few KB (base64 blobs are single enormous lines); review bundler inline-limit config overrides.
- **Runtime or browser test:** Lighthouse (`uses-responsive-images`, `offscreen-images`, `lcp-lazy-loaded`, `total-byte-weight`, "Avoid enormous network payloads"); DevTools Network → Img filter with no scrolling (any below-fold asset loading at t=0 is a hit); build and serve the production bundle inside a Linux container and crawl for 404s (Playwright request-failed listener); bundle analyzer attributing size to inline assets.
- **Corrective rule:** Raster content images MUST be served at or near rendered size via responsive `srcset`+`sizes` or a framework optimizer, and MUST use modern compressed formats (WebP/AVIF) where supported. Below-fold images and iframes SHOULD use native `loading="lazy"`; the LCP candidate MUST load eagerly, be prioritized (`fetchpriority="high"`, framework `priority`/`preload`), and MUST NOT be lazy-loaded. Every asset reference and import specifier MUST match on-disk casing byte-for-byte, with lowercase-hyphen filenames enforced mechanically. Data-URL inlining MUST be restricted to tiny assets (on the order of the bundler's ≤ 8 kB default); larger assets MUST be emitted as separate fingerprinted files, and blur/LQIP placeholders MUST stay tiny (Next.js recommends a ≤ 10 px source image).
- **Valid exceptions:** Small logos/icons (SVG preferred); deliberate full-quality downloads behind a user action; carousels/masonry where "below fold" is not knowable server-side; "heroes" actually below the fold at the target viewport; single-file deliverables (HTML email, embedded widgets) that accept the size cost deliberately.
- **Primary sources:** https://nextjs.org/docs/app/api-reference/components/image ; https://developer.mozilla.org/en-US/docs/Web/Performance/Guides/Lazy_loading ; https://web.dev/articles/lcp-lazy-loading ; https://developer.mozilla.org/en-US/docs/Learn_web_development/Getting_started/Environment_setup/Dealing_with_files ; https://webpack.js.org/guides/asset-modules/
- **Confidence:** High

### 6.3 Dependency and import mismanagement: whole-library and CommonJS imports, legacy/duplicate/outdated/dead dependencies
- **Classification:** inefficient (security risk when a pinned version matches a known CVE)
- **Severity:** High
- **Observable symptom:** Bundle analyzer shows the entire lodash (~70 KB min+gz) or full icon sets in the production bundle for a handful of used exports; CommonJS interop ships whole libraries (web.dev's example: 625 KB with CJS vs 40 bytes with ESM); `moment` plus its full locale set (~160 minified KB in v2.18.1) appears in new code; two date libraries, two animation libraries, or two HTTP clients coexist; a full component kit (MUI, Ant Design, Chakra, Bootstrap) is installed to render one modal and its global resets fight the project's styles; `npx depcheck` reports unused packages; `npm audit` flags hard-pinned stale versions.
- **Likely implementation cause:** Namespace/default imports of re-export-everything entry points without `sideEffects: false` + ESM hygiene (`import _ from 'lodash'` for one `_.debounce`); `require()` or CJS-only packages that defeat static analysis (webpack: tree-shaking "relies on the static structure of ES2015 module syntax"); tutorial-era installs (`moment`, jQuery) reproduced at training-corpus frequency (inference); per-prompt local optimization with no project-wide dependency memory, so each file imports whichever library is statistically prominent for that prompt (inference); install commands generated independently of any importing code (inference); exact old versions copied from dated tutorials; heavy animation libraries statically imported into the route entry chunk for one below-fold effect.
- **Why it matters:** Every unused or duplicated byte is parsed and compiled on the user's main thread at startup — web.dev ties startup JS volume directly to worse INP and, for client-rendered markup, worse LCP. Moment's own maintainers declare it "a legacy project in maintenance mode," discourage new use, and document its tree-shaking failure, mutability bugs, and the all-locales webpack pitfall. Stale pins accumulate unpatched vulnerabilities and block transitive deduplication; unused packages enlarge the audit/supply-chain surface and mislead reviewers about what the app actually does.
- **Static detection:** ESLint `no-restricted-imports` (`lodash` → `lodash-es/<method>`; ban `moment`); grep `require(` in client source; a dependency-capability matrix CI check that fails when ≥ 2 libraries serve one capability (dates, animation, icons, state); a kit-usage tally (`grep "from '@mui/material'"` + AST import-specifier count) gated on "≥ N components used or an ADR exists"; `npx depcheck` / `npx knip`; `npm audit` (fail on high/critical) and `npm outdated` in CI; bundle-analyzer diff on every PR; check `sideEffects` in the project's and key dependencies' package.json; grep static imports of `gsap`/`framer-motion`/`three`/`lottie-web` outside lazy/dynamic boundaries.
- **Runtime or browser test:** webpack-bundle-analyzer / source-map-explorer / vite-bundle-visualizer on production builds; Lighthouse "Reduce unused JavaScript" and `no-vulnerable-libraries` audits; DevTools Coverage tab for unused vendor/animation bundles at load.
- **Corrective rule:** Client code MUST use ESM imports and MUST import only the used modules (`import debounce from 'lodash-es/debounce'` or named imports from a `sideEffects: false` package); namespace imports of utility/icon libraries are forbidden in production code. New code MUST NOT add `moment` (use `Intl.DateTimeFormat`, date-fns, Day.js, or Luxon). Each capability MUST be served by exactly one dependency; a component kit MUST NOT be added for a single component. Every entry in `dependencies` MUST be imported by shipped code. Versions MUST be current stable with semver ranges unless a specific pin is justified by a documented constraint, and `npm audit` high/critical findings MUST block merge. Large animation libraries MUST NOT be statically imported into the entry chunk for below-fold or post-interaction effects (use dynamic `import()` or per-feature imports).
- **Valid exceptions:** Deliberate migration windows; unavoidable transitive duplicates (not imported directly); kits adopted deliberately project-wide or genuinely required complex machinery (data grids); legacy codebases already dependent on Moment (the maintainers' own status page allows this); pins documented against a specific regression; tooling-only packages consumed by config (postcss plugins, types, `server-only` markers); lockfile pinning — the defect is stale declaration, not the lockfile itself.
- **Primary sources:** https://webpack.js.org/guides/tree-shaking/ ; https://web.dev/articles/commonjs-larger-bundles ; https://web.dev/articles/reduce-javascript-payloads-with-code-splitting ; https://momentjs.com/docs/#/-project-status/
- **Confidence:** High

### 6.4 Server-only code imported into the client bundle
- **Classification:** objectively broken (security defect class when secrets ship)
- **Severity:** Critical
- **Observable symptom:** API keys or database connection strings visible in DevTools Sources or network payloads; client-side fetch failures with `undefined` credentials; `fs is not defined` errors in the browser console; non-`NEXT_PUBLIC_` environment variables silently replaced with empty strings.
- **Likely implementation cause:** Data-fetching utilities colocated with UI in one file that is later marked `'use client'` for interactivity, transitively pulling server modules into the client graph — the "environment poisoning" pattern Next.js documents — with no `server-only` guard anywhere.
- **Why it matters:** Next.js documents that any module reachable from a `'use client'` entry joins the client bundle, so literal secrets, DB clients, or `fs` code can ship to the browser; the `server-only` package exists to turn this into a build-time error. This is the only security-class defect in the dependency cluster, and it fails silently (empty-string env vars) rather than loudly.
- **Static detection:** `import 'server-only'` on every server-only module (build-time guard per Next.js docs); lint rule banning `process.env.<NON_PUBLIC>` references in files under a `'use client'` boundary; dependency-graph check (e.g., madge) from client entries for `pg`, `prisma`, `fs`, `node:*` imports.
- **Runtime or browser test:** DevTools: search shipped client chunks for known secret names/DB strings; browser console for `fs`/`process` errors; verify data arrives via Server Components or route handlers.

### 6.5 Imperative DOM access and mutation of framework-managed trees
- **Classification:** objectively broken
- **Severity:** High
- **Observable symptom:** Intermittent crashes after state updates ("Failed to execute 'removeChild'"); UI that loses event handlers or visually reverts after re-render; ghost or duplicated elements; features that silently die when someone adds a wrapper element or reorders markup.
- **Likely implementation cause:** jQuery snippets (`$('.sidebar').toggle()`) or `document.querySelector(...).remove()` writing into subtrees React renders, typically in a `useEffect` with no cleanup; behavior bound to deep positional selectors (`.card > div:nth-child(2) span`), `el.parentElement.children[3]`, `firstChild` (which matches text nodes), or index-based sibling math.
- **Why it matters:** React's reconciliation assumes exclusive ownership of the nodes it rendered; React's own documentation states that modifying, adding children to, or removing children from React-managed nodes "can lead to inconsistent visual results or crashes," and calls `querySelectorAll` traversal "brittle and [it] can break if your DOM structure changes." This is a correctness defect, not a style issue.
- **Static detection:** Grep `$(`, `jQuery`, `document.querySelector(All)?` in `src/**/*.{jsx,tsx}` (allowlist ref-scoped `ref.current` usage); AST flags for combinators/`nth-` in selector strings and for `.parentElement`, `.children[`, `.firstChild`, `.nextSibling` in application code; dependency gate rejecting `jquery` in React projects.
- **Runtime or browser test:** Playwright: perform interactions that toggle state, assert zero `pageerror` events and expected testids; component test that inserts an extra wrapper element and asserts behavior is unchanged; React Strict Mode dev run to surface effects running twice.
- **Corrective rule:** Code MUST NOT mutate DOM nodes managed by a framework; visible changes MUST go through state/props; elements MUST be located via refs, stable IDs, or dedicated `data-*` hooks — never positional or deep structural selectors; refs are reserved for non-destructive actions (focus, scroll, measure) or for subtrees the framework never updates.
- **Valid exceptions:** Integrating a third-party non-React widget (chart, map) inside a ref-owned container React renders empty and never updates — explicitly sanctioned by the React docs; portals; tests and dev tooling.

- **Confidence:** High

### 6.6 Mismanaged server/client boundary: root-level `'use client'`, spinner-first client fetching, non-deterministic render, SSR-unsafe globals
- **Classification:** inefficient | objectively broken
- **Severity:** Critical (hydration mis-wiring, SSR/build crashes); High for the performance-only manifestations
- **Observable symptom:** View-source shows a near-empty HTML shell with skeletons/spinners instead of content; large `/_next/static/chunks` download; the page is inert until hydration finishes; console reports "Hydration failed because the initial UI does not match what was rendered on the server" or the Next.js hydration-error overlay appears; `next build`/SSR dies with `window is not defined`; content flickers and swaps right after hydration; `isClient`-gated sections pop in late; the LCP element's timestamp tracks the JS bundle rather than the HTML.
- **Likely implementation cause:** `'use client'` at the top of `app/layout.tsx` or a route `page.tsx` "to make hooks work everywhere" (Next.js: all imports of a client file join the client bundle, so one root directive pulls the whole tree into the client graph); pages written as Client Components fetching in `useEffect` instead of `async` Server Components (React docs: "Effects don't run on the server… the initial server-rendered HTML will only include a loading state with no data"); runtime CSS-in-JS forcing `'use client'` boundaries on otherwise static sections; `Date.now()` / `Math.random()` / `localStorage` / `window` / `typeof window !== 'undefined'` branches read during render; hydration errors "fixed" with whole-tree `isClient` gates or scattered `suppressHydrationWarning` instead of deterministic output.
- **Why it matters:** Server Components exist to reduce client JavaScript and improve FCP; hydration is expensive — the SSR'd page "may look loaded… but cannot respond to input until the client-side scripts for components are executed" (web.dev), and main-thread JS execution is the primary driver of INP (> 500 ms is poor). `hydrateRoot` expects the first client render to be identical to the server HTML and React treats mismatches as bugs that can attach "event handlers… to the wrong elements." `localStorage` does not exist server-side and can throw `SecurityError` even in browsers. Gates and suppression reintroduce double-render and jarring post-hydration swaps; React's docs: `suppressHydrationWarning` "only works one level deep, and is intended to be an escape hatch. Don't overuse it."
- **Static detection:** Grep leading `'use client'` lines in `app/**/layout.tsx` and `app/**/page.tsx`; route pages containing `useEffect` + `fetch(`/data hooks instead of `async` server functions; grep render scope (component body, JSX, `useState(initial)`) for `Date.now(`, `new Date(`, `Math.random(`, `crypto.randomUUID(`, `window.`, `document.`, `localStorage`, `sessionStorage`, `navigator.`, `typeof window`; count `suppressHydrationWarning` occurrences (≈ 0–1 per app); grep `isClient`/`mounted`/`hydrated` flags gating JSX returns; AST scan for styled/emotion templates with no prop interpolations that nonetheless force client boundaries.
- **Runtime or browser test:** `curl` the route — is substantive content in the served HTML? Disable JavaScript and reload. Lighthouse FCP/LCP/TBT; Playwright console listener asserting zero `/hydrat/i` errors on load; `next build && next start` smoke test of every route; diff view-source HTML against the post-hydration DOM; CPU/network-throttled film of placeholder→content swaps.
- **Corrective rule:** Components MUST remain Server Components by default; `'use client'` belongs on the specific interactive leaves that need state, event handlers, lifecycle logic, or browser-only APIs — move the client boundary down the tree. Initial page data MUST be fetched on the server (async Server Components / framework data APIs); client Effects are for interaction-driven data. The first client render MUST produce output identical to the server render: defer non-deterministic values to Effects, isolate client-only widgets behind `dynamic(..., { ssr: false })`, and scope `suppressHydrationWarning` to single unavoidable leaves. Browser globals MUST be accessed only inside effects, event handlers, or explicit client-only modules, and `localStorage` access MUST guard existence and `SecurityError`. Styles that do not depend on runtime props MUST be static CSS (stylesheet, CSS module, utility classes), not runtime CSS-in-JS. Content that can be the LCP element MUST be present in the server-delivered HTML and discoverable from it.
- **Valid exceptions:** Genuinely fully-interactive routes (a canvas editor page); single-element timestamps with `suppressHydrationWarning`; client-only third-party widgets via `dynamic({ ssr: false })`; dashboards deliberately built as client apps with a cache/dedup layer; guarded top-level app bootstrap outside render; `NEXT_PUBLIC_` variables.
- **Primary sources:** https://nextjs.org/docs/app/getting-started/server-and-client-components ; https://web.dev/articles/rendering-on-the-web ; https://web.dev/articles/inp ; https://web.dev/articles/lcp ; https://nextjs.org/docs/messages/react-hydration-error ; https://react.dev/reference/react-dom/client/hydrateRoot ; https://developer.mozilla.org/en-US/docs/Web/API/Window/localStorage ; https://react.dev/learn/synchronizing-with-effects
- **Confidence:** High

### 6.7 `useEffect` as a universal hammer: derived state, event handling, and unguarded async work
- **Classification:** objectively broken | inefficient
- **Severity:** Critical (stale-response races display objectively wrong data); High (phantom/duplicate user actions); Medium (derived-state churn)
- **Observable symptom:** Typing quickly in a search box ends with results for an older query; an "Added to cart" toast reappears after page reload; a submit POST fires twice under Strict Mode or again on Back navigation; values update one frame late (two render passes per interaction in the Profiler); cascading Effect chains produce a visible staircase of intermediate states.
- **Likely implementation cause:** `useEffect(() => setX(compute(a, b)), [a, b])` for render-computable values; `useState(props.x)` treated as a live binding (it is only an initial value) and patched with a sync Effect; flag-in-state + Effect performing the actual action (`post`, `toast`, `router.push`) instead of the event handler; chains of Effects setting state to trigger each other; `useEffect(() => { fetch(...).then(json => set(json)) }, [q])` with no `ignore` flag, no `AbortController`, no cleanup; `isMounted` refs used to suppress rather than fix.
- **Why it matters:** React's documentation labels these patterns "🔴 Avoid": values calculable from props/state belong in render, not state; by the time an Effect runs "you don't know what the user did," so event logic re-fires on mount, remount, and refresh-with-persisted-state; unguarded fetches race (the docs' own example: `"hell"` may arrive after `"hello"` and overwrite it) and the last write wins — objectively wrong data on screen; subscriptions without cleanup pile up across navigation; Strict Mode's double-mount exists to surface exactly these bugs.
- **Static detection:** Enforce `react-hooks/exhaustive-deps`; AST/regex heuristics: Effect bodies consisting only of `setX(...)` calls over props/state; `useState(props` / `useState({ ...props` patterns; Effects containing `fetch(`/`.then(set` with no returned cleanup; Effects that read handler-only state and then perform external actions; a component-local Effect dependency graph with setState-to-Effect paths of length ≥ 2; `useRef` "did it run" guards suppressing Effects.
- **Runtime or browser test:** React DevTools Profiler: one interaction producing two back-to-back renders, or a staircase of renders for one action; Strict Mode dev run counting network requests/toasts for a single action; DevTools Slow 3G + rapid query changes asserting the final rendered list matches the last query; Playwright with delayed stubbed responses arriving out of order; refresh with persisted state watching for phantom notifications.
- **Corrective rule:** Any value derivable from props/state MUST be computed during render (plain expression; `useMemo` only when verifiably expensive); logic caused by a specific user interaction MUST run in the corresponding event handler; every Effect that initiates async work (fetch, subscription, connection, timer) MUST return a cleanup that cancels, aborts (`AbortController`), or ignores stale results; state that must reset when an identity prop changes MUST use `key`, not a sync Effect; framework/library data fetching (Server Components, route handlers, TanStack Query/SWR-style caches) SHOULD replace ad-hoc Effect fetches.
- **Valid exceptions:** Genuine synchronization with external systems (subscriptions, dependent dropdowns fetching per selection, non-React widget setup, page-view analytics); draft state intentionally forked from a prop; the documented "adjust state during render" prev-comparison pattern; fire-and-forget beacons never written to state.
- **Primary sources:** https://react.dev/learn/you-might-not-need-an-effect ; https://react.dev/learn/synchronizing-with-effects ; https://react.dev/reference/eslint-plugin-react-hooks/lints/exhaustive-deps ; https://react.dev/learn/thinking-in-react ; https://react.dev/learn/choosing-the-state-structure ; https://react.dev/learn/preserving-and-resetting-state
- **Confidence:** High

### 6.8 Unstable or missing list keys
- **Classification:** objectively broken
- **Severity:** Critical when items are stateful or actions are row-targeted; Medium otherwise
- **Observable symptom:** After sorting, filtering, deleting, or inserting, a row shows another row's draft text, checked state, or open menu; focus jumps to the wrong input; a delete/ban/transfer action applies to the wrong record; remount storms on every reorder (entrance animations replay, `<video>` playback resets); React warns "Each child in a list should have a unique key prop."
- **Likely implementation cause:** `items.map((item, index) => <Row key={index} />)` on lists that can be reordered, filtered, inserted into, or that contain stateful children; `key={Math.random()}` / `key={crypto.randomUUID()}` / `key={uuid()}` computed during render; missing keys (React's index fallback).
- **Why it matters:** React associates state with position + key in the render tree. React docs: index keys "often lead to subtle and confusing bugs" once order changes; render-time random keys "cause keys to never match up between renders, leading to all your components and DOM being recreated every time… it will also lose any user input inside the list items." Worst case this is a data-integrity bug — the UI presents record B with record A's interactive state, and a destructive action lands on the wrong entity — plus a performance bug (full unmount/remount instead of move/update).
- **Static detection:** ESLint `react/jsx-key` enforced as error; grep `key={index}`, `key={i}`, `key={idx}` (variable from `map`'s second parameter), `key={Math.random()`, `key={crypto.randomUUID()`, `key={uuid()}` inside JSX (the community rule `react/no-array-index-key` — flagged as non-primary — automates part of this).
- **Runtime or browser test:** Type distinct text into inputs of rows 1 and 2, then reverse sort or delete row 1: text must follow its record, not its position. Playwright: fill an input in row N, click sort, assert the value follows the same record's `data-id`. React DevTools Profiler: reordered items must commit as updates, not mounts.
- **Corrective rule:** Keys MUST be stable, unique-among-siblings identifiers derived from the data (database ID, or an ID generated once at item creation time); index keys are acceptable only for lists that are never reordered, inserted, deleted, or per-item stateful; keys MUST never be generated during render.
- **Valid exceptions:** Truly static lists (fixed footer links, hard-coded steps). `key={Math.random()}`: none.
- **Primary sources:** https://react.dev/learn/rendering-lists ; https://react.dev/learn/preserving-and-resetting-state
- **Confidence:** High

### 6.9 Brittle CSS architecture: `!important` chains, specificity wars, global element selectors, and theming bypass
- **Classification:** inefficient (maintainability) | inaccessible (when hard-coded colors create untested theme pairs that fail contrast)
- **Severity:** Medium; High where dark mode or user color preferences are supported (WCAG SC 1.4.3 exposure)
- **Observable symptom:** Style changes "don't take"; DevTools shows chains of struck-through `!important` declarations fighting each other; selectors like `#root > div > div:nth-child(2) > section > div.card p`; a plain `<p>` or `<span>` inside a component inherits unexpected margins or colors that vary by route; dark mode leaves some panels light; themed (now light) text renders on a hard-coded light background and becomes unreadable; hover/focus states missing on inline-styled elements.
- **Likely implementation cause:** Cascade conflicts resolved by appending `!important`, concatenating ancestors/IDs, or duplicating classes (`.card.card`) instead of fixing layers, order, or specificity; global bare-element rules (`p { margin-bottom: 16px; }`) added to patch one page but matching inside every component; static `style={{ background: '#0f172a', padding: 24 }}` and raw `#hex`/`rgb()` literals duplicating tokens instead of semantic custom properties or theme utilities.
- **Why it matters:** MDN calls `!important`-driven overriding "a bad practice": author-important declarations reverse cascade origin order, blocking themes, user styles, and future fixes; inline styles "always overwrite any normal styles in author stylesheets," can only be beaten by `!important`, and cannot express pseudo-classes (`:hover`, `:focus-visible`) or media queries at all. Long selectors hard-couple CSS to DOM depth, so benign markup changes silently un-apply styles. A hard-coded background combined with a themed foreground creates a pair nobody specified or tested — WCAG SC 1.4.3 requires 4.5:1 (3:1 for large text) and its documented failure F24 covers exactly this unknown-pair situation.
- **Static detection:** stylelint `declaration-no-important`, `selector-max-specificity` (target ≤ 0-2-0), `selector-max-compound-selectors` (≤ 3), `selector-max-id: 0`, `selector-max-type` (0 outside reset/typography scopes), `color-no-hex` and `declaration-strict-value` for color properties; grep `!important` (track count over CI time), the duplicated-class pattern `\.([a-z0-9-]+)\.\1`, `style={{` with color/px literals, and `#[0-9a-fA-F]{3,8}\b|rgb\(` outside token files.
- **Runtime or browser test:** Theming probe: override a component color via a custom property or cascade layer — if only another `!important` can win, an escalation chain exists; Playwright `page.emulateMedia({ colorScheme: 'dark' })` and `'light'` with axe-core `color-contrast` under both schemes; visual regression of the same component across routes (computed-style diff).
- **Corrective rule:** New `!important` declarations introduced to win cascade conflicts MUST be rejected (resolve via `@layer`, source order, or specificity reduction; any retained exception carries an inline justification comment). Rule specificity SHOULD stay flat and IDs MUST NOT be used as styling hooks. Global stylesheets MUST NOT style bare elements outside a documented reset/typography layer — scope via a scope class, `@scope`, CSS Modules, or an ordered `@layer base`. Inline styles MUST NOT set static color, spacing, or typography; runtime-dynamic values MUST pass through CSS custom properties (`style={{ '--progress': pct }}`). All color values MUST come from the token/custom-property layer, and every themed component MUST be contrast-verified in each supported color scheme.
- **Valid exceptions:** Overriding un-editable third-party CSS (prefer importing it into a lower cascade layer); defensive utilities such as `.sr-only`; documented resets/normalize (Tailwind Preflight); genuinely runtime-computed geometry in `style`; logotypes, documented data-visualization palettes, and third-party brand assets (explicit 1.4.3 exceptions).
- **Primary sources:** https://developer.mozilla.org/en-US/docs/Web/CSS/important ; https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_cascade/Specificity ; https://www.w3.org/WAI/WCAG22/Understanding/contrast-minimum.html (SC 1.4.3; F24) ; https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-color-scheme ; https://tailwindcss.com/docs/dark-mode
- **Confidence:** Medium overall (High for the contrast failure mode when theming is in scope)

### 6.10 Fixed pixel heights and magic offsets on text-bearing layout
- **Classification:** inaccessible | objectively broken
- **Severity:** High
- **Observable symptom:** At 200% text zoom, or when a user stylesheet applies WCAG 1.4.12 spacing metrics, text is cut off mid-glyph, buttons show half a label, headings overlap the following paragraph; top-of-page content slides under the fixed header when the nav wraps or a banner appears; in-page anchor links land with the target heading hidden behind the sticky header. Also breaks on translation to longer languages and on font-fallback metric changes.
- **Likely implementation cause:** `height: 56px; overflow: hidden` on cards, buttons, headings, and list rows, measured from the mockup's current content; `main { padding-top: 72px; }` duplicating the header's current pixel height as a second, un-synced source of truth; anchor targets given no `scroll-margin-top`.

WCAG 1.4.10 Understanding explicitly warns that sticky/fixed content can obscure content at zoomed viewports (advisory technique C34: un-fix sticky headers), and MDN's `position` reference carries the same accessibility note. Fixed heights assume content metrics never change; they always do. (See Section 4 for the broader reflow/320 px failures this pattern contributes to.)
- **Static detection:** Grep `height:\s*[0-9]+px` in styles of text-bearing components and flag `height` + `overflow:\s*hidden` pairs on the same rule (a stylelint custom rule can force `min-height` on text containers); compare `padding-top`/`margin-top`/`top` literals against the header's height value (a duplicated literal in two files = defect); check anchor-target components for `scroll-margin-top`.

- **Corrective rule:** Text containers MUST size from content (`min-height`, padding, intrinsic sizing) and MUST NOT combine fixed height with `overflow: hidden`; where truncation is used it MUST use ellipsis with the full content available on focus/activation or a linked page (the explicit 1.4.12 ellipses condition). Header clearance MUST be derived, not duplicated (header in normal flow or a grid row, or `position: sticky` within flow); anchor targets MUST use `scroll-margin-top` sized from a shared token; sticky/fixed headers SHOULD un-fix at small or zoomed viewports (technique C34).
- **Valid exceptions:** Non-text media, icons, canvas; documented truncation patterns meeting the 1.4.12 ellipses exception; truly fixed-height chrome referenced via a single shared token constant.
- **Primary sources:** https://www.w3.org/WAI/WCAG22/Understanding/resize-text.html (SC 1.4.4; F69) ; https://www.w3.org/WAI/WCAG22/Understanding/text-spacing.html (SC 1.4.12; F104) ; https://www.w3.org/WAI/WCAG22/Understanding/reflow.html (SC 1.4.10; C34) ; https://developer.mozilla.org/en-US/docs/Web/CSS/scroll-margin-top ; https://developer.mozilla.org/en-US/docs/Web/CSS/position

### 6.11 Unmanaged scheduled and subscribed work: magic `setTimeout` sequencing, missing teardown, ungated loops, animation instances outside the framework lifecycle
- **Classification:** objectively broken | inefficient
- **Severity:** High
- **Observable symptom:** A modal sometimes unmounts before its close animation; focus fires before the target exists (or never); E2E passes locally and flakes in CI; flows break after the tab was backgrounded; one keypress triggers an action N times after N visits; animations speed up or duplicate after navigating away and back; tweens fire on detached nodes and `from()` tweens leave elements stuck at start values; JS heap and listener counts grow monotonically; fan/battery drain from an idle background tab or from animations scrolled far out of view.

- **Valid exceptions:** Debounce/throttle of user input; deliberate UX delays (toast dwell time); polling fallbacks; test waits; true app-lifetime singletons installed once at bootstrap; GSAP/Motion driving nodes entirely outside React's tree; always-visible rAF chrome (rAF auto-pauses in hidden tabs); animations whose state must advance in real time while hidden (compute from timestamps, not tick counts).

### 6.12 Z-index escalation and stacking-context traps
- **Classification:** objectively broken
- **Severity:** High
- **Observable symptom:** Opening a modal shows the backdrop dimming *on top of* the dialog, or the dialog slides under the sticky header; buttons inside the dialog cannot be clicked (clicks land on the layer above); keyboard focus is trapped invisibly behind the overlay; z-index values escalate over time (999 → 9999 → 99999 → 2147483647) — and the bump sometimes does not even work.
- **Likely implementation cause:** Ad-hoc per-component magic z-index values with no global scale; the modal is not portaled to `document.body`, so an ancestor with `transform`, `filter`, `opacity < 1`, `will-change`, or `isolation` creates a stacking context that caps the modal's effective z; backdrop and dialog live in different stacking contexts.
- **Why it matters:** Per MDN's stacking-context documentation, `z-index` values are only compared within the same stacking context — a `z-index: 9999` inside a low context loses to `z-index: 1` in a sibling higher context, so magic numbers do not even guarantee wins; they only destroy predictability. The WAI-ARIA APG Dialog (Modal) pattern requires that windows under a modal dialog be inert and visually obscured — a dialog that cannot be seen or clicked breaks that contract and maroons keyboard focus. This is deterministic rendering breakage, verifiable with `elementFromPoint`.
- **Static detection:** Inventory all `z-index` declarations / `z-` utilities into a project z-map; flag more than ~8 distinct values or any literal ≥ 9999 (or > 100 outside a token scale); stylelint `declaration-strict-value` restricting `z-index` to token variables; verify dialog/modal components render through a portal or native `<dialog>`; check modal mount-point ancestors for stacking-context creators (`transform`, `filter`, `opacity`, `will-change`, `isolation: isolate`).
- **Runtime or browser test:** Open each modal and assert `document.elementFromPoint()` at the dialog's center returns an element inside the dialog; Playwright `page.getByRole('dialog')` visibility/clickability assertions; DevTools Layers panel to inspect stacking contexts; open modal + dropdown + toast simultaneously and verify paint order.
- **Corrective rule:** Overlay UI MUST be rendered in a top-level layer — native `<dialog>.showModal()` (top layer, no z-index needed), the Popover API, or a portal — or use a single documented token z-scale owned by the design system; arbitrary per-component z-index escalation MUST be rejected in review.
- **Valid exceptions:** Native `<dialog>` top-layer rendering (stacking handled by the platform); layering within a self-contained widget whose root establishes its own stacking context; third-party widgets with fixed z-index assumptions, isolated in their own stacking context.
- **Primary sources:** https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_positioned_layout/Understanding_z-index/Stacking_context ; https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/ ; https://www.w3.org/WAI/WCAG22/Understanding/focus-not-obscured-minimum.html
- **Confidence:** High

### 6.13 Non-composited and paint-bound animation; shotgun `will-change`
- **Classification:** inefficient
- **Severity:** High
- **Observable symptom:** Janky slide-ins, accordion opens, drawers, and grow-on-hover effects; DevTools Performance shows non-zero Rendering/Layout blocks during the animation; Lighthouse "Avoid non-composited animations" lists offending nodes; Paint flashing covers large regions on every frame of a shadow/blur effect; scroll jank and battery drain under a fixed frosted navbar on mobile; the Layers panel shows dozens of promoted layers on static content with elevated GPU memory.
- **Likely implementation cause:** CSS keyframes/transitions interpolating `width`, `height`, `top`, `left`, `margin` (or `transition: all`), JS/`setInterval` mutating `element.style.left` per frame, Framer Motion/GSAP animating layout values (`animate={{ width: ... }}`); `transition: box-shadow` with large blur/spread radii, keyframes interpolating `filter: blur()` or `backdrop-filter: blur()`; large-area `backdrop-filter: blur(12px+)` on sticky headers, hero overlays, or full-screen modal backdrops over scrolling content; `will-change: transform` on global utility classes, card grids, or list rows, left permanently in stylesheets.
- **Why it matters:** Animating a layout property forces JS/CSS → Style → Layout → Paint → Composite every frame, for the element and its descendants/siblings — up to 60–120 full layout passes per second against a ~16.7 ms frame budget (≈ 10 ms usable) — and physically moves elements the CLS algorithm measures (the Lighthouse non-composited-animations audit exists for exactly this). Blurs and shadows repaint expensively every frame ("anything involving a blur… is going to take longer to paint than drawing a red box" — web.dev). `backdrop-filter` resamples and re-blurs everything painted behind the element on every scroll/animation frame (cost scales with area × radius), and backdrop roots (`filter`, `opacity < 1`, `mask`, `clip-path`, `mix-blend-mode` ancestors) silently clip its scope — a correctness trap as well. MDN is normative on `will-change`: "intended to be used as a last resort… should not be used to anticipate performance problems"; excessive use "will result in excessive memory use."
- **Static detection:** Grep `transition:` declarations and `@keyframes` blocks containing `width|height|top|left|right|bottom|margin|padding`; flag `transition: all`; grep per-frame `style.left|style.top|style.width` writes; flag Framer Motion `animate`/`initial` objects containing layout properties and `boxShadow`/`filter` keys; grep `backdrop-filter` / Tailwind `backdrop-blur-*` on elements with `fixed`, `sticky`, `inset-0`, `w-screen`, `h-screen`, `min-h-screen`; grep `will-change` on multi-element selectors or occurrences not paired with script that restores `auto`.
- **Runtime or browser test:** Chrome DevTools Performance recording during the animation (non-zero Rendering time per frame) plus the FPS meter; Lighthouse "Avoid non-composited animations"; Rendering tab Paint flashing during shadow/blur effects; Layers panel layer counts on a "static" page; verify on a mid-range Android device or with DevTools CPU/GPU throttling that no frames drop.
- **Corrective rule:** Animations that run on scroll, hover, or in a loop MUST animate only `transform` and `opacity` (translate for position, scale for size, opacity for visibility). Shadow/filter animations SHOULD cross-fade `opacity` between two pre-rendered layers rather than interpolating the shadow/filter itself. `backdrop-filter` SHOULD be confined to small, mostly static regions with a modest radius, be reduced or disabled on mobile via media query or feature detection, and MUST NOT sit over continuously scrolling or animating content without a profile demonstrating 60 fps on a mid-range device. `will-change` MUST be applied only to elements actually about to animate, shortly before the animation, and removed after — never preemptively on lists, grids, or global utilities.
- **Valid exceptions:** One-shot layout animations where FLIP semantics genuinely require geometry change on a small subtree; `height: auto`-style accordion reveals that profile cleanly; small elements with modest shadow radii and one-shot transitions; a single permanently interactive surface (MDN's slide-deck case); a draggable element during the drag gesture.
- **Primary sources:** https://web.dev/articles/animations-guide ; https://web.dev/articles/rendering-performance ; https://developer.chrome.com/docs/lighthouse/performance/non-composited-animations ; https://developer.mozilla.org/en-US/docs/Web/CSS/backdrop-filter ; https://developer.mozilla.org/en-US/docs/Web/CSS/will-change
- **Confidence:** High (Medium on any fixed blur-radius threshold — cost is area- and radius-dependent, so the normative rule is profile-based)
- **Cross-reference:** Motion UX questions — animation purpose, duration, and the `prefers-reduced-motion` requirement — are catalogued in Section 3.

### 6.14 Interaction main-thread overload: high-frequency `setState`, blocking handlers, layout thrashing, over-broad context
- **Classification:** inefficient
- **Severity:** High (Critical when measured INP > 500 ms)
- **Observable symptom:** Scroll stutter and mouse-parallax lag; clicks visibly hang before the UI responds (users double-click thinking the first click failed); DevTools Performance shows repeating > 50 ms tasks per scroll tick or a single > 200 ms task between input and next paint; purple Layout blocks flagged "forced reflow" repeat many times within one frame; typing in one input re-renders large parts of the app (Profiler flame graph lights up broadly).
- **Likely implementation cause:** `onScroll={() => setScrollY(window.scrollY)}` / `onMouseMove={(e) => setPos(...)}` on high-level components with no throttling, rAF batching, or ref-based DOM writes; click/change handlers that synchronously sort/filter/map thousands of items, `JSON.parse` large payloads, or build `innerHTML` strings before yielding; loops alternating DOM writes and layout reads (`el.style.width = x; el2.style.height = el.offsetWidth`) per frame or per item; mega-providers stacked at the root whose inline `value={{ user, theme, cart }}` object changes on every keystroke, broadcasting to all consumers.
- **Why it matters:** INP (good ≤ 200 ms, poor > 500 ms at the 75th percentile) is input delay + processing duration + presentation delay; scroll and pointermove fire per frame or faster, so reconciling a large tree per event exceeds the frame budget, and heavy synchronous work per click delays both processing and the frame presentation. web.dev's optimize-INP guidance prescribes minimal work in the event callback, yielding/task-splitting so the visual update renders first, avoiding layout thrashing (write-then-read "forces the browser to perform synchronous layout work that it could have otherwise waited to do asynchronously"), and minimizing DOM size. React docs: a changed context value re-renders all consumers below; Next.js docs: render providers "as deep as possible in the tree."
- **Static detection:** AST grep: `addEventListener('scroll'|'mousemove'|'pointermove', …)` or JSX `onScroll`/`onMouseMove` whose handler's synchronous path invokes a state setter with no `requestAnimationFrame` wrapper, throttle/debounce, or ref write; review heuristic flagging interaction handlers containing unbounded loops, payload-size `JSON.parse`, `innerHTML` building, or `.map()` over list state re-rendering large trees; flag layout reads (`offsetWidth|offsetTop|getBoundingClientRect|getComputedStyle|clientWidth|scrollHeight`) appearing after a style/class/DOM mutation within one function/loop body; count root-layout providers and flag volatile values and inline `value={{...}}` object literals.
- **Runtime or browser test:** React DevTools Profiler while scrolling/mousing — commits per second should be ≈ 0 for visual-only effects; DevTools Performance: measure task duration between the input event and the next painted frame, and count Layout blocks per frame ("Forced reflow" warnings); `web-vitals` `onINP()` in a RUM snippet or Playwright-driven interaction; Lighthouse TBT as a lab proxy (explicitly a proxy, not a replacement, per web.dev).
- **Corrective rule:** Visual responses to scroll/pointermove MUST NOT route through React state at native event frequency — use rAF-batched ref/DOM writes, CSS/scroll-driven animations, or throttling, reserving state updates for discrete logical changes (e.g., Intersection-Observer-computed threshold booleans). Event handlers MUST keep synchronous work within a budget that holds INP ≤ 200 ms on a mid-range device; non-visual work (saving, analytics, secondary updates) MUST be deferred to a separate task (`setTimeout`/`scheduler.postTask`) after the visual update. DOM reads and writes MUST be batched into separate phases — measure everything first, then mutate. Context providers MUST wrap only the subtree that consumes them, be split by change frequency, and keep `value` referentially stable.
- **Valid exceptions:** Genuinely state-dependent UI on small, profiled subtrees (a slider driving controlled form logic); virtualization libraries' own engineered scroll handling; intrinsically expensive discrete operations (export 100k rows) that paint a busy state first and run deferred or in a Web Worker; single measure-then-set pairs done once per frame (FLIP); app-global low-frequency contexts (theme, session, locale).
- **Primary sources:** https://web.dev/articles/inp ; https://web.dev/articles/optimize-inp ; https://web.dev/articles/rendering-performance ; https://react.dev/learn/passing-data-deeply-with-context ; https://nextjs.org/docs/app/getting-started/server-and-client-components
- **Confidence:** High

### 6.15 Missing virtualization for long lists
- **Classification:** inefficient
- **Severity:** High
- **Observable symptom:** A page with a big table or feed takes seconds to become interactive; scrolling feels heavy; every keystroke or click lags; DevTools shows tens of thousands of DOM nodes; style-recalculation blocks span large parts of the frame; Lighthouse reports "Avoid an excessive DOM size."
- **Likely implementation cause:** `{items.map(item => <Row ... />)}` over unbounded API data with no `react-window` / `react-virtualized` / `@tanstack/react-virtual` / `react-virtuoso`; "infinite scroll" that appends without windowing; no pagination.
- **Why it matters:** web.dev's virtualization guide: rendering every item of a large list "can significantly affect performance" — windowing keeps rendered DOM proportional to the viewport and recycles nodes leaving the window. web.dev optimize-INP names DOM size a direct input to both initial render cost and interaction-update cost, and notes that infinite loading without windowing still "ends up with thousands of row entries" in the live DOM.
- **Static detection:** Grep `.map(` in JSX returning row components where the data source is an unbounded prop/array and no virtualization library is imported; check for pagination. Review heuristic: any list whose length is server/data-driven and unbounded is suspect.
- **Runtime or browser test:** DevTools Elements node count (or `document.querySelectorAll('*').length`); Performance recording during scroll looking for long Recalculate Style/Layout blocks; Lighthouse excessive-DOM-size diagnostic.
- **Corrective rule:** Lists and tables that can exceed roughly 100–200 rendered rows MUST use windowing/virtualization (or true pagination); infinite loading MUST be combined with windowing, not raw DOM append; `overscanCount` SHOULD stay small (react-window default 1; raise only to mask blank-flash).
- **Valid exceptions:** Bounded small lists; `content-visibility: auto` on long but simple static sections as a lighter-weight mitigation for render cost (it does not fix DOM-memory size).
- **Primary sources:** https://web.dev/articles/virtualize-long-lists-react-window ; https://web.dev/articles/optimize-inp
- **Confidence:** High

## 7. Validation and QA Failures

Validation and QA failures are the most predictable cluster in this report. Observed pattern (inference, not sourced fact): AI generators render the happy path — the populated dashboard, the successful submit — because that is what prompts, mockups, and training screenshots depict, while the loading, error, empty, and confirmation states that surround every asynchronous operation have no visual anchor and are systematically dropped. The result is code that passes a screenshot review and fails the first time a request is slow, a server returns 500, a list comes back empty, or a user double-clicks. Several of these failures are not merely brittle but deceptive: fabricated success toasts, perpetual skeletons, and swallowed errors assert system states that are false. Because these behaviors are invisible in static review, they are also the cheapest class of slop to catch with the runtime tests specified below — a reviewer who never throttles the network or forces a 500 will approve all of them.

### 7.1 Async data rendered with no loading state (content pop-in, layout shift)
- **Classification:** objectively broken | inefficient
- **Severity:** High
- **Observable symptom:** On load or tab switch, sections are missing and then slam into place; content below the fold shifts; the page looks broken, or finished-loading, before data arrives.
- **Likely implementation cause:** `useEffect(() => { fetch(url).then(r => r.json()).then(setData) }, [])` with only a `data` state — no `isLoading` state, no `<Suspense>` boundary, no `loading.tsx`; initial render returns `null` or an empty fragment for the data section.
- **Why it matters:** Layout instability is a measured Core Web Vital (CLS: ≤ 0.1 good, > 0.25 poor at p75), and "DOM elements dynamically added before existing content" is a named CLS cause. Users also cannot distinguish "loading" from "empty" or "broken."
- **Static detection:** AST heuristic — a component containing `fetch(`/`axios`/`useSWR` whose returned JSX has no conditional branch on a loading flag and no `<Suspense>` ancestor; grep the component for `isLoading|isPending|isFetching|Skeleton|Spinner|role="progressbar"|aria-busy` — zero hits is a flag. In Next.js App Router projects, flag route segments fetching in `page.tsx` with no sibling `loading.tsx`.
- **Runtime or browser test:** Playwright with CDP network throttling (`Network.emulateNetworkConditions`, "Slow 3G"): reload, screenshot at +500 ms — a loading affordance or stable skeleton should occupy the final content box. Lighthouse/Web Vitals: CLS of the load ≤ 0.1; DevTools Performance panel → Layout Shift regions.
- **Corrective rule:** Every client-side asynchronous data source MUST have a loading state that reserves the approximate final layout space (skeleton or sized placeholder); route-level server fetching in Next.js SHOULD be paired with `loading.tsx`.
- **Valid exceptions:** Data statically prerendered/SSR'd into the initial HTML (no client wait); intentionally deferred non-critical widgets where pop-in is by design; React Transitions deliberately keeping stale content instead of a fallback (documented React pattern).
- **Primary sources:** web.dev, Cumulative Layout Shift — https://web.dev/articles/cls ; React docs, `<Suspense>` — https://react.dev/reference/react/Suspense ; Next.js docs, `loading.js` — https://nextjs.org/docs/app/api-reference/file-conventions/loading
- **Confidence:** High

### 7.2 No error boundary: one crashing component whitescreens the entire app
- **Classification:** objectively broken
- **Severity:** Critical
- **Observable symptom:** One component throws during render (e.g., `data.user.name` when `data` is null after a failed fetch) and the entire page goes blank white; header, nav, and unrelated content all disappear.
- **Likely implementation cause:** No class implementing `static getDerivedStateFromError`/`componentDidCatch`, no `react-error-boundary`, and (Next.js) no `error.tsx`/`global-error.tsx`; combined with render-time property access on possibly-null fetched data.
- **Why it matters:** React's documented default: "if your application throws an error during rendering, React will remove its UI from the screen" — an unbounded throw is a whole-app outage triggered by one leaf component.
- **Static detection:** Grep the repo for `getDerivedStateFromError|componentDidCatch|ErrorBoundary|error.tsx|global-error.tsx` — zero hits in an app with async components is a finding; check that boundaries exist at route-segment or major-section granularity, not just at the root.
- **Runtime or browser test:** Playwright: `page.route('**/api/user', r => r.fulfill({ status: 200, body: 'null' }))` (or abort), reload, assert `document.body.textContent.trim()` is non-empty and a scoped error message appears instead of a blank page. React DevTools can toggle error boundaries to simulate throws.
- **Corrective rule:** The application MUST contain error boundaries at least per independent route/section, each rendering a recovery UI (message + retry), so a single component failure cannot unmount the whole tree; Next.js apps MUST ship `error.tsx` (and `global-error.tsx`) for segments that can fail.
- **Valid exceptions:** None for production apps with any async rendering. Error boundaries legitimately do not catch event-handler or plain async errors — those need their own error states (see 7.3 and 7.6).
- **Primary sources:** React docs, Error Boundaries — https://react.dev/reference/react/Component#catching-rendering-errors-with-an-error-boundary ; Next.js docs, `error.js` / `global-error.js` — https://nextjs.org/docs/app/api-reference/file-conventions/error
- **Confidence:** High

### 7.3 Failed request renders a permanently blank section
- **Classification:** objectively broken | deceptive
- **Severity:** High
- **Observable symptom:** A widget or section shows nothing when its API 500s or the network drops — no message, no retry. A tester with a blocked network sees an empty page region that looks intentional.
- **Likely implementation cause:** `fetch(...).then(r => r.json()).then(setData)` with no `.catch` and no `error` state; or `res.ok` never checked so an HTTP 500 body is parsed as if valid; or `catch {}` swallowing into the existing `null` initial state that renders nothing.
- **Why it matters:** "Request failed" and "there are zero items" are different truths with different user actions; collapsing both into a blank area misinforms the user and removes any recovery path. The blank region is indistinguishable from "no data exists," so users make wrong decisions from it.
- **Static detection:** AST — fetch/axios call chains in components lacking both `.catch(` (or surrounding `try/catch`) and any error-state setter (`setError`); flag `res.json()` called without a preceding `res.ok` check. `@typescript-eslint/no-floating-promises` catches some un-awaited/un-caught promises. Grep for empty catches: `catch {}`, `catch (e) {}`, `.catch(() => {})`.
- **Runtime or browser test:** Playwright `page.route('**/api/**', route => route.abort())` or `fulfill({status: 500})`; assert a visible, textual error indication with a retry affordance appears in the affected region.
- **Corrective rule:** Every asynchronous data path MUST have a visible, textual error state (region-scoped, with retry where refetch is possible); HTTP status MUST be validated (`res.ok`) before parsing; a rejection MUST NOT render identically to a successful empty result.
- **Valid exceptions:** Non-critical decorative enhancements explicitly designed to fail silently (e.g., an optional avatar that falls back to initials) — must be intentional, not accidental.
- **Primary sources:** React docs, Error Boundaries (error display patterns) — https://react.dev/reference/react/Component#catching-rendering-errors-with-an-error-boundary ; Next.js docs, `error.js` (segment-scoped error UI with retry) — https://nextjs.org/docs/app/api-reference/file-conventions/error
- **Confidence:** High

### 7.4 Loading indicator that never resolves (no error exit, no timeout, hardcoded skeletons)
- **Classification:** objectively broken | deceptive | inaccessible
- **Severity:** High
- **Observable symptom:** Skeleton shimmer, a spinner, or a progress bar stays forever when the request fails, hangs, or rejects; the user waits indefinitely with no message and no way out. Worst variant: skeleton markup (`<div className="animate-pulse">`) ships as the final UI with no fetch behind it at all, or `aria-busy="true"` is hardcoded.
- **Likely implementation cause:** `setLoading(false)` appears only in the success `.then` branch, not in `catch`/`finally`; the fetch has no `AbortSignal.timeout()` and no timeout UI; a rejected promise inside a Suspense data path with no error boundary leaves the streamed fallback in place forever; or the loading state was generated as the *only* state — skeleton markup in static JSX with no `isLoading` flag or data fetch.
- **Why it matters:** A perpetual busy indicator is objectively wrong information about system state. WCAG SC 4.1.3's own examples cover the "waiting state of an application" (a busy icon): a real busy state must be programmatically determinable and must *end*. MDN defines `aria-busy` as `"true"` while an element is being updated and `"false"` when complete — a perpetual `true` tells assistive technology the region is never readable. ARIA's `progressbar` role asserts that "the application is making progress toward completing the requested action" — a fake progress bar asserts progress that is not happening.
- **Static detection:** AST — for each `setLoading(true)`, verify a reachable `setLoading(false)` on every code path (success, catch, finally); grep fetches for `signal:` / `AbortSignal.timeout` presence; flag skeleton components rendered purely on `isLoading` with no sibling error branch. Grep `animate-pulse|animate-spin|Skeleton|role="progressbar"|aria-busy="true"` and check each hit for a corresponding state variable that can turn it off; loading JSX returned unconditionally (not behind an `isLoading` conditional) is a finding. axe-core rule `aria-progressbar-name` covers progress-bar naming as a hygiene pass.
- **Runtime or browser test:** Playwright: abort or 500 the request, wait a bounded time (e.g., 15 s), assert the spinner/skeleton is gone and an error+retry UI replaced it; test a hung request (a route that never responds) to verify a timeout path exists. Load the page and wait with network idle: any skeleton still visible is a finding (`expect(skeleton).toBeHidden()` after load). Screen-reader pass: the region must not be stuck in a busy state.
- **Corrective rule:** Loading states MUST terminate on every outcome — resolve, reject, and timeout (`setLoading(false)` in `finally` or equivalent) — and long-running requests SHOULD use `AbortSignal.timeout()` or equivalent with a user-visible failure+retry state. Loading indicators MUST be driven by real asynchronous state with a guaranteed terminal state (content, error, or empty state with explanation); `aria-busy="true"` MUST be cleared on completion; hardcoded skeleton markup as final UI is forbidden.
- **Valid exceptions:** Indeterminate progress with an explicit user-cancel affordance; deliberately indefinite background polling that is not gating visible content; genuine short-lived indeterminate operations; SSR streaming shells that resolve; skeletons in design-system documentation.
- **Primary sources:** MDN, AbortController — https://developer.mozilla.org/en-US/docs/Web/API/AbortController ; MDN, AbortSignal.timeout() — https://developer.mozilla.org/en-US/docs/Web/API/AbortSignal/timeout_static ; React docs, `<Suspense>` (fallback/error behavior) — https://react.dev/reference/react/Suspense ; WCAG 2.2 SC 4.1.3 Status Messages — https://www.w3.org/TR/WCAG22/#status-messages and Understanding — https://www.w3.org/WAI/WCAG22/Understanding/status-messages.html ; MDN, `aria-busy` — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Attributes/aria-busy ; WAI-ARIA 1.2, `progressbar` role — https://www.w3.org/TR/wai-aria-1.2/ (verified build: https://w3c.github.io/aria/#progressbar)
- **Confidence:** High

### 7.5 Empty list/table renders as a void (no zero state)
- **Classification:** objectively broken
- **Severity:** Medium
- **Observable symptom:** A filtered list, table, search result, or cart with zero items shows a blank area: no "No results," no guidance, no next action. With a broken filter, the user cannot tell whether the query matched nothing or the UI failed.
- **Likely implementation cause:** JSX goes straight to `{items.map(...)}` with no `items.length === 0` branch; tables render headers over an empty `<tbody>`; the empty case was never in the mock data the code was generated against.
- **Why it matters:** WCAG's own SC 4.1.3 examples treat "No results returned" as a status message worth surfacing (including to screen readers); an unexplained void hides which of three system states (loading/empty/error) is true. Not functionally broken, but objectively ambiguous — not a taste issue.
- **Static detection:** AST — `.map(` over fetched/filterable arrays in JSX with no `length === 0` / ternary-empty sibling branch; grep the component for `length === 0|No .* found|empty` — zero hits flags it.
- **Runtime or browser test:** Playwright: mock the endpoint to return `[]`; assert visible zero-state text and, where applicable, a recovery action (clear filters, create-first-item CTA); assert the zero state is inside a status region (see 7.11).
- **Corrective rule:** Every data-driven collection UI MUST render an explicit empty state (what happened + what to do next) that is visually distinct from both the loading and error states.
- **Valid exceptions:** Collections that are non-empty by construction (e.g., fixed navigation); UI where emptiness is self-evident from surrounding context and adds no ambiguity.
- **Primary sources:** WCAG 2.2 Understanding SC 4.1.3 (status-message examples include "No results returned") — https://www.w3.org/WAI/WCAG22/Understanding/status-messages.html
- **Confidence:** High

### 7.6 Mutation outcome missing or falsified (silent success, swallowed failure, fake success)
- **Classification:** deceptive | objectively broken
- **Severity:** High — Critical for transactional flows (contact, checkout, signup) where fabricated success silently discards user data; Medium for low-stakes missing confirmations.
- **Observable symptom:** Three variants: (a) the user clicks Save/Send and nothing visibly changes — no toast, no status, no state change — so they re-click or navigate away unsure; (b) the dialog closes or the UI sits unchanged while the POST actually 500'd, and the user believes the data persisted until a reload proves otherwise; (c) a "Thanks! We'll be in touch" message, success toast, or confetti appears with no network request firing at all, or the success UI is shown regardless of the response status.
- **Likely implementation cause:** Mutation handler ends at `await fetch(...)` with no feedback call; `.catch(() => {})` or `try { ... } catch {}`; success UI (toast/close/redirect) triggered in `.then` without checking `res.ok`; `finally`-style cleanup that closes the dialog regardless of outcome; `onSubmit={(e) => { e.preventDefault(); setSubmitted(true); }}` with no mutation behind it; a `setTimeout(() => setSuccess(true), 800)` demo path left in production.
- **Why it matters:** False feedback about system state is worse than no feedback. WCAG SC 4.1.3's status-message examples ("Your form was successfully submitted", technique G199) presume the message is true, and Understanding SC 3.3.1 notes it is "not sufficient to only re-display the form without providing any hint that the submission failed." A fabricated success converts a wiring defect into data loss plus a false confirmation. (Secondary background, not conformance evidence: Nielsen Norman Group's visibility-of-system-status heuristic — missing feedback is what drives users to repeat clicks.)
- **Static detection:** Grep mutation paths for `catch {}`, `catch (e) {}`, `.catch(() => {})`, `.catch(console.log)`; AST — `.then(` running success UI without a preceding `res.ok`/`response.status` check; `closeModal()`/`navigate(` reachable when the request rejected; `onSubmit` bodies containing only `preventDefault` + state set + optional `console.log`; success state set on a path that does not await the mutation result; `setTimeout`-only success paths; mutation functions (`POST|PUT|PATCH|DELETE` fetch, `mutate(` calls) whose completion path contains no UI feedback at all. Code-review heuristic: "what does the user see when this succeeds / fails?" has no answer in the JSX.
- **Runtime or browser test:** Playwright with network interception: submit and assert exactly one mutation request fires and success UI appears only after a 2xx response; force a 500 and assert (a) an error message is shown, (b) no success toast/navigation occurred, (c) entered data is still present or recoverable. Assert the feedback sits inside `role="status"`/`role="alert"` (see 7.11).
- **Corrective rule:** Every user-initiated mutation MUST end in an unambiguous, truthful outcome signal — success feedback only after confirmed success (`res.ok`/resolved promise), a textual error on failure, a pending state in between. A rejected/failed mutation MUST NOT trigger success-side effects (toast, close, redirect) that imply the write persisted. A form with no backend MUST be clearly framed as a demo or disabled — never fake success.
- **Valid exceptions:** Mutations whose result is inherently and immediately visible in the acted-upon UI (an inline edit where the field itself updates); high-frequency micro-interactions (likes) with instant local visual change; fire-and-forget telemetry/analytics pings; honest optimistic UI that rolls back and signals on failure (see 7.7); offline-first queues that communicate a "will sync" status.
- **Primary sources:** WCAG 2.2 Understanding SC 4.1.3 (status-message examples, technique G199) — https://www.w3.org/WAI/WCAG22/Understanding/status-messages.html ; WCAG 2.2 Understanding SC 3.3.1 — https://www.w3.org/WAI/WCAG22/Understanding/error-identification.html ; MDN, Constraint validation (client validation never removes the need for server validation — failures are expected) — https://developer.mozilla.org/en-US/docs/Web/HTML/Constraint_validation ; (secondary background) Nielsen Norman Group, Visibility of System Status — https://www.nngroup.com/articles/visibility-system-status/
- **Confidence:** High

### 7.7 Optimistic update with no rollback path
- **Classification:** deceptive | objectively broken
- **Severity:** High
- **Observable symptom:** An item vanishes from the list on click; on refresh (or never, in-session) the failed delete is revealed — the item is back, or a "sent" message never actually sent. Under failure, the UI and the server permanently disagree until reload.
- **Likely implementation cause:** Local state mutated first (`setItems(items.filter(...))`), then `await fetch(...)` with no catch branch that restores the snapshot; no use of React `useOptimistic` (whose documented semantics roll back automatically when the Action fails) or library mutation rollback (TanStack Query `onError` with context rollback).
- **Why it matters:** The UI asserts a fact (item deleted, message sent, like counted) that the server rejected; state diverges from the backend silently. React documents the correct pattern explicitly: on Action failure "the item automatically reappears in the list" and "you can catch the error to show a message to the user." Hand-rolled optimistic code without these two behaviors is objectively incorrect, not a style choice.
- **Static detection:** AST — state setters that remove/modify entities located textually before the `await` of the corresponding request, with no companion restore in a `catch`; absence of `useOptimistic`/`onMutate`+`onError` patterns in files doing pre-request local mutation.
- **Runtime or browser test:** Playwright: fail the mutation endpoint, trigger the optimistic action, assert (a) the optimistic change reverts (item reappears / count restores), and (b) an error message is surfaced.
- **Corrective rule:** Every optimistic update MUST define its failure behavior: automatic rollback to the pre-mutation state (via `useOptimistic`, library rollback, or manual snapshot/restore) plus a user-visible error indication.
- **Valid exceptions:** Truly idempotent, retry-guaranteed background sync where transient divergence is acceptable and bounded; operations where failure is impossible by construction (pure local state).
- **Primary sources:** React docs, `useOptimistic` — "Optimistic delete with error recovery" — https://react.dev/reference/react/useOptimistic
- **Confidence:** High

### 7.8 No submission-in-flight guard (duplicate submissions)
- **Classification:** objectively broken
- **Severity:** High
- **Observable symptom:** Double-clicking (or impatient re-clicking) Save/Pay/Send fires the request multiple times: duplicate records, double charges, repeated comments. The button gives no "working…" feedback between click and completion; nothing visible changes for 300–3000 ms, so the user clicks again.
- **Likely implementation cause:** `<button type="submit">` / `onClick={submit}` with `disabled` not bound to any pending flag; no `isPending` from `useTransition`/`useActionState`/`useFormStatus`, no local `submitting` state, no form-level idempotency key.
- **Why it matters:** Duplicate creates/orders/messages/payments are real-world data corruption, not cosmetic — the defect manufactures its own trigger because a static UI reads as a failed click. React's own docs make `disabled={isPending}` the canonical submit-button pattern. For checkout/legal/financial flows, unguarded resubmission also undermines WCAG SC 3.3.4's error-prevention intent (submissions should be checkable/reversible/confirmed).
- **Static detection:** AST/grep — submit buttons in async forms whose `disabled` prop is absent or static; handler functions containing `await fetch(` with no guard variable (`if (submitting) return`); grep form components for `isPending|isSubmitting|isLoading|useFormStatus` — zero hits flags it. Heuristic: absence of both a pending flag and an idempotency key.
- **Runtime or browser test:** DevTools Network, throttle to Slow 3G, double-click Submit, count requests (must be 1). Playwright: `page.route` the endpoint with a 2 s artificial delay, `Promise.all([btn.click(), btn.click()])`, assert exactly one request was captured and the button became disabled/showed pending state after the first click.
- **Corrective rule:** Every submit/mutation trigger MUST enter a pending state immediately on activation and MUST ignore additional activations until settlement (`disabled={isPending}` or equivalent guard); destructive or financial submissions SHOULD additionally be idempotent server-side (idempotency key). Disable only during the in-flight request — do NOT keep submit pre-disabled as a validation strategy (see 7.9; industry guidance: Vercel Labs Web Interface Guidelines, "disable during the in-flight request … include an idempotency key").
- **Valid exceptions:** Actions that are naturally idempotent and instant (toggles resolved locally); multi-submit wizards where re-entry is explicitly designed and deduplicated by request keys; pure read operations (search GETs) where duplicates are harmless; mutation libraries that already dedupe in-flight identical requests (keep the pending feedback regardless).
- **Primary sources:** React docs, `useTransition` (`disabled={isPending}` pattern) — https://react.dev/reference/react/useTransition ; React docs, `useOptimistic` (pending-flag submit button) — https://react.dev/reference/react/useOptimistic ; HTML Standard §4.10.3 The form element — https://html.spec.whatwg.org/multipage/forms.html#the-form-element ; WCAG 2.2 SC 3.3.4 Error Prevention (Legal, Financial, Data) — https://www.w3.org/TR/WCAG22/#error-prevention-legal-financial-data ; Vercel Labs Web Interface Guidelines (industry guidance, labeled as such) — https://github.com/vercel-labs/web-interface-guidelines
- **Confidence:** High

### 7.9 Validation errors absent, or delegated to native browser bubbles only
- **Classification:** inaccessible | objectively broken
- **Severity:** High
- **Observable symptom:** Submit an invalid form: either a transient browser bubble appears at the first invalid field only (it disappears, does not zoom, shows one error at a time), or — with `noValidate` / JS-driven submit — absolutely nothing happens; the form just does not submit and the user has no idea why. Variant: the submit button is disabled until the form is "valid" with no explanation of why.
- **Likely implementation cause:** Relying solely on `required`/`type="email"` bubbles; or `<form noValidate>` / `e.preventDefault()` + `form.submit()` (MDN: `submit()` does NOT trigger constraint validation) / custom onClick submission, with client-side validity checks that set no visible text — e.g., only `if (!valid) return;`.
- **Why it matters:** WCAG Understanding SC 3.3.1 documents native-bubble disadvantages: messages may not be permanent, do not magnify with zoom, are generic (risking SC 3.3.3), and expose only the first error — and where errors are detected but not described in text, SC 3.3.1 (Level A) fails outright. "Nothing happens on submit" is objectively broken feedback, and a submit button disabled with no explanation fails SC 3.3.2's instruction requirement in practice.
- **Static detection:** Grep forms for `noValidate|novalidate`, `preventDefault()` submits, and `.submit()` calls; for each, verify an error-rendering path exists (error state rendered as text per field or in a summary). Flag handlers matching `if\s*\(!.*(valid|error).*\)\s*return` with no message side effect; flag inputs named `email`/`password` with no `type`/`required`/`pattern` and no validation schema (zod/yup/react-hook-form resolver) in the file.
- **Runtime or browser test:** Playwright: fill invalid data, submit, assert visible error text nodes exist in the DOM (not merely `input.validity.valid === false`); assert every invalid field is identified, not just the first. Manual matrix: empty / wrong format / boundary values; keyboard-only and zoom-to-200% pass on the messages.
- **Corrective rule:** Forms MUST validate (natively via `required`/`type`/`pattern` at minimum, or via JS), and every detected input error MUST be surfaced as persistent, per-field (or summarized + linked) text rendered by the page. Native constraint validation MAY be the mechanism only where its documented limitations are acceptable, and MUST NOT be silently disabled (`novalidate`, programmatic `submit()`) without a replacement message system. Keep the submit button enabled and let validation explain failures (industry guidance: Vercel Labs Web Interface Guidelines, "Don't pre-disable submit").
- **Valid exceptions:** Live-preview inputs that constrain input rather than reject it (a numeric stepper that cannot hold an invalid value); server-authoritative flows where server errors are rendered back as text; truly free-form inputs (a search box, a comment field); native browser validation alone technically satisfies SC 3.3.1 per its Understanding doc, though with documented drawbacks.
- **Primary sources:** WCAG 2.2 Understanding SC 3.3.1 (incl. "User agent native HTML form validation" disadvantages) — https://www.w3.org/WAI/WCAG22/Understanding/error-identification.html ; WCAG 2.2 Understanding SC 3.3.3 — https://www.w3.org/WAI/WCAG22/Understanding/error-suggestion.html ; MDN, Constraint validation — https://developer.mozilla.org/en-US/docs/Web/HTML/Constraint_validation ; HTML Standard §4.10.1.4 Client-side form validation — https://html.spec.whatwg.org/multipage/forms.html ; WAI Forms Tutorial, Validating required input — https://www.w3.org/WAI/tutorials/forms/validation/ ; Vercel Labs Web Interface Guidelines (industry guidance, labeled as such) — https://github.com/vercel-labs/web-interface-guidelines
- **Confidence:** High

Cross-reference: form-structure failures that remove the validation plumbing itself — no real `<form>`, missing `preventDefault()`, broken Enter-key submission — are covered in Section 2.

### 7.10 Errors signaled by color only; error text not associated with its field
- **Classification:** inaccessible
- **Severity:** High
- **Observable symptom:** An invalid field gets a red border (`border-red-500`) and nothing else; or a red error string exists but is not linked to the input — a screen reader focusing the field announces only the label, never "invalid entry" or the error message.
- **Likely implementation cause:** Conditional Tailwind class on the input with no sibling error element; an error `<p>` rendered without `aria-invalid="true"` on the control and without `aria-describedby` pointing at the message `id`; CSS-only `:invalid` styling as the entire error UI.
- **Why it matters:** SC 3.3.1 requires the item in error to be identified and the error described in text; technique ARIA21 (`aria-invalid`) is the documented sufficient programmatic mechanism, and MDN instructs setting `aria-invalid="true"` *and* exposing the message programmatically (`aria-errormessage`/`aria-describedby`). Color alone fails SC 1.4.1 Use of Color. This is a conformance failure, not a design preference.
- **Static detection:** AST — inputs with conditional error styling but no rendered error text node; grep for `aria-invalid` / `aria-describedby` in form components — absent where errors are shown. axe-core catches invalid ARIA usage but NOT the absence of messages; review CSS for `:invalid`/red-border-only error styling.
- **Runtime or browser test:** Playwright: trigger the error, then assert the input has `aria-invalid="true"` and `aria-describedby` resolves to a visible element whose text describes the error; axe-core scan as a hygiene pass; manual NVDA/VoiceOver spot check (focus the field → "invalid entry" plus the message announced); DevTools Accessibility pane shows `aria-invalid: true` and description = error text.
- **Corrective rule:** Every field-level error MUST (a) be described in visible text, (b) set `aria-invalid="true"` on the control while invalid, and (c) associate message and control via `aria-describedby` (or `aria-errormessage`); color MAY be used additionally, never solely. `aria-invalid` MUST NOT be pre-set on empty required fields before the first submit attempt (MDN guidance).
- **Valid exceptions:** None for the text requirement when an input error is automatically detected; association can also be achieved by including the error in the accessible name where `aria-describedby` is impractical; native browser validation bubbles are acceptable per Understanding SC 3.3.1 (with documented drawbacks).
- **Primary sources:** WCAG 2.2 Understanding SC 3.3.1 (technique ARIA21) — https://www.w3.org/WAI/WCAG22/Understanding/error-identification.html ; WCAG 2.2 SC 1.4.1 Use of Color — https://www.w3.org/TR/WCAG22/#use-of-color ; MDN, `aria-invalid` — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Reference/Attributes/aria-invalid ; WAI-ARIA 1.2, `aria-invalid` — https://www.w3.org/TR/wai-aria-1.2/#aria-invalid
- **Confidence:** High

### 7.11 Status changes not announced to assistive technology
- **Classification:** inaccessible
- **Severity:** High
- **Observable symptom:** "Added to cart," "Saved," "5 results returned," busy spinners, and inline form errors appear visually; a screen-reader user hears nothing — the action's outcome is invisible to them unless they manually hunt for it. Toast-only variants auto-dismiss in ~3 s, before or without any announcement.
- **Likely implementation cause:** Toast/message components rendered as plain `<div>`s; a cart badge updating a number with no live region; a spinner with no `role="status"`/`aria-busy` or sr-only "Loading" text; a toast library dropped in with default config (no live-region role); a live region created and populated in the same frame in ways AT misses (MDN recommends a persistent empty `aria-live` element that is then updated).
- **Why it matters:** SC 4.1.3 (Level AA) covers exactly these cases — success/results of an action, waiting state, progress, existence of errors; the Understanding doc's own examples include "Add to Shopping Cart," "Your form was successfully submitted," and busy indicators. Un-announced status messages are named failure technique F103; sufficient techniques are ARIA22 (`role="status"`), ARIA19 (`role="alert"`), and ARIA23 (`role="log"`).
- **Static detection:** AST/grep — toast/notification/badge/spinner components lacking `role="status"|role="alert"|role="log"|aria-live`; flag dynamically injected success/error strings not inside a live region; check spinners for `role="progressbar"`/`role="status"` or sr-only text; verify the toast library is configured with a live-region role (many default to polite regions, many do not).
- **Runtime or browser test:** Playwright: perform the action, then assert `[role="status"], [role="alert"], [aria-live]` exists, contains the message text, and focus was not moved; assert `role="alert"` is reserved for important/time-sensitive messages (APG). Manual NVDA/VoiceOver pass for announcement timing; DevTools Accessibility tree shows the message node with role status/alert. Note: axe-core has no automated rule for a missing live region — a clean axe run is not proof of conformance here.
- **Corrective rule:** Every dynamic status message (success, error, busy, result count) MUST be rendered inside an element with an appropriate role or live-region property (`role="status"`/`aria-live="polite"` for status, `role="alert"` for important time-sensitive errors) that is present in the DOM before/at injection time, without moving focus. Per the APG Alert Pattern, alerts must not affect keyboard focus and should not auto-disappear.
- **Valid exceptions:** Changes of context (dialogs that take focus, route changes) are explicitly out of SC 4.1.3 scope; trivial state changes already exposed via SC 4.1.2 (e.g., a checkbox state change) need no status message; purely decorative animation with a redundant textual cue elsewhere.
- **Primary sources:** WCAG 2.2 Understanding SC 4.1.3 (techniques ARIA22/ARIA19/ARIA23, failure F103) — https://www.w3.org/WAI/WCAG22/Understanding/status-messages.html ; WCAG 2.2 SC 4.1.3 normative text — https://www.w3.org/TR/WCAG22/#status-messages ; MDN, `aria-live` — https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Attributes/aria-live ; WAI-ARIA Authoring Practices Guide, Alert Pattern — https://www.w3.org/WAI/ARIA/apg/patterns/alert/
- **Confidence:** High

### 7.12 Destructive action executed on a single click (no confirmation or undo)
- **Classification:** objectively broken | deceptive
- **Severity:** High where user-controllable data is deleted (WCAG SC 3.3.4, Level AA, applies); Medium otherwise (recoverable items, low stakes).
- **Observable symptom:** Clicking "Delete" on a record instantly and permanently removes it — a mis-click or cat-on-keyboard destroys data with no confirm dialog, no undo snackbar, no trash/recovery.
- **Likely implementation cause:** `onClick={() => deleteItem(id)}` wired directly to the DELETE request; no confirmation dialog, no soft-delete/undo window, no recovery endpoint.
- **Why it matters:** SC 3.3.4 requires that actions deleting user-controllable data be reversible, checked, or confirmed (techniques G99 recover deleted information, G168 request confirmation). A one-click permanent delete is a conformance failure for covered data and an objective data-loss hazard everywhere else.
- **Static detection:** AST — event handlers whose name/body performs delete/remove/destroy calling a mutation directly in the click handler, with no intervening dialog state (`setConfirming`, `<AlertDialog>`, `confirm()`) and no undo affordance in the success path.
- **Runtime or browser test:** Playwright: click delete, assert a confirmation step appears OR the deletion is reversible (undo control with working restore) within the UI; verify confirm-cancel leaves the item intact.
- **Corrective rule:** Actions that delete or destructively modify user-controllable data MUST require confirmation, be reversible (undo/soft-delete with recovery), or be pre-checked with a correction opportunity — at least one of the three, per WCAG SC 3.3.4.
- **Valid exceptions:** Deleting trivially re-creatable data (removing a tag, clearing a draft character); actions already guarded by an explicitly designed undo system; SC 3.3.4 explicitly excludes routine save/edit commands.
- **Primary sources:** WCAG 2.2 Understanding SC 3.3.4 (situations B: information deletion; techniques G99, G168) — https://www.w3.org/WAI/WCAG22/Understanding/error-prevention-legal-financial-data.html
- **Confidence:** High for deletion of user-controllable data; Medium otherwise (strong convention rather than conformance).
